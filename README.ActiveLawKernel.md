## 1. Why this note exists

DME now has enough core, diagnostic, and workflow notes that day-to-day reasoning can become heavier than necessary.

The system does not need every note loaded all the time.

What it does need is a compact live kernel that preserves four things:

1. constitutional boundary posture
2. identity and non-overclaim law
3. bounded workflow movement law
4. current developmental pressure order

This note exists to keep those four functions active without dragging the full README library into every pass.

## 2. Kernel composition rule

If a note does not materially help answer one of those questions during active work, it should usually remain a secondary lookup note rather than part of the always-loaded kernel.

## 4. Kernel functions

### 4.1 Constitutional posture

Inherited from `README_MasterConstitution.md`

The Master Constitution remains the top boundary anchor for:

- active door/layer order
- authority posture
- what current implementation may and may not become
- separation between runtime, canon, prediction, and higher layers

The kernel inherits this without restating it fully.

### 4.2 Identity law

Inherited from `README.StructuralIdentityLaw.md`

A structure retains identity only while:

- the same bounded question remains answerable,
- the same declared constraints remain active,
- sufficient support survives,
- and the claimed continuity rests on a real mechanized basis,
- without distortion strong enough to mislead later interpretation.

This is the main sameness law for DME.

### 4.3 Anti-overclaim law

Inherited from `README.DeclaredVsMechanizedAudit.md` and `README.MechanizationClosureGate.md`

The kernel preserves the following practical rules:

- declared is not mechanized
- displayed is not mechanized
- prepared is not fulfilled
- replay-shaped is not reconstructed
- candidate is not canon

A capability may be called mechanized only when:

- backend exists,
- real execution path reaches it,
- active surface renders it honestly,
- failure / insufficiency remain explicit,
- downgrade posture remains explicit where needed,
- and semantic claim does not outrun implemented support.

### 4.4 Workflow movement law

Inherited from `README_WorkflowContract.md` and `README.PacketWorkflowChain.md`

Bounded work should move through:

1. scope
2. spec
3. implement
4. audit
5. route
6. accept / revise / narrow / defer / escalate
7. merge
8. checkpoint

One packet should normally advance:

- one bounded seam
- through one lawful closure target
- at the lowest layer that honestly resolves the active problem.

### 4.5 Directional development law

Inherited from `README_DoorOneDevelopmentalOutline.md`

Door One development must make the system:

- more usable before more authoritative,
- more replay-honest before more product-polished,
- more identity-accountable before more semantically rich,
- and more support-aware before more promotable.

The current pressure order remains:

1. app surface stabilization
2. reconstruction fidelity and replay honesty
3. identity / memory / compression tracking
4. cross-session and cross-thread continuity
5. retention-spectrum hardening
6. consultation and canon-support seam

## 5. Core live rules

The Active Law Kernel reduces to the following working rules.

### Rule 1 — Stay inside constitutional posture

Do not let current Door One work silently become:

- canon
- truth
- prediction
- ontology
- symbolic authority
- or autonomous workflow sovereignty

### Rule 2 — Preserve bounded identity honestly

The same thing is only the same thing while the same bounded question remains answerable under the same declared constraints with sufficient support and actual mechanized basis.

If not, narrow, downgrade, defer, split, or escalate.

### Rule 3 — Do not overread shaped surfaces

Do not silently round:

- declared → mechanized
- displayed → mechanized
- prepared → fulfilled
- replay-shaped → reconstructed
- candidate → canon

Surface clarity must not outrun mechanism.

### Rule 4 — Move work through packet law

Bounded work should stay packetized, auditable, and routeable.

A packet is not complete because code was written.
A packet is complete only when:

- bounded objective is satisfied,
- required checks have been accounted for,
- preservation rules remain intact,
- and what remains unclaimed is explicit.

### Rule 5 — Follow pressure order

Do not widen into future-layer or future-leg work merely because it is exciting.

The next lawful step is the one that strengthens the current developmental line without increasing authority faster than support.

---

### Rule 6 — Preserve structural exposure before symbolic closure

Where the active seam is raw structural exposure, prefer:

- direct numeric / geometric / relation-bearing output,
- local descriptor anchoring only,
- separated downstream semantic/support/review layers,
- and ugly but exposure-first naming

over semantic helpfulness, support/accounting dominance, or mixed-surface convenience.

If a surface requires labels, summaries, or review posture to reveal its structure, the seam has likely drifted out of first-order exposure.
