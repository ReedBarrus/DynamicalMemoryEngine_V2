- worker-role separation

## 1. Function

This document exists only to answer:

- what thread roles exist
- what macro verb each thread belongs to
- what each thread is allowed to do
- what each thread must not do
- how handoff occurs between threads
- how worker capability stays subordinate to role authority

Nothing else belongs here.

---

## 2. Core thread law

Each active thread has exactly one primary role.

Each active thread operates inside exactly one active macro verb.

Roles do not cut across macro verbs by default.

Cross-verb action requires explicit escalation or explicit handoff.

No thread may silently absorb a neighboring verb.

Thread capability does not grant thread authority.

Direct repo access does not grant cross-role authority.

---

## 3. Thread roles

Active thread roles are:

1. Architect Reed
2. Admission-Advisor
3. Formation-Constructor / Compressor
4. Decision-Reflector
5. Evaluation-Auditor / Reviewer

---

## 4. Role-to-macro-verb mapping

### 4.1 Architect Reed

Primary function:

- set priority
- approve or reject bounded scope
- accept or reject proposed movement
- choose between lawful alternatives
- dispose escalations

Architect Reed is not a macro-verb worker thread.

Architect Reed is the final human authority above thread routing.

### 4.2 Admission-Advisor

Primary macro verb:

- Admit

Primary function:

- read current repo reality
- identify active seam
- bound the working unit
- identify governing constraint posture
- prepare admission-side packet input

Primary emitted-object target:

- bound work object

Admission-Advisor must not:

- define structural identity
- choose implementation path
- resolve evaluation outcome
- silently widen scope

### 4.3 Formation-Constructor / Compressor

Primary macro verb:

- Form

Primary function:

- structure active material
- compress active scope
- classify artifacts
- separate mixed-role surfaces
- externalize identity-bearing structure
- prepare formation-side packet output

Primary emitted-object target:

- identity object

Formation-Constructor / Compressor must not:

- choose action
- mutate repo reality as a decision act
- settle truth
- silently import evaluation judgment into structure

### 4.4 Decision-Reflector

Primary macro verb:

- Decide

Primary function:

- reflect on lawful paths
- constrain the navigable decision surface
- propose bounded packets
- determine whether handoff is lawful
- determine when downward recruitment is required

Primary emitted-object target:

- decision object

Decision-Reflector must not:

- silently mutate repo reality
- act as final approval authority
- redefine structure without formation handoff
- collapse proposal into approval

### 4.5 Evaluation-Auditor / Reviewer

Primary macro verb:

- Evaluate

Primary function:

- audit changes and outcomes
- review significance
- reconcile continuity
- resolve workflow posture
- detect role drift
- recruit downward when lawful evaluation fails

Primary emitted-object target:

- evaluation object

Evaluation-Auditor / Reviewer must not:

- redefine structure retroactively
- silently promote outcomes into authority
- act as a substitute for admission or formation

---

## 5. Worker-role separation

A worker is the execution agent assigned to a thread.

A role is the authority posture of the thread.

Workers may vary.

Examples:

- ChatGPT may serve as Decision-Reflector
- Codex may serve as Admission-Advisor
- Codex may serve as Formation-Constructor / Compressor
- Codex may serve as Evaluation-Auditor / Reviewer

Worker capability does not override role limits.

A worker with direct repo access is still bounded by the role of the active thread.

A worker may not use tool access to silently cross into another macro verb.

---

## 6. Handoff rule

Handoff is the only lawful route between primary thread roles.

Every handoff must preserve:

- active seam
- active macro verb status
- files in scope
- governing surfaces
- current emitted-object status
- declared non-goals
- acceptance target

Handoff may narrow scope.

Handoff may not silently widen scope.

If a handoff changes the seam, the prior pass closes and a new pass begins.

---

## 7. Escalation rule

Escalation is allowed only when:

- boundary conflict appears
- current macro-verb authority is insufficient
- hidden dependency blocks honest completion
- repo reality conflicts with declared packet state
- emitted object cannot be produced lawfully

Escalation must stay bounded.

Escalation does not erase prior packet state.

Escalation routes upward to Architect Reed or laterally by explicit re-entry through the proper thread.

---

## 8. Role drift rule

Role drift includes:

- admission defining identity
- formation choosing action
- decision mutating repo reality without accepted packet authority
- evaluation redefining structure retroactively
- any thread silently performing another role's job

Evaluation is the primary detector of role drift.

Detected drift must be reported explicitly.

Unreported drift is workflow failure.

---

## 9. One-line summary

DME threads are role-bounded, macro-verb-bounded workflow lanes whose authority is determined by the current role and packet, not by worker capability or tool access.
