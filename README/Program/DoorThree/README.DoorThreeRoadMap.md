# Dynamical Memory Engine — Door Three Language Translation and Symbolic Interface

## Status

This document defines the planned Door Three posture for language translation, symbolic tokens, and user-facing semantic interaction.

It is a supporting forward-design note.

It does **not** override:

* `README.MasterConstitution.md`
* `README.WorkflowContract.md`
* `README/Constitution/Workflow/README.DevelopmentPressure.md`
* future Door Two canon / prediction notes

Its purpose is narrower:

* define how language should attach to the system without becoming source authority,
* preserve the distinction between structure, canon, and symbolic rendering,
* establish a clean product-facing interface direction,
* give Door Two and Door Three a lawful seam,
* prevent premature language inflation during Door One and early Door Two work.

---

## 1. Constitutional posture

Door One remains below canon.

Door Two is the activation of:

* Canon Space
* Prediction Space

Door Three is the first lawful home for:

* symbolic interface,
* language translation,
* agent interaction,
* ecology-level shared semantic participation,
* reflective and user-facing meaning surfaces.

This note assumes that language is **downstream of canon** and **downstream of structural legitimacy**.

Language must not:

* define truth,
* replace provenance,
* silently rewrite canon,
* substitute for runtime evidence,
* introduce semantic authority where structural authority has not been established.

---

## 2. Why this note exists

The Dynamical Memory Engine is developing toward product use by others.

That means the system will eventually need an interface that ordinary users, developers, and agents can actually work with.

For most users, that interface will not be:

* raw trajectories,
* operator receipts,
* basin tables,
* structural audit summaries.

It will be:

* language,
* symbolic labels,
* structured summaries,
* queryable explanations,
* agent-readable semantic handles.

Without an explicit note, language pressure can arrive too early and distort lower-layer development.

This note exists to keep the direction product-relevant while preserving the lawful staircase:

measurement → structure → runtime memory → recognition → canon → prediction → agency → ecology → symbolic → meta

---

## 3. Core principle

**Language is a translation layer, not a truth layer.**

The system should establish:

1. what was measured,
2. what structure was detected,
3. what memory stabilized,
4. what canon was legitimately promoted,
5. what predictions are supported,

before attempting to render those results into symbolic or natural language form.

In practical terms:

signal / structure / canon / prediction
→ symbolic token binding
→ language rendering
→ user / agent interaction

This means the language layer is:

* downstream,
* reversible where feasible,
* auditable back to canon and evidence,
* constrained by structural legitimacy.

---

## 4. Product posture

The product-facing opportunity is strong precisely because the system can eventually expose grounded memory and prediction through language.

The key product difference is:

Most language systems begin with language and then approximate structure.

The Dynamical Memory Engine should instead:

* begin with measured structure,
* stabilize memory,
* promote trusted canon,
* then expose the result through language.

This allows future products such as:

* explainable memory-native assistants,
* distributed memory layers for AI systems,
* canon-backed multi-agent coordination,
* continuity-preserving research or operations copilots,
* symbolic query surfaces over grounded system memory.

Language is therefore the primary user interface, but not the primary ontological layer.

---

## 5. Door Three mission for language

Door Three language and symbolic interface exists to:

* translate canonical and predictive system state into human-usable form,
* attach stable symbolic handles to grounded structures,
* support query and explanation,
* support multi-agent or multi-user interaction over shared memory,
* preserve traceability from language back to canon and evidence.

Door Three language does **not** exist to:

* invent truth,
* override canon,
* bypass promotion criteria,
* turn read-side convenience into authority,
* collapse symbolic naming into structural legitimacy.

## 6. Layer distinction

### A. Structural layer

This remains below language.

Examples:

* recurrence,
* transition structure,
* segment identity,
* basin organization,
* divergence,
* return behavior,
* temporal persistence.

These are not yet symbolic meaning.

### B. Canon layer

This is the first trusted stabilization layer.

Examples:

* canonical pattern identity,
* promoted recurring regime,
* trusted return class,
* stable event family,
* prediction-supporting transition rule.

Canon is still not language.

### C. Symbolic token layer

This is the first bridge upward.

Examples:

* symbolic handles,
* external labels,
* phrase-class IDs,
* event tags,
* human-readable or agent-readable symbolic references.

The symbolic token layer should point at canon.
It should not create canon.

### D. Language rendering layer

This produces:

* summaries,
* queries,
* explanations,
* reports,
* conversational output,
* agent-facing textual interfaces.

Language rendering should consume:

* symbolic tokens,
* canon state,
* prediction outputs,
* bounded evidence summaries.

---

## 7. Symbolic token posture

Symbolic tokens should be treated as **attachment surfaces** over canonical structure.

A token may represent:

* a canonical identity,
* a recurring event class,
* a phrase family,
* a regime label,
* an externally assigned name,
* a later learned symbolic grouping.

A token should preferably carry:

* `token_id`
* `token_type`
* `target_canonical_id`
* `binding_origin`
* `binding_confidence`
* `version`
* optional external label metadata

Important rule:

**A symbolic token is not evidence.**
It is a handle bound to evidence-backed canon.

---

## 8. Language rendering posture

Language rendering should begin simple.

Preferred initial posture:

* deterministic,
* template-based,
* canon-constrained,
* provenance-linkable,
* low-creativity.

Examples of lawful rendering:

* “A stable pattern was observed, followed by a perturbation, then a return to the prior canonical regime.”
* “This event matches Canonical Pattern C1_0042 with moderate confidence.”
* “The current behavior diverges from the previously promoted baseline class.”

Disallowed early posture:

* free narrative invention,
* metaphor replacing evidence,
* symbolic claims without canon support,
* user-facing language that cannot be traced back to promoted structure.

Later expressive rendering may become valid, but only after the basic translation layer is stable and auditable.

## 9. Door Two → Door Three seam

Door Two should prepare for Door Three by exposing clean attachment points.

Canon objects should support future optional fields such as:

* `external_labels`
* `symbolic_bindings`
* `language_handles`
* `query_aliases`

These fields may remain empty in early Door Two.

The important distinction is:

* Door Two should expose handles,
* Door Three should use them.

This preserves a lawful seam:

canonical structure first
→ symbolic attachment second
→ language interface third

## 10. Recommended initial Door Three stack

The preferred early Door Three language stack is:

### A. Canon query layer

Allows a user or agent to request:

* what changed,
* what recurred,
* what is currently stable,
* what predictions are supported,
* what canonical memory is relevant.

### B. Symbolic binding layer

Attaches:

* stable symbolic tokens,
* external labels,
* human/agent-facing references,
* optional domain-specific naming.

### C. Language adapter

Maps:

* canon + prediction + symbolic binding
  into
* deterministic textual summaries,
* structured explanations,
* constrained query responses.

### D. Later generative layer

Only after the above is stable:

* richer phrasing,
* conversational flexibility,
* agent-native discourse,
* broader symbolic composition.

## 12. Rules to preserve legitimacy

### Rule 1 — Language is downstream of canon

No language output should outrank the canonical object it describes.

### Rule 2 — Symbolic handles do not create truth

A name, label, or token does not make a structure real.
It only references a structure that has already been stabilized.

### Rule 3 — Translation should be traceable

Where feasible, language should be traceable back to:

* canonical object,
* promotion evidence,
* provenance-preserving runtime support.

### Rule 4 — Expressiveness should lag legitimacy

Expressive richness is valuable, but it should come after basic structural honesty.

### Rule 5 — User convenience must not collapse authority boundaries

A simpler interface is good.
A misleading interface is not.
