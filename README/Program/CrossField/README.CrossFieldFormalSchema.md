# Dynamical Memory Engine — Cross-Field Formal Schema / Disclosure

## Status

This document is a supporting disclosure and terminology-accounting note for the Dynamical Memory Engine (DME).

It is not constitutional authority.

It does **not** override:

- `README.MasterConstitution.md`
- `README.WorkflowContract.md`
- `README.DoorOneProbeSurfaceNexus.md`
- `README.IntegrationBridge_to_DME.md`

Its purpose is narrower:

- disclose which established disciplines DME touches,
- state how DME relates to those disciplines,
- reduce accidental reinvention of already-established mechanics,
- reduce accidental overclaiming of novelty,
- anchor DME language to existing fields where appropriate,
- clarify where DME uses local terms for project-specific reasons,
- prepare a clean cross-field vocabulary for Door One and projected Door Two / Door Three discussion.

This document is a language, legitimacy, and disclosure surface.

It is not a proof of equivalence to any one field.

---

## 1. Constitutional posture

DME operates under a strict layer discipline.

For active Door One work:

- measurement remains below structure,
- structure remains below canon,
- runtime remains below truth claims,
- read-side interpretation remains below promotion,
- semantic interpretation must not silently rewrite structural artifacts.

Accordingly, this note does **not** authorize:

- ontology fusion,
- semantic uplift from structural similarity alone,
- novelty inflation,
- field capture claims,
- or canon claims derived from cross-field resemblance.

Its job is the opposite:

- disclose overlap honestly,
- preserve non-equivalences,
- and keep terminology accountable.

---

## 2. Why this note exists

DME now touches enough established disciplinary space that local project language can drift in two dangerous directions:

1. **reinvention drift**  
   where an already-known mechanism is redescribed as if it were entirely new,

2. **fusion drift**  
   where a useful analogy is overread as identity, equivalence, or proof.

This note exists to prevent both.

The rule is:

- reuse established field language where it already fits,
- disclose when DME is assembling known pieces into a new arrangement,
- disclose when a local DME term names a project-specific seam,
- and explicitly mark where DME is only analogous to a field rather than claiming equivalence.

---

## 3. Core disclosure rule

For every field DME touches, this note should answer five questions:

1. **What part of DME touches this field?**
2. **What established mechanisms or terms are already available there?**
3. **Is DME borrowing, extending, rearranging, or only analogizing?**
4. **What is DME explicitly not claiming?**
5. **What terminology should be preferred when speaking with that field?**

This keeps the project rigorous without forcing it to collapse into any one existing discipline.

---

## 4. Relationship types

Every entry in this schema should use one or more of the following relationship types.

### 4.1 Terminology reuse
DME uses a term that is already standard in a field and intends roughly the same meaning.

### 4.2 Method reuse
DME uses an already-established method or procedure from a field without claiming novelty in that method itself.

### 4.3 Structural analogy
DME is meaningfully analogous to a field concept, but is not claiming equivalence.

### 4.4 New composition of known pieces
DME combines already-established mechanisms or vocabularies into a new arrangement, workflow, or layered architecture.

### 4.5 Local project construct
DME uses a term or object that is specific to the project and not intended as a standard field term.

### 4.6 Projected-only contact
A field is not yet active in Door One, but is relevant to Door Two / Door Three projection.

---

## 5. Non-equivalence rule

Cross-field resemblance does not imply identity.

The following safeguards remain active:

- structural clustering is not semantic meaning,
- runtime memory is not canon,
- feature space is not automatically dimension space,
- query/retrieval is not understanding,
- read-side diagnostic usefulness is not ontological closure.

So when this document says “DME touches” a field, it does **not** mean:

- DME is reducible to that field,
- DME has solved the field,
- DME has replaced the field’s standard terminology,
- or DME is entitled to import downstream claims from that field automatically.

## 10. Terminology discipline rule

When speaking across fields:

1. prefer the established field term if DME means the same thing,
2. preserve the DME term if it names a genuinely local seam or governance distinction,
3. explicitly disclose when the relationship is only analogical,
4. explicitly disclose when a field term is being avoided to prevent ontology leakage.

Examples:

- keep `canon` as a DME governance term  
  because it is not just generic “memory”

- keep `basin` only with disclosure that it is currently a structural neighborhood term, not attractor truth

- keep `replay` as a DME experiment/reporting term  
  while anchoring it to repeated-run comparison language

- keep `distortion audit` as a project-local fidelity accountability term  
  while disclosing its overlap with distinguishability / resolution / compression concerns
