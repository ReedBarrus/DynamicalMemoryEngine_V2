## 1. Why this note exists

DME now has:

- anomaly detection,
- trajectory history,
- basin classification,
- identity law,
- semantic overlay discipline,
- and a cleaned structural/semantic split.

What remains missing is one bounded seam that can answer:

**What happened after a perturbation, and what structurally survived?**

Without this seam, the system risks:

- treating anomaly as if it already implies settlement,
- treating settlement as if it implies identity continuity,
- treating migration as if it implies lawful sameness,
- or forcing these judgments into later read-side narrative compression.

The semantic overlay cleanup already established that structural runtime must remain primary, and that interpretation must be late-bound, removable, and bounded by active query/claim posture. :contentReference[oaicite:2]{index=2} :contentReference[oaicite:3]{index=3}  
This seam exists to extend that same honesty into post-perturbation continuity reading.

One-line summary:

**The perturbation settlement seam classifies observed post-perturbation structural behavior over a bounded window; it does not restore, predict, or decide identity.** :contentReference[oaicite:4]{index=4}

---

## 2. Core posture

The perturbation settlement seam must remain:

- read-side first,
- structurally primary,
- bounded-window based,
- measurement-first,
- non-restorative,
- non-predictive,
- non-authoritative,
- and explicitly separate from identity preservation.

Settlement is a **behavioral classification** over already-observed trajectory behavior.  
Identity is a **continuity claim** about whether the same bounded question remains answerable under the same lawful path.  
These may co-occur, but neither implies the other. :contentReference[oaicite:5]{index=5}

Corollary rules:

- anomaly is not automatically settlement,
- settlement is not automatically identity continuity,
- migration is not automatically lawful continuity,
- and re-stabilization is not automatically same-object preservation.

## 3. Why this is read-side first

This seam should begin as a **report seam**, not a core runtime operator.

Reason:

- an `Op` name implies coupling into the execution graph,
- current DME law warns against prematurely rounding declared/visible seams into mechanized attainment,
- and the present goal is empirical study of real perturbation behavior before mechanizing a stronger continuity classifier. :contentReference[oaicite:7]{index=7} :contentReference[oaicite:8]{index=8}

Therefore the first lawful form is:

**`PerturbationSettlementReport`**

not yet:

**`PerturbationSettlementOp`**

Promotion to a true operator may be considered later only if:

- the bounded report outputs stabilize across real perturbation cases,
- the seam can satisfy a bounded mechanization closure gate,
- and it can be inserted without violating runtime structural primacy. :contentReference[oaicite:9]{index=9} :contentReference[oaicite:10]{index=10}

## 4. Seam role

The settlement seam sits conceptually between:

- `AnomalyOp`
- trajectory history / segment history
- basin classification
- and later continuity / identity interpretation

It does **not** replace any of those.

Its job is narrower:

- classify what happened **after** perturbation,
- over an explicit bounded window,
- using structural measurements first,
- then optional semantic posture second. :contentReference[oaicite:11]{index=11}

One-line role:

**Anomaly detects deviation. Settlement classifies post-deviation behavior. Identity audit decides whether the same bounded object still remains lawful.**

The smallest lawful input bundle is:

- anomaly event record

  - magnitude
  - timestamp

- active basin classification from `BasinOp`

This input bundle is sufficient for bounded observation of what happened after perturbation, and does not require prediction or restoration. :contentReference[oaicite:12]{index=12}

## 6. Structural outputs (primary)

Structural emissions must come first.

The report should emit, in this order:

1. **post-perturbation trajectory distance from pre-perturbation basin centroid**
   - raw scalar or bounded structural magnitude

2. **basin membership before and after**
   - basin IDs, not semantic labels

3. **transition structure similarity score**
   - pre/post transition-shape comparison

4. **invariance stress measurements**
   - deviation magnitudes by relevant V-class
   - measurement, not pre-declared violation

5. **window bounds**
   - `window_start`
   - `window_end`
   - `window_basis`

These are the primary structural outputs.  
Everything else must remain interpretive overlay above them. :contentReference[oaicite:13]{index=13}

### Structural emission rule

Use measured deviation, not declared violation.

Good:
- `v1_deviation_magnitude: 0.23`
- `v4_response_pattern_distance: 0.41`

Bad:
- `v1_violated: true`

Violation is an interpretation.  
Magnitude is a measurement.  
Interpretation belongs above the structural layer. :contentReference[oaicite:14]{index=14}

## 8. What the seam must not do

The settlement seam must **not**:

- restore trajectory,
- steer toward any attractor,
- smooth instability,
- predict future behavior,
- emit canon or promotional claims,
- or assert that settlement implies identity preservation. :contentReference[oaicite:17]{index=17}

Any implementation that says:

- “the structure will restabilize,”
- or uses settlement output to influence runtime trajectory behavior directly,
- or upgrades identity posture without explicit downstream review,

has crossed out of Door One observational honesty and into a higher-layer violation.

## 9. Smallest lawful window rule

Settlement classification must remain bounded by an explicit observation window.

### Lower bound
The window must be at least as long as the anomaly detection horizon required for the relevant deviation to be classable.

### Upper bound
The window must not cross a segment transition or rupture boundary.

### Required emission
Every settlement record must emit:

- `window_start`
- `window_end`
- `window_basis`

A settlement classification without explicit window bounds is structurally incomplete and must be treated as unresolved by downstream consumers. :contentReference[oaicite:19]{index=19}

This prevents the silent failure mode where the seam expands context until things “look stable.”

---

## 10. Lawful migration rule

Migration is not lawful merely because a trajectory moved and then settled.

Migration is lawful only when **all three** hold:

1. the bounded question remains answerable after migration,
2. invariance loss accounting is explicit,
3. identity posture downgrade is explicit.

That means lawful migration is not:
- “same thing but elsewhere,”
- or “probably conserved because it settled.”

It is:

- same bounded question still answerable,
- under explicitly changed structural conditions,
- with explicitly emitted downgrade posture. :contentReference[oaicite:20]{index=20}

Without those three, migration must be treated as degraded continuity or unresolved continuity, not lawful same-object carryover.

---

## 11. Settlement does not imply identity continuity

Not even probabilistically.

Settlement is:

- a behavioral observation about post-perturbation trajectory behavior within a bounded window.

Identity continuity is:

- a claim that the same bounded question remains answerable through the same lawful path under the same declared constraints and sufficient support. :contentReference[oaicite:21]{index=21}

A structure can:
- settle and fail identity continuity,
- preserve identity continuity during partial non-settlement,
- or remain unresolved on identity even when settlement looks strong. :contentReference[oaicite:22]{index=22}

Probability language here is unsafe because it encourages:
- “probably the same object”
instead of:
- “resolved,”
- “not resolved,”
- or “unresolved.”

That is a coherence-drift failure mode, not humility. :contentReference[oaicite:23]{index=23}

## 12. Identity break condition

The smallest failure condition that forces:

**`new_identity_class_required`**

is:

> the defining bounded query no longer resolves through the same lawful path.

This does **not** mean:
- same value required,
- same coordinates required,
- or same basin required.

It means:

- the same bounded question must still resolve,
- through the same class of constraint/support path,
- without requiring a different identity class to make the question answerable. 

### Key distinction

**Value change** may be compatible with same-object continuity.  
**Resolution path change** is not.

If the post-perturbation structure requires:
- different constraints,
- different support surfaces,
- or a different basin-dependent support path
to answer the defining query,

then the old identity claim is broken and:

**`new_identity_class_required`**

must be emitted. :contentReference[oaicite:25]{index=25}

This is the cleanest current identity-break rule for the settlement seam.

## 13. QueryOp dependency and open implementation bottleneck

This specification now depends on one unresolved implementation-concept seam:

**QueryOp must surface its own resolution behavior strongly enough that path equivalence is checkable.**

That means QueryOp may need to emit a structured record of:

- which constraints were active,
- which support surfaces were accessed,
- which basin or structural neighborhood was consulted,
- and what resolution path class was actually used.

Without that, “resolution path equivalence” remains conceptually strong but only partially checkable.

This is not yet a contradiction in the architecture.  
It is the next real formalization bottleneck between:

- declared identity law,
- and fully mechanized identity-path audit. 

One-line summary:

**The settlement seam is spec-ready, but QueryOp path surfacing is the next conceptual/mechanization bottleneck.**

## 14. Merge relation

Settlement classification must remain separate from Merge classification.

Merge still needs to distinguish at least four cases:

- lawful merge
- basin migration
- stitched false whole
- unresolved competing structures

A trajectory that settled after migration is not automatically eligible for merge continuity with pre-migration or adjacent structures.  
If the bounded question resolves differently merged vs separate, merge must fail closed. :contentReference[oaicite:27]{index=27}

Settlement therefore informs merge review, but does not decide merge lawfulness by itself.
