# Dynamical Memory Engine — Rebuild Posture

## Status

This document defines the active rebuild posture for the current DME repo clone.

It is a supporting rebuild-governance note.

It does **not** override:

- `README_MasterConstitution.md`
- `README_WorkflowContract.md`
- `README.StructuralIdentityLaw.md`
- `README.DeclaredVsMechanizedAudit.md`
- `README.MechanizationClosureGate.md`
- `README.PacketWorkflowChain.md`
- `README_DoorOneDevelopmentalOutline.md`

Its purpose is narrower:

- define why this repo clone exists,
- define what was intentionally not preserved from the prior implementation stack,
- define the governing rebuild floor,
- define the anti-hybridization posture for new implementation,
- and establish the current rebuild as a fresh substrate rather than a legacy continuation by default.

This note governs **rebuild posture** only.

It does **not** govern:

- operator contracts by itself,
- future canon posture,
- later product packaging,
- or full implementation detail.

---

## 1. Why this repo exists

This repo clone exists because the prior implementation substrate was no longer trustworthy as a clean basis for continued development.

The problem was not only isolated bugs or bad helpers.

The deeper problem was accumulated coupling between:

- structural artifacts,
- support/accounting objects,
- diagnostics,
- read-side review posture,
- and later interpretive or mixed surfaces.

That coupling made bounded cleanup insufficiently trustworthy over time.

This rebuild therefore exists to create a fresh implementation substrate under explicit lane, floor, and regime law.

One-line summary:

**This repo is a rebuild substrate, not a preservation shell for the prior runtime.**

---

## 2. Core rebuild rule

**Nothing from the prior runtime or operator stack is preserved by default.**

Corollary rules:

- legacy existence is not admission,
- prior usefulness is not admission,
- prior implementation effort is not admission,
- and compatibility with old mixed artifact families is not a rebuild requirement unless explicitly re-admitted.

If an old object, helper, runtime seam, or operator behavior is to return, it must be reintroduced only through explicit admission against the current contracts.

---

## 3. What was intentionally not preserved

This rebuild intentionally does **not** preserve the old implementation stack as active code truth.

That includes, where applicable:

- old runtime scaffolding,
- old operator implementations,
- old mixed artifact families,
- old helper clusters,
- old workbench/HUD/viewer coupling,
- old substrate/review/report cohabitation,
- and old downstream seams that relied on mixed artifacts.

This does **not** mean all prior work was meaningless.

It means prior implementation is no longer treated as an automatically lawful substrate.

---

## 4. What is preserved

This rebuild does preserve the active law stack and newly stabilized rebuild architecture.

That includes:

- constitutional notes,
- workflow law,
- structural identity law,
- anti-overclaim/mechanization law,
- packet workflow law,
- developmental pressure order,
- regime framing,
- lane law,
- floor contracts,
- and admission criteria notes created for the rebuild.

One-line distinction:

**Law is preserved. Legacy implementation is not preserved by default.**

## 6. Regime posture

The repo now recognizes three broad regimes of information:

### A. TemporalRegime
Measurement admission and direct temporal carriers.

Examples:
- live stream
- `.wav`
- sensor output

### B. DerivedRegime
Already-derived but still structural objects.

Examples:
- transformed arrays
- compressed signatures
- basin candidates
- anomaly slabs
- structural descriptor tables

### C. SymbolicRegime
Interpretive, review, query, and overlay objects.

Examples:
- semantic overlay packets
- review matrices
- language-kernel frames
- canon-candidate dossiers

At present, active implementation rebuild is occurring only in **TemporalRegime**.

The existence of the other regimes does not widen the active implementation scope.

---

## 7. Anti-hybridization rule

This rebuild must not silently hybridize new floor contracts with legacy object families or legacy runtime assumptions.

Forbidden rebuild behaviors include:

- reusing old mixed artifacts because they are convenient,
- reintroducing support/accounting into primary objects,
- reintroducing review/interpretive posture into structural seams,
- silently adapting old helpers to new floors without explicit admission,
- or preserving old pipeline identity in naming while changing the underlying law.

If a legacy seam is reused, it must be explicitly reclassed and re-admitted under the current rebuild law.

---

## 8. Admission rule for reuse

Any reuse from older work must answer all of the following before admission:

1. what exact object, helper, or seam is being reused?
2. what current lane/regime/floor does it belong to?
3. what old coupling does it carry?
4. what must be removed before admission?
5. why can this not be rebuilt more cleanly from the current floor law?

If those cannot be answered clearly, the item remains out.

---

## 9. Current implementation order

The current preferred build order is:

1. rebuild posture and law surface
2. TypeScript object families
3. runtime validators
4. `IngestOp`
5. `ClockAlignOp`
6. `WindowOp`
7. `TransformOp`
8. post-transform admission only after explicit tests

This preserves one lawful move at a time.

## 10. Source-of-truth posture

For this rebuild:

- the active README law stack defines meaning and boundary posture
- the new floor contracts define implementation law
- new code must conform to the rebuilt floor rather than preserve old runtime shape
- any old notes that still describe the prior implementation as active should be compressed, archived, rewritten, or removed

This repo should not carry two competing active pipeline stories.

---

## 11. Non-goals

This rebuild posture does **not** require:

- preservation of old runtime continuity,
- backward compatibility with legacy mixed artifacts,
- immediate rebuild of downstream operators beyond `P3`,
- or early admission of symbolic/review architecture.

Those remain deferred.

---

**This repo is a fresh DME rebuild substrate governed by the new regime, lane, floor, and admission contracts, with legacy implementation excluded by default and reusable only through explicit re-admission.**
