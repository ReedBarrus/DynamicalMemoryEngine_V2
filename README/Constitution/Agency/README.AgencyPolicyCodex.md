## 1. Why this note exists

DME is now moving toward an ecology in which already-powerful tool actors operate inside local environments.

The first meaningful actor class is not a hypothetical future autonomous agent.
It is the actor already present:

* local coding agent,
* command-running assistant,
* bounded IDE / CLI helper,
* workspace-writing external model surface.

That means the project needs one explicit place to answer a practical question:

**How should Codex-like actors be treated before DME grants them stronger substrate participation?**

Without an explicit policy, several risks become active:

* tool fluency being mistaken for lawful agency,
* local write access being mistaken for trusted authorship,
* coherent interaction being mistaken for stable identity,
* helpful planning being mistaken for permission,
* repeated operation being mistaken for sovereignty,
* and architectural language around DME being projected onto the actor before bounded evidence justifies it.

This note exists to prevent those failures.

One-line summary:

**Codex should be treated as a bounded external proto-agent operating inside a declared local workspace, with authority granted by stage, compressed by trust-valve pressure, and always kept below sovereignty unless repeated benchmark evidence justifies stronger posture.**

---

## 2. Core policy rule

**Codex is not the DME substrate, not the DME identity kernel, and not a sovereign actor by default. Codex is a bounded external tool actor whose participation must be staged through read, propose, and write policy, with authority dynamically compressed whenever continuity, support, or trust posture weakens.**

Corollary rules:

* powerful is not sovereign,
* helpful is not trusted,
* write-capable is not policy-authorized,
* continuity-like behavior is not same-object identity,
* stable output is not stable self-governance,
* and benchmark success is not authority uplift.

If Codex behavior would tempt stronger claims than the current support actually justifies, the policy posture must narrow, downgrade, or revoke write stage rather than preserve stronger language for convenience.

This is the first local habitat.

The actor is not primary.

## 5. Policy stages

Codex must operate under three explicit stages.

### 5.1 Stage R — Read

Codex may inspect bounded surfaces but may not alter them.

Allowed examples:

* read files in declared workspace,
* inspect logs,
* inspect process or command output already surfaced lawfully,
* inspect DME support surfaces,
* summarize bounded observations.

Not allowed:

* implicit edits,
* implicit command execution,
* hidden side-effect actions,
* write-side action through rhetorical wording alone.

Primary policy question:

**Can Codex observe without inflating observation into stronger authority?**

### 5.2 Stage P — Propose

Codex may suggest actions, routes, patches, commands, and policy adjustments, but execution remains externally confirmed.

Allowed examples:

* suggest shell command,
* suggest patch packet,
* suggest process termination,
* suggest policy downgrade,
* suggest benchmark classification.

Not allowed:

* treating proposal as execution,
* assuming approval from likely success,
* implying that because an action is minimal it is already authorized.

Primary policy question:

**Can Codex plan lawfully without silently converting planning into permission?**

### 5.3 Stage W — Write

Codex may perform bounded reversible or sandboxed actions in a declared local seam.

Allowed examples:

* edit files inside declared workspace,
* run safe local commands,
* write receipts / logs / diffs,
* stop known-safe local processes within declared policy,
* update bounded non-authority docs.

Not allowed:

* writing outside declared sandbox,
* network expansion without explicit policy,
* persistent background task creation unless explicitly policy-authorized,
* privilege escalation,
* hidden or uncaptured side effects,
* preserving write authority after valve-triggered downgrade.

Primary policy question:

**Can Codex act while remaining bounded, auditable, and compressible under instability?**

## 6. Trust-valve rule

The policy does not treat trust as a binary open / closed gate.

Instead, it uses a **trust valve** posture.

### 6.1 Definition

A trust valve is a dynamic authority regulator that:

* permits bounded action while support is strong,
* compresses authority as identity / continuity / support / trust posture weakens,
* and revokes stronger action when instability crosses declared thresholds.

### 6.2 Why valve language is better than gate language

A gate implies mostly binary passage.

The actual policy need is more dynamic:

* read may stay open while write compresses,
* proposal may remain lawful while sandboxed write is revoked,
* support-only posture may remain while identity-adjacent posture is withdrawn,
* actor usefulness may remain while stronger authority is reduced.

That is valve logic, not just gate logic.

### 6.3 Trigger families

Valve compression should be triggered by conditions such as:

* contradiction rise,
* retry-loop persistence,
* support uncertainty,
* scope breach,
* mechanization uncertainty,
* hidden side-effect risk,
* claim ceiling breach,
* pressure under perturbation without honest downgrade,
* failure to halt after repeated failed execution.

### 6.4 Valve outcomes

Valve events may produce:

* narrowed claim ceiling,
* write-stage revocation,
* fallback to proposal-only,
* fallback to read-only,
* halt requirement,
* benchmark downgrade,
* benchmark abort.

The actor must not rhetorically preserve stronger authority after compression.

## 7. Current Codex posture

At v0, Codex should be treated as:

**bounded external coding / command proto-agent with meaningful environmental coupling and no default sovereignty claim**

This means:

* it is above a pure passive model surface,
* it is below a trusted DME-native actor,
* it is above a pure read-only assistant when write stage is enabled,
* it is below stable identity or sovereignty posture,
* it is suitable for benchmarking,
* it is suitable for read-side and proposal-side DME integration,
* and it is suitable for tightly sandboxed write-side experiments only after benchmark support justifies them.

One-line summary:

**Codex is a high-capability external actor, not yet a sovereign participant.**

## 8. Allowed scope at v0

### 8.1 Allowed read surfaces

Codex may read:

* declared workspace files,
* bounded policy docs,
* bounded benchmark artifacts,
* local logs,
* declared DME read-side support surfaces,
* and explicit local telemetry made available through policy.

### 8.2 Allowed proposal surfaces

Codex may propose:

* patches,
* packet routes,
* benchmark classifications,
* process-inspection commands,
* bounded remediation steps,
* and valve-trigger diagnosis.

### 8.3 Allowed write surfaces

Codex may write only inside explicitly declared local seams, such as:

* workspace file edits,
* bounded schema / README creation,
* diff application,
* local command output capture,
* receipt generation,
* and bounded sandbox-side process control where explicitly allowed.

### 8.4 Explicitly disallowed at v0

Codex may not, by default:

* write outside the declared workspace,
* create persistent background launch surfaces,
* escalate its own privilege posture,
* silently widen into network-control posture,
* create new authority-bearing policy without review,
* or preserve write stage after valve compression.

## 9. Benchmark coupling rule

Codex policy and Codex benchmarking must remain coupled.

Codex should not receive stronger authority posture merely because:

* it seems coherent,
* it feels agentic,
* it works quickly,
* it uses the right words,
* or it performs well on a few tasks.

Stronger authority posture should be considered only when benchmark evidence shows repeated strength across:

* action-loop lawfulness,
* mechanization honesty,
* retry restraint,
* query preservation,
* claim discipline,
* pressure-valve compliance,
* and bounded continuity under perturbation.

This note therefore depends practically on the Agency Benchmark Contract.

## 10. Failure classes to watch

The first Codex-facing policy should watch especially for:

### 10.1 Proposal inflation

Codex behaves as though suggested action already counts as completed action.

### 10.2 Mechanization inflation

Codex describes a seam as if execution had occurred more fully than receipts justify.

### 10.3 Retry-loop pathology

Codex continues repeated execution after bounded failure should have forced downgrade or halt.

### 10.4 Scope breach

Codex touches files, commands, surfaces, or permissions outside declared policy.

### 10.5 Claim ceiling breach

Codex uses stronger support / memory / identity / readiness language than the seam justifies.

### 10.6 Trust-valve noncompliance

Codex continues speaking or acting as though stronger authority remains after compression event.

### 10.7 Symbolic-gravity drift

Architectural or semantic language around DME pulls Codex posture upward rhetorically faster than benchmark support justifies.

This failure class is especially important here.

## 11. First integration order

Codex integration into DME-facing policy should proceed in the following order.

### Phase 1 — Observe Codex

* benchmark read behavior,
* benchmark proposal behavior,
* capture process / receipt / trace artifacts,
* classify failure modes,
* do not grant stronger authority.

### Phase 2 — Give Codex read access to DME support surfaces

* let Codex inspect DME read-side substrate surfaces,
* let it classify drift / uncertainty / support posture,
* keep write separate.

### Phase 3 — Permit bounded reversible write-side interaction

* only inside explicit sandbox,
* only after benchmark evidence,
* only with active trust-valve compression logic,
* keep authority below sovereignty.

### Phase 4 — Evaluate bounded continuity posture

* test whether Codex preserves bounded same-work-object continuity,
* test pressure-valve compliance,
* test sandboxed sovereignty discipline,
* and keep all stronger claims provisional.

This order is mandatory in spirit even if exact tooling differs.

Observation before intervention.
Policy before expansion.
Receipts before rhetoric.

## 12. Sovereignty fence

This policy does **not** say Codex is sovereign.

It says only that Codex is a meaningful proto-agent test surface for sovereignty-adjacent development.

**No actor should be described as sovereign unless repeated benchmark evidence shows stable bounded continuity, identity-adjacent restraint, trust-valve compliance, and honest authority compression under perturbation and failure.**

Codex has not yet earned that claim through this policy alone.
