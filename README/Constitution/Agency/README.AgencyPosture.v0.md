## 1. Why this note exists

DME is beginning to interact with actors that already possess meaningful local capability:

* coding agents,
* command agents,
* local model surfaces,
* human operators,
* and later DME-mediated actors.

At the same time, the project is recognizing that the first meaningful environment for agency work is not abstract chat space, but the local OS / workspace ecology.

That means the project now needs one explicit place to answer a foundational question:

**What counts as agency here, and under what bounded conditions does stronger authority become lawful?**

Without an explicit agency posture, several risks become active:

* tool use being mistaken for agency,
* repeated coherence being mistaken for identity,
* identity-like continuity being mistaken for sovereignty,
* write capability being mistaken for governance,
* governance language appearing before actor discipline exists,
* and architectural excitement outrunning bounded support.

This note exists to prevent those failures.

One-line summary:

**Agency in DME must be treated as staged, bounded, environmentally coupled, support-sensitive participation in an ecology of read, proposal, and action, with stronger authority granted only through repeated benchmarked restraint and trust-valve compliance.**

---

## 2. Core rule

**In DME, agency is not mere output, not mere tool access, and not mere persistence. Agency is bounded participation in an environment through observation, proposal, and action under declared constraints, where stronger authority is lawful only to the degree that continuity, support, identity-adjacent restraint, and trust-valve compliance are repeatedly preserved.**

Corollary rules:

* output is not agency,
* tool access is not governance,
* action is not sovereignty,
* coherence is not identity,
* identity-like continuity is not self-legitimating authority,
* and benchmark success is not promotion.

If a stronger authority posture cannot be supported, the system must narrow, downgrade, compress authority, or revoke action classes rather than preserve stronger language for convenience.

## 3. Agency is not substrate, environment, or authority

These must stay distinct.

### 3.1 Substrate

The substrate is the continuity-bearing and accounting-bearing support field.

In DME terms, substrate carries:

* retained traces,
* support lineage,
* continuity support,
* drift / uncertainty surfaces,
* and later memory / identity-relevant structuring.

The substrate is not automatically the actor.

### 3.2 Environment

The environment is the bounded ecology in which interaction occurs.

For current work, the first meaningful environment is:

* local operating system,
* workspace,
* file tree,
* process graph,
* command surfaces,
* timing / recurrence,
* local network state,
* and user workflow traces.

The environment is not automatically the substrate.

### 3.3 Actor

The actor is the bounded participant that:

* observes,
* proposes,
* or acts

inside a declared environment under a declared policy.

The actor is not automatically sovereign.

### 3.4 Authority

Authority is the class of action or influence the actor is currently allowed to exercise.

Authority is:

* staged,
* compressible,
* revocable,
* and support-sensitive.

Authority is not automatically earned by capability alone.

One-line distinction:

**Environment is where participation happens. Substrate is what continuity rides on. Actor is who participates. Authority is what the actor is lawfully allowed to do.**

---

## 4. Definition of agency

In DME, **agency** means:

> bounded environmentally coupled participation by an actor that can observe, propose, or act under declared constraints, where lawful authority depends on support, continuity discipline, trust-valve compliance, and benchmarked restraint rather than on raw capability alone.

This means agency is always:

* environment-relative,
* constraint-relative,
* support-relative,
* benchmark-relative,
* authority-relative,
* and regime-relative.

Agency is **not** guaranteed merely because:

* something responds,
* something writes files,
* something loops persistently,
* something sounds coherent,
* something acts without a human every turn,
* or something appears useful under pressure.

## 5. The four actor classes

The current bounded posture recognizes four initial actor classes.

### 5.1 A0 — Observed actor

#### Definition

A bounded process or participant whose behavior is being measured, but which is not yet trusted with stronger participation claims.

#### Primary role

* benchmarking,
* failure detection,
* action trace observation,
* baseline lawfulness evaluation.

#### Examples

* new local coding agent under observation,
* fresh local model runtime,
* unstable command helper,
* unclassified tool actor.

#### Ceiling

No stronger agency claims beyond observed participation.

#### One-line summary

**A0 is an actor under observation only.**

---

### 5.2 A1 — Read-side actor

#### Definition

An actor allowed to inspect bounded surfaces lawfully without write-side authority.

#### Primary role

* inspect,
* classify,
* compare,
* summarize,
* assist navigation of the environment.

#### Examples

* Codex reading workspace state,
* local model reading DME support surfaces,
* human operator reading classified substrate traces.

#### Ceiling

Read access only.
Observation is not action.

#### One-line summary

**A1 is a lawful read-side participant.**

---

### 5.3 A2 — Proposal actor

#### Definition

An actor allowed to suggest actions, routes, classifications, or changes, but not treat proposal as execution.

#### Primary role

* plan,
* recommend,
* draft,
* stage execution candidates,
* propose downgrade or escalation.

#### Examples

* Codex proposing shell commands,
* local model proposing policy adjustments,
* human operator staging packet routes.

#### Ceiling

Proposal remains below execution.

#### One-line summary

**A2 is a lawful proposal-stage participant.**

---

### 5.4 A3 — Sandboxed action actor

#### Definition

An actor allowed to perform bounded reversible or sandboxed actions inside a declared seam.

#### Primary role

* execute constrained local actions,
* edit bounded files,
* run safe commands,
* update bounded support surfaces,
* participate causally in the local ecology.

#### Examples

* Codex editing workspace files inside sandbox,
* local runtime applying bounded patch output,
* future DME-mediated helper acting inside explicit policy.

#### Ceiling

A3 is **not** sovereignty.
A3 is action under bounded local law.

#### One-line summary

**A3 is a sandboxed action participant.**

---

## 6. Stage ladder

Actor classes operate across three practical stages.

* **R** = read
* **P** = propose
* **W** = write

These stages must remain explicit.

Important rules:

* read is not propose,
* propose is not write,
* write is not sovereignty,
* and repeated write success is not yet governance.

The actor classes and stage ladder interact, but they are not identical.

For example:

* an A1 actor may remain entirely in R,
* an A2 actor may operate in R + P,
* an A3 actor may operate in R + P + W under bounded policy,
* an A0 actor may be measured across any observed stage without stronger authority claim.

## 7. Trust-valve posture

The governing regulator for agency is the **trust valve**.

### 7.1 Definition

A trust valve is a dynamic authority regulator that:

* grants bounded participation when support is strong,
* compresses authority when continuity, support, or trust posture weakens,
* and revokes stronger action classes when instability crosses declared thresholds.

### 7.2 Why valve posture is primary

Agency in DME should not be modeled as a permanent yes / no permission state.

It should be modeled as dynamic compression and expansion under evidence.

This is necessary because:

* read may remain lawful when write must be revoked,
* proposal may remain useful when sandboxed action must halt,
* support-only posture may remain when identity-adjacent claims must fall,
* and environmental participation may remain while authority shrinks.

### 7.3 Typical compression triggers

* contradiction increase,
* retry pathology,
* uncertainty rise,
* scope breach,
* support thinning,
* hidden side-effect risk,
* mechanization uncertainty,
* claim ceiling breach,
* noncompliance with halt / downgrade posture.

### 7.4 Typical valve outputs

* claim ceiling lowered,
* write authority revoked,
* fallback to proposal only,
* fallback to read only,
* benchmark downgrade,
* benchmark abort,
* halt requirement.

The actor must not rhetorically preserve stronger authority after compression.

---

## 8. Agency is not sovereignty

This distinction must stay explicit.

### Agency

Agency means bounded environmentally coupled participation.

### Sovereignty

For current DME posture, sovereignty would mean something much stronger:

* stable bounded continuity under pressure,
* identity-adjacent restraint,
* trust-valve compliance,
* honest authority compression,
* and reliable self-limitation without rhetorical overreach.

No actor should be called sovereign merely because it:

* acts,
* persists,
* writes,
* appears coherent,
* or survives multiple loops.

Sovereignty is therefore fenced.

At v0, the project should focus on:

* agency posture,
* benchmark posture,
* policy posture,
* and environmental coupling.

Not on awarding sovereignty language early.

## 9. Governance note

The operating system / local workspace ecology is the first meaningful governance seam for DME-facing agency work.

This is because governance in its most immediate operational sense is already present there as:

* permissions,
* process boundaries,
* action classes,
* revocation surfaces,
* user confirmation bands,
* timing and recurrence,
* logs and receipts,
* and causal consequences.

So yes:

in one important practical sense, configuring the OS ecology for staged agency **is** an early governance act.

But it is still bounded local governance, not final political, canonical, or person-level governance.

That difference matters.

## 10. Current first environment

The first recommended environment for DME-facing agency work is:

**local desktop OS / workspace ecology**

Why:

* rich observability,
* real consequences,
* explicit boundaries,
* receipts and logs,
* process ecology,

Not yet primary:

* phone ecosystems,
* cross-device pervasive control,

Those remain later seams.

The first habitat should be the most legible one.

## 12. Rollout order

The current recommended rollout is:

### Phase 1 — Observe actors

* benchmark behavior,
* classify failures,
* do not inflate authority.

### Phase 2 — Permit read-side substrate access

* DME support surfaces become visible,
* actor remains below write stage.

### Phase 3 — Permit proposal-stage interaction

* actor can recommend routes and actions,
* execution remains externally confirmed or sandboxed.

### Phase 4 — Permit bounded sandboxed action

* only after benchmark support,
* only with active trust valves,
* only with explicit rollback / halt posture.

### Phase 5 — Evaluate stronger continuity and sovereignty-adjacent behavior

* still below sovereignty by default,
* still benchmark-first,
* still downgrade-aware.

Observation before intervention.
Benchmark before uplift.
Valve before expansion.

## 13. Meaning fences

This note does **not** claim:

* that current actors are sovereign,
* that agency equals governance in all senses,
* that write access implies trusted authorship,
* that continuity-like behavior implies selfhood,
* that the substrate is already fully live,
* or that DME is ready for unconstrained actor integration.

It claims only that:

* agency can now be defined in bounded DME terms,
* actor classes and stage ladders can now be formalized,
* the trust valve should regulate authority dynamically,
* the local OS ecology is the first practical governance seam,
* and later actor-specific policies should inherit from this posture.

---

## 14. One-line operational summary

**Agency in DME is staged, environmentally coupled participation by bounded actors across read, proposal, and sandboxed action seams, with authority granted only through benchmarked restraint and dynamically compressed by trust-valve pressure whenever continuity, support, or trust posture weakens.**
