# Dynamical Memory Engine — Structural Identity Law

## Status

This note defines the bounded law by which DME determines when a structure may still be treated as the same bounded identity across emission, compression, retention, replay, reconstruction, query, and workflow transformation.

It is a supporting implementation-governance note.

It does **not** override:

* `README.MasterConstitution.md`
* `README.WorkflowContract.md`
* `README.RepoLayout.md`
* `README/Constitution/Mechanization/README.DeclaredVsMechanizedAudit.md`
* `README.DeterministicInvarianceThreshold.md`
* `README.DistortionAuditProtocol.md`
* `README.WorkflowMechanizationScope.md`

Its purpose is narrower:

* define structural identity in DME in one explicit place,
* unify mechanization honesty and deterministic invariance under one bounded law,
* prevent declared continuity from outrunning conserved identity,
* prevent displayed coherence from being mistaken for actual sameness,
* define the minimum tests required before identity language may be preserved,
* define lawful downgrade posture when identity can no longer be conserved honestly,
* support later replay, query, retention, workflow, and canon gating without inflating ontology.

This note is not:

* a canon policy,
* a truth theory,
* an ontology note,
* or a symbolic identity manifesto.

It is a bounded structural identity note.

---

## 1. Why this note exists

DME now has enough runtime, retention, replay, request, inspection, workflow, and canon-adjacent structure that one recurring failure mode has become visible across multiple seams:

**a surface can appear to preserve identity more strongly than the system can actually justify.**

This appears in several forms:

* declared capability being mistaken for mechanized capability,
* replay-shaped output being mistaken for true reconstruction,
* preserved packets being treated as if they preserve the same bounded object unchanged,
* query similarity being mistaken for identity continuity,
* workflow progression being treated as if staged transformation preserved the same object automatically,
* or repeated reuse being mistaken for trusted sameness.

DME already has strong partial controls for these dangers:

* the Declared vs Mechanized audit prevents surface clarity from being mistaken for actual mechanism,
* the Deterministic Invariance Threshold prevents retained or reconstructed continuity from outrunning conserved support,
* the Distortion Audit Protocol prevents flattened summaries from erasing meaningful distinctions.

What has been missing is one explicit unifying law for **structural identity itself**.

This note defines that law.

---

## 2. Core rule

**A structure retains identity in DME only while it can still answer the same bounded question under the same declared constraints with sufficient surviving support and actual mechanized basis, without inventing support or collapsing distinctions strongly enough to mislead later interpretation.**

Conversely:

**If the bounded question changes, the declared constraints drift, the support basis no longer survives, or the claimed continuity is only declared/displayed rather than mechanized, the system must not silently preserve identity language.**

It must instead:

* narrow,
* downgrade,
* defer,
* emit unresolved posture,
* admit a new identity class,
* or escalate for review.

One-line summary:

**Structural identity in DME is conserved only while the same bounded question remains answerable under the same declared lens, tier, and workflow constraints by a real mechanized support path with sufficient surviving distinctions; when that threshold fails, continuity language must narrow, downgrade, or split rather than silently pretending sameness remains intact.**

## 4. Definition of structural identity

In DME, **structural identity** means:

> the continued lawful ability to treat a structure, object, packet, replay surface, retained surface, query target, or workflow object as the same bounded object of inquiry because the same bounded question remains answerable under the same declared constraints with sufficient surviving support and without authority inflation.

This means structural identity is always:

* question-relative,
* constraint-relative,
* support-relative,
* lens-relative,
* tier-relative where relevant,
* and mechanization-relative.

Structural identity is **not** guaranteed merely because:

* names remain the same,
* shapes remain similar,
* summaries feel coherent,
* the object is still useful,
* a display looks complete,
* a packet still exists,
* or a query still returns something nearby.

---

## 5. The four identity conditions

A structure may be treated as the same bounded identity only if all four conditions below hold strongly enough for the question being asked.

### 5.1 Bounded question preservation

The same bounded question must still be answerable.

Examples:

* can this replay surface still support the same replay question?
* can this retained packet still support the same review claim?
* can this workflow object still answer the same closure question?
* can this query still preserve the same identity-comparison question?

Identity fails at the first level when:

* a repair question becomes a redesign question,
* a same-family comparison becomes broad resemblance,
* a replay support question becomes a general interpretation question,
* or a bounded development object quietly becomes a roadmap object.

If the question changes, the object is not silently “the same thing but extended.”
The identity claim has already weakened or broken.

### 5.2 Declared constraint preservation

The same declared constraints must still govern the object.

Relevant constraints may include:

* declared lens,
* retained tier,
* compression path,
* replay posture,
* workflow route class,
* scope boundary,
* source family boundary,
* authority posture,
* allowed downstream use.

Constraint preservation fails when:

* the lens changes without declaration,
* the retained tier no longer supports the same claim,
* the workflow route silently changes,
* a narrow source-family object is reused cross-family,
* or an inspection/review surface is treated as if it had stronger authority than declared.

### 5.3 Support survival

Enough support must survive for the same bounded question to remain answerable honestly.

Support may include, where relevant:

* lineage,
* source and stream identity,
* channel / modality identity,
* bounded time range,
* receipt references,
* declared lens context,
* retained-tier posture,
* support basis descriptors,
* audit visibility,
* acceptance evidence,
* required invariants for the active object class.

Support survival fails when:

* the retained surface no longer preserves enough lineage,
* required invariants no longer survive compression,
* important distinctions collapse into one convenient story,
* or the system would need to invent missing support to keep saying “same object.”

### 5.4 Mechanized basis sufficiency

The claimed identity continuity must rest on an actual mechanized path, not only on declaration, display, or prepared shape.

This means:

* a replay-shaped object is not yet a true reconstructed identity surface unless the reconstruction path is mechanized,
* a request object is not a fulfilled workflow state,
* a displayed closure gate is not a functioning gate,
* a packet candidate is not promoted canon,
* a declared workflow object is not a lawful identity carrier unless the bounded support path exists in mechanism.

Mechanized basis sufficiency fails when the system can describe the object more clearly than it can actually justify or perform it.

---

## 6. Identity is question-relative

Structural identity can only be evaluated relative to a **bounded question**.

Without a bounded question, identity becomes vague enough that almost any useful resemblance can masquerade as continuity.

Examples of lawful bounded questions:

* Can this retained surface still support provenance reconstruction for this run?
* Can this query still preserve same-family comparison under the declared lens?
* Can this replay request still support the declared replay posture honestly?
* Can this workflow object still govern the same closure decision?
* Can this candidate packet still support the same narrow trust-uplift claim?

Questions that are too broad should not be used for identity preservation.

Examples of overbroad questions:

* Is this basically the same thing?
* Does this still feel equivalent?
* Is this close enough to count?

DME requires the bounded question to be explicit enough that identity preservation can be audited.

---

## 7. Identity is not similarity

Similarity may contribute to support.
It does not by itself preserve structural identity.

A query result may be similar while failing to preserve:

* lineage,
* declared lens,
* retained-tier sufficiency,
* scope honesty,
* or the original bounded question.

Therefore:

* similarity is not identity,
* retrieval is not deterministic continuity,
* resemblance is not lawful equivalence,
* utility is not preservation.

If a path depends mainly on broad resemblance rather than bounded continuity under declared constraints, it must be named as a different dynamic class.

---

## 8. The five identity tests

The following five tests should be used whenever DME needs to decide whether identity language remains lawful.

### 8.1 Question preservation test

Question:

**Does the current object still answer the same bounded question?**

If yes:

* identity may still be conserved.

If no:

* the identity claim must narrow, split, or downgrade.

### 8.2 Constraint preservation test

Question:

**Does the current object remain under the same declared constraints?**

At minimum, inspect relevant declared constraints such as:

* lens,
* retained tier,
* workflow route,
* source-family scope,
* allowed-use scope,
* authority posture.

If these drift, identity continuity weakens even if the object still looks similar.

### 8.3 Support survival test

Question:

**Do the required invariants and support surfaces still survive strongly enough for the bounded question to remain answerable?**

If no:

* continuity language must be downgraded,
* and the surface must not claim more than the surviving support justifies.

### 8.4 Mechanization test

Question:

**Is the claimed continuity grounded in an actual bounded implementation path, or only in declared/displayed shape?**

If the mechanism is absent:

* the surface may remain useful,
* but it must not be treated as fully identity-preserving.

### 8.5 Distortion threshold test

Question:

**Has the current representation collapsed distinct structural realities strongly enough that later interpretation would be misled if identity language were preserved?**

If yes:

* the object has crossed the identity threshold at the current layer,
* even if it remains useful as a narrower or weaker support surface.

## 9. Identity failure modes

Structural identity most commonly fails in DME through five recurring modes.

### 9.1 Surface illusion

A declared or displayed surface is mistaken for real mechanized continuity.

Examples:

* replay-shaped object treated as reconstruction,
* gate-shaped object treated as working gate,
* request-shaped object treated as fulfilled protocol.

### 9.2 Continuity illusion

A retained, compressed, or replay-prepared surface is treated as if it preserved the same bounded identity question when it no longer does.

Examples:

* compressed support treated as full replay continuity,
* packet persistence treated as full identity preservation,
* derived digest treated as equivalent to receipts.

### 9.3 Question drift

The object answers a different question while preserving the old identity language.

Examples:

* same-object comparison drifting into broad similarity,
* repair object drifting into redesign object,
* bounded closure question drifting into roadmap expansion.

This is one of the deepest identity failures because it can feel coherent while silently changing what is being answered.

### 9.4 Constraint drift

The object is reused or interpreted outside the declared lens, retained tier, route class, or scope boundary.

Examples:

* same-lens object treated as cross-lens stable,
* narrow workflow object reused as general architectural authority,
* review surface reused as canon-like reference.

### 9.5 Support distortion

Important distinctions collapse or support is implicitly invented to preserve continuity language.

Examples:

* multiple transition realities compressed into one summary,
* query similarity treated as lineage continuity,
* scaffold treated as implemented capability,
* repeated utility treated as truth uplift.

## 10. Lawful downgrade postures

When structural identity can no longer be conserved honestly, DME should not force clarity for convenience.

Preferred downgrade postures include:

* `unresolved`
* `insufficient_support`
* `narrowed_scope`
* `degraded_identity`
* `declared_only`
* `displayed_only`
* `partially_mechanized`
* `new_identity_required`
* `review_required`
* `retained_tier_insufficient`
* `query_not_equivalent`

The exact posture depends on which identity condition failed.

The governing rule is:

**when the object cannot remain the same honestly, the system must weaken the claim rather than strengthen the rhetoric.**

---

## 11. Identity outcomes

A practical identity audit should produce one of the following broad outcomes.

### 11.1 Conserved identity

The same bounded question remains answerable under the same declared constraints with sufficient support and mechanized basis.

### 11.2 Conserved but narrowed identity

The object remains the same only under reduced scope, tighter use, or stricter declared conditions.

### 11.3 Degraded identity

Useful continuity remains, but not enough to preserve the old identity claim without qualification.

### 11.4 New identity required

The object now behaves as a different bounded object and should be renamed, split, or reclassified.

### 11.5 Unresolved identity

Support is too weak to justify either strong continuity or strong break.

Unresolved posture is lawful.
Overconfident continuity is not.

## 12. Relation to declared vs mechanized audit

The Declared vs Mechanized Audit answers one critical part of structural identity:

**does the claimed capability or continuity rest on real mechanism, or only on declared/displayed surface form?**

Structural Identity Law incorporates that audit but is broader.

A surface may be:

* mechanized yet no longer identity-conserving,
* or elegantly declared yet still not mechanized enough to carry identity lawfully.

Therefore:

* declared vs mechanized answers the **basis** question,
* structural identity answers the **sameness under bounded question and constraints** question.

Both are required.

## 13. Relation to deterministic invariance threshold

The Deterministic Invariance Threshold note provides the strongest continuity discipline for retention, replay, reconstruction, and query.

Structural Identity Law generalizes that posture across additional DME surfaces, including:

* workflow objects,
* request objects,
* closure gates,
* development packets,
* canon-adjacent claims.

Therefore:

* deterministic invariance is a critical special case of structural identity,
* especially for replay, retained tiers, and reconstruction questions.

This note should not replace the Deterministic Invariance Threshold.
It provides the broader law within which that threshold remains one of the most important operational tests.

## 14. Relation to distortion audit

The Distortion Audit Protocol helps determine whether the current representation is collapsing distinctions strongly enough to make downstream interpretation misleading.

Structural identity depends on this directly.

If distortion severity crosses the point where the same bounded question can no longer be answered honestly without flattening meaningful distinctions, the identity claim must weaken or fail.

Distortion therefore acts as one of the main stress detectors for structural identity failure.

---

## 15. Relation to workflow mechanization

Workflow mechanization introduces bounded development objects that move through staged transformation.

Those objects also require structural identity discipline.

A workflow object retains identity only while it preserves:

* the same bounded development question,
* the same object class,
* the same function,
* the same authority posture,
* the same lawful routeability,
* and enough mechanized support to justify its current progression state.

This means structural identity applies not only to runtime and replay surfaces, but also to:

* scaffold emissions,
* delta emissions,
* request emissions,
* audit objects,
* closure gates,
* regime records,
* and packet-chain objects.

Workflow progression must not silently preserve identity language when staged transformation has actually produced a different workflow object.

---

## 16. Relation to canon gating

Canon promotes bounded claims, not vibes.

Therefore a canon candidate should not preserve the same promoted-claim language unless structural identity remains conserved for the claim being proposed.

This is especially important for:

* claim scope,
* declared lens,
* support basis,
* allowed downstream use,
* governability,
* and challenge handling.

A canon claim that no longer preserves its bounded question or declared constraints should not remain “the same canon but slightly stretched.”
It should be:

* narrowed,
* contested,
* suspended,
* superseded,
* or revoked,

according to the relevant canon lifecycle rules.

---

## 17. Structural identity gradient

Structural identity is not always binary.

A useful practical gradient is:

* `fully_conserved`
* `bounded_conserved`
* `narrowed_conserved`
* `degraded`
* `unresolved`
* `broken`

### `fully_conserved`

The same bounded question, constraints, support, and mechanism all remain intact.

### `bounded_conserved`

Identity remains conserved clearly, but only for a narrow declared question.

### `narrowed_conserved`

Identity survives only under stricter scope or tighter use conditions.

### `degraded`

Useful continuity remains, but identity language requires explicit weakening.

### `unresolved`

The system cannot yet justify strong continuity or strong break.

### `broken`

The original identity claim can no longer be preserved honestly.

This gradient is optional as a reporting aid.
It should not become decorative inflation.

## 19. Decision rule

Preserve identity language only when all of the following are true:

1. the same bounded question remains explicit,
2. the same declared constraints remain active,
3. enough support survives for that question to remain answerable honestly,
4. the continuity claim rests on a real mechanized basis,
5. distortion has not crossed the point where later interpretation would be misled.

If any of the five fail, the object may still be useful.
But it must not silently keep the same identity claim.

---

## 20. One-line practical rule

**In DME, the same thing is only the same thing while the same bounded question remains mechanized, constrained, and supportable enough to be answered honestly without invented support or misleading compression.**

## Structural constraint vs semantic constraint

This section clarifies a distinction that becomes load-bearing as DME moves deeper into replay, retention, recurrence, cross-session continuity, and cross-family coherence work.

It is a supporting clarification section.

It does **not**:
- override the Master Constitution,
- replace the Declared vs Mechanized Audit,
- replace the Deterministic Invariance Threshold,
- replace the Distortion Audit Protocol,
- or create a new authority layer.

Its purpose is narrower:
- clarify what kind of constraint may actually preserve structural identity,
- clarify what kind of constraint may only describe or summarize it,
- prevent semantic coherence from being mistaken for preserved support,
- and keep later continuity, coherence-surface, and interpretive work from silently carrying stronger identity claims than the structural layer justifies.

### Why this clarification is needed

As DME develops, it becomes increasingly possible to derive useful summaries, alignments, overlays, correspondences, or coherence surfaces from retained and replayable structure.

Those derived surfaces may become:
- intuitive,
- legible,
- compressive,
- and operationally useful.

But usefulness does not by itself make them preservation-bearing.

Without an explicit distinction here, a serious drift risk appears:

- semantic summaries may begin to carry continuity claims that belong only to structural support,
- interpretive alignment may begin to masquerade as identity preservation,
- or cross-session / cross-family coherence may begin to feel stronger than the surviving support actually justifies.

This section exists to prevent that drift.

### Core rule

**Structural identity in DME may be preserved only by structural constraint with sufficient surviving support and mechanized basis. Semantic constraint may describe, summarize, align, compare, or expose structure, but it must not by itself carry identity, replay legitimacy, reconstruction legitimacy, or memory-bearing status.**

One-line form:

**Structural constraint may preserve; semantic constraint may interpret.**

### Structural constraint

For the purposes of this note, structural constraint means constraint that remains anchored to actual surviving support and a real mechanized basis.

Examples may include, where relevant:

- retained invariants
- bounded support traces
- lineage-preserving structure
- declared lens and tier constraints
- quantitative thresholded continuity bounds
- replay and reconstruction support conditions
- real transform-domain relationships
- retained structural signatures
- mechanized identity-support conditions

Structural constraint is the layer that may lawfully carry:

- bounded identity continuity
- replay legitimacy
- reconstruction legitimacy
- memory-support posture
- and later memory-bearing claims if separately justified

Structural constraint is therefore preservation-bearing.

### Semantic constraint

For the purposes of this note, semantic constraint means a derived or interpretive layer that helps describe, navigate, compare, summarize, or expose relationships among structural objects without itself being the surviving support basis.

Examples may include:

- labels
- summaries
- interpretive regime names
- coherence overlays
- cross-family correspondence descriptions
- qualitative clustering names
- derived relationship narratives
- user-facing simplifications
- semantic navigation handles

Semantic constraint may be:

- useful
- compact
- operationally meaningful
- and important for legibility

But semantic constraint is not automatically preservation-bearing.

Semantic constraint is therefore interpretive unless separately backed by structural support.

### Preservation-bearing vs interpretive

The governing distinction is:

#### Structural constraint may preserve when:
- the same bounded question remains answerable,
- the same declared constraints remain active,
- enough support survives,
- a real mechanized basis exists,
- and distortion has not crossed the threshold where later interpretation would be misled.

#### Semantic constraint may interpret when:
- it helps the operator understand,
- compare,
- align,
- summarize,
- or navigate structural objects,

but it must remain explicitly below the structural basis that actually justifies the continuity claim.

### What semantic constraint must not carry by itself

Semantic constraint must not be the sole carrier of:

- structural identity preservation
- replay legitimacy
- reconstruction legitimacy
- retained-tier sufficiency
- memory-bearing classification
- canon-candidate trust uplift
- cross-session identity continuity
- cross-family identity fusion

If any of those claims depend only on the semantic layer, the claim is too strong.

### Cross-session and recurrence rule

This distinction becomes especially important under recurrence and cross-session continuity.

Repeated recurrence may strengthen the system’s ability to detect, refine, or accumulate structural constraint.

That is lawful only when the accumulation remains anchored to structural support.

The system must not silently convert:

- repeated semantic resemblance
- repeated usefulness
- repeated naming stability
- or repeated interpretive coherence

into structural identity preservation.

Recurrence may strengthen constraint only when the retained support basis and declared constraints still justify that strengthening.

### Cross-family correspondence rule

Distinct source families may later produce useful correspondence or coherence surfaces.

Those surfaces may be real derived objects.

But they must not collapse source-family identity, provenance, or support conditions.

Therefore:

- correspondence is not identity
- coherence is not fusion
- alignment is not replay equivalence
- shared interpretation is not shared structural basis

A cross-family coherence surface may become a lawful derived object.
It must still remain below the structural supports from which it was derived.

### Audit rule

When evaluating whether a claim is being carried lawfully, ask:

1. What exact structural support survives here?
2. What declared constraints remain active?
3. What mechanized basis exists?
4. What part of this claim is being carried by structural support?
5. What part is only being carried by semantic summary or interpretation?

If the semantic layer is doing the preservation work, the claim is overstated.

### Removal test

Use the following practical test:

**If removing the semantic layer destroys the system’s ability to justify identity, replay legitimacy, reconstruction legitimacy, or memory-preservation posture, then those claims were not structurally grounded strongly enough.**

This is a practical anti-drift rule.

Semantic aids may improve access.
They must not secretly replace support.

### Relation to later interpretive surfaces

Future interpretive layers, coherence surfaces, or operator-facing continuity surfaces may still be valuable and lawful.

But they must remain explicitly:

- derived
- non-authoritative
- support-referencing
- and structurally subordinate

Their role is to expose or help navigate structure.

Their role is not to become the structure they expose.

### One-line review question

**Is this continuity or memory-related claim still carried by surviving structural support and mechanized basis, or has semantic coherence begun carrying more of the claim than the structure itself can honestly justify?**

### One-line summary

**In DME, structural constraint may preserve identity and memory-support posture when surviving support and mechanized basis remain sufficient; semantic constraint may interpret, summarize, and align that structure, but it must not by itself carry preservation claims.**
