# README.SourceFamilyWorkflowTypes.md

# Dynamical Memory Engine — Source Family / Workflow Types

## Status

This document defines the bounded source-family and workflow-type posture for DME pre-ingest handling.

It is a supporting implementation-governance note.

It does **not** override:

- `README.MasterConstitution.md`
- `README.WorkflowContract.md`
- `README.RepoLayout.md`

Its purpose is narrower:

- define the difference between source-family type and import/transport type,
- reduce naming drift around “real source,” “file import,” and related workflow labels,
- clarify what belongs to generic adapter normalization versus declared family/workflow preprocessing,
- preserve lawful pre-ingest handling below runtime authority,
- guide future source-family growth toward lawful, governable, and utility-driven usage.

This note is about **pre-ingest source/workflow posture**.
It is not a new runtime layer.
It is not a new ontology.
It is not a canon policy.

---

## 1. Why this note exists

As DME becomes more usable, source diversity increases.

The project now has or anticipates:

- synthetic generated traces,
- file-backed numeric and sampled imports,
- captured real-world signal families,
- derived traces from higher-order systems,
- future web/object lifecycle traces,
- future LLM runtime traces,
- and future live device/socket streams.

Without a clear source-family note, several kinds of confusion become likely:

- treating file type as if it were source meaning,
- treating “real source” as if it meant only live streaming,
- treating generic import capability as if it already implied a family-specific workflow,
- or pushing semantic interpretation into adapters rather than preserving lawful pre-ingest handling.

This note exists to prevent that confusion.

---

## 2. Core rule

**Source family is not the same thing as file format or transport path.**

A source family describes the origin/posture/workflow class of the signal or trace.

A file format or transport path describes how the data arrives.

Examples:

- a WAV file may carry:
  - synthetic exported audio,
  - captured real-world microphone input,
  - or a bounded replay cohort from an experiment

- a JSON file may carry:
  - a direct lawful ingest object,
  - a derived numeric trace,
  - or a bounded smart-tag lifecycle projection

- a live stream may carry:
  - synthetic samples,
  - real-world sensor data,
  - or derived external telemetry

The file or transport does not decide the family by itself.

## 4. Distinction rule: family vs format vs workflow

Three things must remain distinct.

### A. Source family

What kind of origin/posture the input belongs to.

Examples:
- synthetic generated trace
- real-world captured signal family
- abstract / arbitrary imported numeric trace
- derived LLM runtime metric family
- smart-tag lifecycle family
- presence trajectory family

### B. Format / transport

How the input arrives.

Examples:
- JSON
- CSV
- WAV
- socket stream
- browser microphone capture
- sampler flush
- API response
- local file selection

### C. Workflow / preprocessing posture

Any declared pre-ingest shaping or family-specific handling that remains below runtime authority.

Examples:
- known replay-phase labeling
- cohort-specific provenance shaping
- family-specific import metadata
- bounded extraction from larger recordings
- declared channel selection
- declared time-window or sequence preparation

The workflow posture may be family-specific.
It must still remain pre-ingest only.

## 5. The three early source-family classes

The current practical family split should be:

### 5.1 Synthetic source families

Definition:
Inputs generated by fixtures, deterministic generators, or simulated source paths.

Examples:
- `makeTestSignal`
- synthetic replay cohorts
- deterministic seeded traces
- generated amplitude/frequency/noise variants

What they are good for:
- control
- regression
- deterministic development
- operator/runtime hardening
- explicit perturbation design

What they are not:
- automatically lower value
- automatically “less lawful”
- automatically semantically weaker

A synthetic source family is lawful when it preserves provenance and enters the same ingest boundary honestly.

---

### 5.2 Real-world source families

Definition:
Inputs captured from real devices, real environments, or real measured phenomena, even when later replayed from files.

Examples:
- microphone-recorded WAV cohorts
- `daw_mic_sine_400hz`
- `daw_mic_input`
- later browser mic capture
- hardware sensor logs
- live environmental telemetry

Important clarification:
A real-world source family may still arrive through a file.

A WAV containing microphone capture is still a real-world source family input even though the transport is file-backed.

What makes it “real-world” is:
- provenance-bearing capture from real measurement conditions,
- not whether it arrives as a file versus a live stream.

What they are good for:
- usefulness evaluation
- real-device diagnostic questions
- provenance-bearing replay
- family/workflow robustness testing
- bounded same-family consultation and anchor testing

---

### 5.3 Abstract / arbitrary source families

Definition:
Inputs that are lawful measured or numeric traces, but are not inherently tied to one established physical cohort family.

Examples:
- generic JSON numeric traces
- CSV time series from unknown external systems
- bounded tool-call cadence traces
- latency or token-rate traces
- exported numerical process logs
- imported smart-tag lifecycle projections
- imported presence/update traces

These are “abstract” or “arbitrary” only in the sense that they are not yet stabilized into one declared DME real-world family.

They are still lawful sources if:
- provenance is declared,
- ordering is preserved,
- normalization is honest,
- and they remain below semantic authority.

---

## 6. Import format is not family meaning

The following formats should be treated as **generic import mechanisms**, not family names:

- JSON
- CSV
- WAV

These formats answer:

**How did the data arrive?**

They do not answer by themselves:

- what family the data belongs to,
- what declared workflow posture applies,
- whether the source is synthetic, real-world, or abstract,
- or whether the input should later support consultation/candidate workflows.

This means:

- WAV import is generic file-backed capability
- real-world audio family workflow is a declared source-family/workflow class
- JSON import is generic structure import capability
- a smart-tag lifecycle family may still arrive through JSON
- CSV import is generic tabular trace import capability
- a device telemetry family may still arrive through CSV

---

## 7. Workflow preprocessing posture

Some source families require more than generic format parsing.

This note allows **workflow preprocessing** or **family-specific pre-ingest processing** when it remains:

- explicit,
- provenance-preserving,
- reversible,
- bounded,
- and below runtime authority.

Examples:
- selecting channel 0 from a multichannel WAV
- tagging a known replay phase for a controlled cohort
- recording family-specific source metadata
- trimming to a declared bounded slice
- applying declared sequence preparation to lifecycle events
- preserving extraction context from a larger imported object

This is lawful only if it does **not**:
- assign truth,
- infer ontology,
- decide canon,
- hide unsupported assumptions,
- or bypass the ingest contract.

## 8. Family/workflow naming guidance

To reduce drift, use names like:

### Good
- `synthetic_signal`
- `real_world_audio_family`
- `file_import_generic`
- `derived_trace_family`
- `smart_tag_lifecycle_family`
- `presence_signal_family`
- `audio_real_family_workflow`
- `workflow_preprocessing`

### Avoid
- `real_source` when you actually mean “live stream only”
- `wav_source` when you mean a real-world audio cohort
- `json_family` when you really mean an abstract lifecycle/object trace
- `semantic_source` when you really mean bounded numeric/object projection
- `real_audio` when you really mean “known DME cohort workflow over WAV or live capture”

The naming rule is:
**prefer origin/workflow meaning over transport convenience.**

## 9. Family-specific workflow examples

### Example A — Generic WAV import

- format: WAV
- family: unknown or generic file-backed sampled trace
- workflow: decode, normalize, validate, ingest

This is generic adapter capability.

### Example B — Real-world audio family workflow

- format: WAV or live mic capture
- family: `daw_mic_input`
- workflow:
  - decode/capture
  - preserve cohort/source identity
  - preserve declared replay phase or experiment metadata where applicable
  - normalize to lawful ingest
  - pass through runtime

This is not just generic WAV import.
It is a declared real-world source-family workflow.

### Example C — Smart tag lifecycle import

- format: JSON
- family: smart-tag lifecycle family
- workflow:
  - parse declared lifecycle events
  - project to bounded ordered trace(s)
  - preserve provenance + extraction context
  - normalize to lawful ingest-ready payload
  - pass through runtime

This is not “JSON family.”
It is a smart-tag family arriving through JSON.

## 10. Utility-driven family growth rule

New source families should be admitted only when they satisfy all of:

### A. Lawfulness
They can normalize to the same ingest boundary honestly.

### B. Governability
They do not introduce hidden semantics or unstable preprocessing authority.

### C. Utility
They enable a real new diagnostic or operational use.

### D. Compression value
They reduce manual handling, repeated setup, or repeated re-establishment cost in a bounded way.

### E. Reversibility
They can be removed or fenced without corrupting the active runtime architecture.

This follows the development-pressure rule that new mechanisms should exist only when they unlock real capability, simplify the system, or improve observability/replayability honestly. :contentReference[oaicite:3]{index=3}

## 11. Relationship to candidates and canon

Source-family classification does **not** decide candidate eligibility or canon worthiness.

However, source-family posture may matter later when evaluating:

- replay-honest lineage,
- real-source support,
- mixed-source support,
- scope honesty,
- allowed reuse,
- and governability.

This means source-family notes may support later:
- consultation packaging
- candidate packet evidence basis
- review posture
- canon utility/governability

But family type itself does not uplift authority.

## 12. Recommended immediate implementation posture

For the current phase:

### Keep separate
- generic file import capability
- real-world source-family workflow support
- synthetic family support
- abstract/derived family support

### Clarify in the shell
Make it explicit when a selector refers to:
- a generic adapter/import mechanism
- versus a declared family/workflow class

### Prefer future names like
- `file_import_generic`
- `real_world_audio_family_workflow`
- `derived_trace_family`
- `smart_tag_lifecycle_family`
- `presence_signal_family`

### Do not yet overbuild
Do not create a giant ontology of source families too early.
Keep the family set small, explicit, and utility-driven.

## 14. One-line summary

Source family, format, and workflow are not the same thing: DME should treat synthetic, real-world, and abstract/derived source families as distinct provenance/workflow classes while keeping JSON/CSV/WAV as generic import mechanisms and preserving all family-specific preprocessing below runtime authority.
