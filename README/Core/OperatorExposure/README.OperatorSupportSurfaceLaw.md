## 1. Why this note exists

DME now has enough runtime, support, replay, visualization, and workflow pressure that one recurring confusion has become explicit:

**support has been easy to confuse with both primary exposure and semantic explanation.**

That confusion creates several risks:

- support/accounting replacing primary emitted output,
- support surfaces drifting into semantic summaries,
- counts and lineage being mistaken for structure itself,
- prose being smuggled in under the name of "support,"
- and operator-facing surfaces becoming harder to audit because primary, support, and semantic layers are no longer clearly separated.

This note exists to stabilize that seam.

One-line summary:

**Support is derived structural/accounting exposure. It is not primary emission, and it is not semantic closure.**

---

## 2. Core rule

**An operator support surface may expose derived structural, numeric, timing, threshold, lineage, and accounting-adjacent output around a seam, but it may not replace the seam's primary emitted output and may not drift upward into semantic, review, or readiness posture.**

Corollary rules:

- support is not primary exposure,
- support is not prose explanation,
- support is not semantic overlay,
- support is not review,
- support is not identity or memory claim by default,
- and support usefulness does not excuse primary-plane absence.

If a support surface becomes the operator's primary way of understanding a seam that has not been directly exposed, the support surface has already become distortive.

One-line summary:

**Support may assist inspection. It may not substitute for emission.**

---

## 3. Constitutional posture

This note operates under already-active inherited rules:

- runtime is not canon,
- query is not truth,
- substrate is not ontology,
- declared is not mechanized,
- displayed is not mechanized,
- replay-shaped is not reconstructed,
- structural exposure must remain direct, pre-semantic, and inspectable,
- operator planes must expose first-order emitted output before support, semantic, or review layers,
- semantic overlay must remain downstream and removable.

Accordingly, support in this note means:

- second-order or adjacent structural/accounting exposure,
- not semantic explanation,
- not review classification,
- not readiness posture,
- not a broad "helpfulness" layer,
- and not permission to smuggle prose into structural surfaces.

This note remains below canon.

## 4. Definition of an operator support surface

In DME, an **operator support surface** means:

> a bounded surface that exposes derived structural fields, accounting fields, timing fields, threshold fields, lineage fields, or other seam-adjacent numeric/categorical outputs strongly enough to assist inspection of a seam without replacing the seam's primary emitted output and without drifting into semantic or review closure.

This means an operator support surface is always:

- seam-relative,
- derived-output-relative,
- axis/field-relative,
- placement-relative,
- and mechanization-relative where a real execution path matters.

A support surface is **not** guaranteed merely because:

- it contains useful numbers,
- it displays counts,
- it exposes ids,
- it feels informative,
- or it is placed next to a structural plane.

One-line distinction:

**A support surface shows derived or adjacent fields about a seam. It does not stand in for the seam's first-order emission.**

---

## 5. Support is not primary exposure, semantics, or review

These distinctions must remain explicit.

### 5.1 Primary exposure

Primary exposure shows:
- what the seam emitted directly

Examples:
- samples
- aligned timestamps
- transform values
- anomaly field
- merge mapping

### 5.2 Support exposure

Support exposure shows:
- derived fields around the seam
- accounting-adjacent numeric/categorical fields
- seam-local instrumentation values
- retained lineage/threshold/timing descriptors

Examples:
- counts
- ids
- timing deltas
- min/max/rms
- threshold values
- window extents
- loss values
- retained tier
- receipt refs
- support basis fields

### 5.3 Semantic overlay

Semantic overlay shows:
- bounded interpretation
- descriptors that summarize or classify
- downgrade-aware semantic posture

### 5.4 Review

Review shows:
- classification
- routing
- acceptance/defer/narrow/escalate posture
- consultation/readiness posture

One-line summary:

**Primary shows emission. Support shows derived fields. Semantics interpret. Review classifies.**

---

## 6. The four support classes

The current bounded law recognizes four practical support-surface classes.

---

### 6.1 S0 - Instrumentation support

#### Definition

Numeric or tabular support that helps inspect seam-local emitted output.

#### Typical contents

- sample count
- min/max/rms
- timing values
- frame extents
- frequency axis info
- magnitude scale info

#### Main role

- instrument readability
- direct seam-local assistance
- measurement/accounting support

#### One-line summary

**S0 is local numeric instrumentation support.**

---

### 6.2 S1 - Structural-derived support

#### Definition

Derived structural quantities that are not themselves the primary emitted output but help inspect it.

#### Typical contents

- alignment deltas
- residual values
- compression loss metrics
- anomaly magnitudes
- merge region sizes
- retained signature counts

#### Main role

- expose second-order structural consequences
- help compare primary outputs
- stay below semantic closure

#### One-line summary

**S1 is derived structural support.**

---

### 6.3 S2 - Support/accounting support

#### Definition

Fields that preserve support traces, lineage, identifiers, thresholds, retained posture, or execution-local accounting.

#### Typical contents

- source id
- run id
- lineage handles
- receipt refs
- retention tier
- threshold ids
- upstream seam ids
- mechanization/accounting fields

#### Main role

- preserve auditable context
- support seam-local accounting
- help locate the payload in the chain

#### One-line summary

**S2 is lineage/accounting support.**

---

### 6.4 S3 - Cross-surface support

#### Definition

Support that relates more than one seam or plane while still remaining below review.

#### Typical contents

- pre/post comparison ids
- source-to-window mapping
- retained-vs-replay support links
- seam-local relation tables
- cross-plane reference indices

#### Main role

- relate planes without collapsing them
- preserve inspectable chain continuity
- remain below semantic "story" formation

#### One-line summary

**S3 is relation support across adjacent seams or surfaces.**

---

## 7. What support may lawfully contain

Operator support surfaces may contain, where relevant:

- numbers
- tables
- ids
- counts
- timing fields
- threshold fields
- alignment deltas
- window extents
- source/run references
- lineage handles
- retained-tier fields
- structural loss metrics
- anomaly magnitudes
- merge-size/count fields
- coordinate readouts
- field values selected from the primary plane
- other seam-local derived structural/accounting fields

Support may be:

- tabular
- scalar
- minimally geometric
- toggleable
- sidecar-local
- separate-plane-local

One-line summary:

**Support may be rich numerically. It must remain seam-local and non-semantic.**

---

## 8. What support may not lawfully contain

Operator support surfaces must not contain as support:

- semantic verdicts
- review posture
- readiness posture
- memory claims
- identity claims
- "same object" posture
- prose explanation
- "what the signal is doing" summaries
- human-helpful narrative labels
- class inflation language
- semantic overlay text disguised as metadata

If any of these appear, the surface has likely crossed into semantic or review territory and should be reclassified.

One-line summary:

**Support may be labeled numerically. It may not narrate.**

---

## 9. Placement rule

Support surfaces may lawfully appear in one of four places.

### 9.1 External numeric sidecar

A compact support panel adjacent to the primary plane.

Best for:
- local readouts
- selected values
- instrument fields
- current seam-local metadata

### 9.2 Separate support plane

A standalone plane or tab exposing support fields without crowding the primary plane.

Best for:
- larger tables
- lineage/accounting fields
- heavy support payloads

### 9.3 Derived inspection plane

A support-informed later plane that remains clearly secondary to the primary exposure plane.

Best for:
- comparison overlays
- residual maps
- relation graphs

### 9.4 Hidden/export-only support path

Support that is too dense for direct display but remains exportable and inspectable.

Best for:
- large support tables
- full receipt/linkage maps
- dense cross-surface indices

One-line summary:

**Support should be placed where it assists inspection without crowding primary exposure.**

---

## 10. No-replacement rule

A support surface becomes unlawful when any of the following occur:

- primary emitted output is absent but support is present
- support becomes the first visible layer of a seam
- support is used as evidence that the seam is inspectable when P0 is missing
- a support table is treated as "good enough" instead of building the primary plane
- support fields are overread as structure itself

Examples:

- H1/M1 counts without transform exposure
- retention/accounting table without retained signature exposure
- replay posture fields without replay-support relation exposure
- transform metadata without transform values

One-line summary:

**Support may accompany emission. It may not replace it.**

---

## 11. Label rule for support

Support labels are allowed, but only as numeric/categorical field labels.

Allowed:
- `sample_count`
- `window_N`
- `hop_N`
- `mean_delta_ms`
- `max_offset_samples`
- `retained_tier`
- `source_id`
- `run_id`
- `loss_percent`

High-risk and usually disallowed on support surfaces:
- `stable`
- `broken`
- `memory-bearing`
- `same object`
- `healthy`
- `returning`
- `ready`
- `complete`
- `resolved`

Those belong to semantic or review layers if they belong anywhere at all.

One-line summary:

**Support labels should name fields, not meanings.**

---

## 12. Numeric-first rule

Support should default to:

- numeric values
- categorical field values
- tabular rows/columns
- explicit units
- explicit ids
- explicit local descriptors

It should not default to:

- prose
- explanatory paragraphs
- semantic summaries
- hidden derived judgments

This rule exists because support becomes much harder to audit once it becomes verbal rather than field-based.

One-line summary:

**Support should remain field-shaped, not story-shaped.**

---

## 13. Relationship to the primary plane

For any seam, the lawful order is:

1. primary emitted output
2. support fields
3. derived inspection forms
4. semantic overlay if ever needed
5. review later if needed

This order should remain explicit.

If support becomes primary and primary becomes optional, inspectability has already weakened.

---

## 14. Null and failure posture for support

Support surfaces may lawfully show:

- zero values
- empty tables
- blank sidecar
- null rows
- unpopulated field shells
- explicit absent/unknown numeric states

Support surfaces must not react to missing data by producing semantic filler or review posture.

Examples of lawful support null/failure posture:

- `sample_count = 0`
- `run_id = null`
- empty row set
- blank support plane
- threshold unavailable
- alignment delta unavailable

Examples of unlawful fallback:

- "no meaningful structure found"
- "signal appears unresolved"
- "system could not interpret"
- "insufficient support for analysis"

Those are no longer support surfaces.

---

## 15. Mechanization rule for support surfaces

A support surface may be called mechanized only when:

1. the support fields actually exist in backend/runtime state,
2. the fields are reachable through a real execution path,
3. the surface renders actual fields honestly,
4. missing fields remain explicit,
5. support remains support rather than semantic/review substitution,
6. and the support surface does not overclaim the seam's primary inspectability.

Displayed is not mechanized.
A table shell is not mechanized.
A fake metadata panel is not mechanized.

One-line summary:

**A support surface is mechanized only when it renders real derived/support fields through a real path without pretending to be primary exposure.**

---

## 16. Support-chain rule

Support surfaces should remain traceable to the operator exposure-plane chain.

For every support field, it should be possible to say:

- what seam it belongs to
- whether it is S0 / S1 / S2 / S3
- what primary plane it supports
- what it must not replace
- and whether it is local, cross-surface, or later derived support

This is especially important during inventory, retirement, and unthreading work.

One-line summary:

**Every support surface should know what seam it serves and what primary plane it is not allowed to replace.**

---

## 17. Failure modes specific to support

Common support-surface failures include:

- support becoming the only visible output
- support labels drifting into semantic posture
- support rows becoming prose-heavy
- counts replacing structure
- lineage replacing replay/retained relation exposure
- timing/accounting panels being mistaken for inspectable operator output
- support fields from multiple seams being collapsed into one mixed table
- support being used to hide missing primary planes

These should be treated as real seam failures, not mere presentation mistakes.
