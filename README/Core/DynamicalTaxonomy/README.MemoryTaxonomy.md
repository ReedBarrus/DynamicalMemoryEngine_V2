## 1. Why this note exists

DME now has enough replay, retention, compression, reconstruction, workflow, and identity pressure that the project needs one explicit place to answer a recurring question:

**What counts as memory, and what only looks memory-like because something was retained, replayed, restated, or reused?**

Without a bounded memory taxonomy, several distortion risks become active:

- retention being mistaken for memory,
- lineage being mistaken for present applicability,
- replay-shaped surfaces being mistaken for usable memory,
- repeated utility being mistaken for durable memory,
- compressed support being mistaken for full memory preservation,
- and continuity language outrunning what the surviving support and mechanism actually justify.

This note exists to prevent those drifts.

---

## 2. Core rule

**In DME, memory is not mere retention. Memory is retained structure that remains reapplicable for a bounded question under declared constraints with sufficient surviving support and, where required, an actual mechanized reuse path.**

Corollary rules:

- stored is not automatically memory,
- retained is not automatically reusable,
- traceable is not automatically applicable,
- useful once is not automatically durable,
- replay-shaped is not automatically memory-bearing,
- compressed continuity is not automatically preserved memory.

If a structure cannot still answer the same bounded question honestly under its declared constraints, the system must not silently preserve the stronger memory claim.

It must instead narrow, downgrade, defer, or classify the surface more weakly.

## 4. Definition of memory

In DME, **memory** means:

> retained structure that remains reapplicable for a bounded question under declared constraints strongly enough that its continuity can still be used honestly rather than only displayed, preserved, or restated.

This means memory is always:

- question-relative,
- constraint-relative,
- support-relative,
- mechanization-relative where reuse requires a real path,
- and regime-relative under perturbation, compression, or boundary shift.

Memory is **not** guaranteed merely because:

- something was stored,
- something was retained,
- lineage still exists,
- a replay surface looks coherent,
- a summary is useful,
- a structure feels familiar,
- or a past state can be narrated.

## 5. Memory is not identity

Memory and identity are related, but not the same.

### Memory

Memory is a **support class**.

It carries, preserves, traces, or re-enables continuity for a bounded use.

### Identity

Identity is a **continuity class**.

It is the bounded structure whose continuity may be supported by memory.

Therefore:

- memory can remain while identity degrades,
- identity can fail while some memory support survives,
- and retained support is not automatically an identity-preserving memory.

One-line distinction:

**Memory carries reapplicable continuity. Identity is the bounded structure whose continuity may be supported by memory.**

## 6. The four memory classes

The current bounded taxonomy recognizes four primary memory classes.

---

### 6.1 M0 — Retentive memory

#### Definition

Raw preservation of prior structure, signal, state, or support.

#### Bounded question

**Was something retained at all?**

#### Constraint relation

Persistence constraint only.

#### Primary role

- storage,
- retained support,
- replay substrate,
- lineage basis.

#### Not enough for

- identity continuity by itself,
- present applicability,
- strong reuse claims,
- or memory-bearing status in the stronger sense.

#### Common examples

- preserved receipts,
- retained outputs,
- archived support surfaces,
- stored but unclassified structure.

#### Main failure modes

- corruption,
- detachment,
- residue without applicability,
- preserved but unusable surface.

#### One-line summary

**M0 is preserved structure.**

---

### 6.2 M1 — Lineage memory

#### Definition

Retained structure whose continuity to origin remains traceable.

#### Bounded question

**Can this current structure still be lawfully traced back to its prior origin or support path?**

#### Constraint relation

Persistence + provenance / lineage constraints.

#### Primary role

- auditability,
- continuity tracking,
- replay support,
- provenance-bearing retention.

#### Not enough for

- proving that the structure is still usable for the same bounded function,
- or proving that it still answers the same bounded question.

#### Common examples

- retained structures with intact receipts,
- traceable packet history,
- replay surfaces with preserved support basis,
- lineage-bearing workflow objects.

#### Main failure modes

- lineage break,
- forged continuity,
- provenance collapse,
- traceable but functionally dead structure.

#### One-line summary

**M1 is preserved and traceable structure.**

---

### 6.3 M2 — Operational memory

#### Definition

Retained, lineage-bearing structure that can still be successfully reapplied to answer the same bounded question under the same declared constraints.

#### Bounded question

**Can this structure still perform its intended bounded use honestly?**

#### Constraint relation

Persistence + lineage + bounded question preservation + declared constraint preservation + support sufficiency + mechanized reuse path where required.

#### Primary role

- bounded reuse,
- functional continuity,
- honest reapplication,
- memory-bearing use in the stronger DME sense.

#### What makes this important

This is the first class that fully deserves the name **memory** in the stronger operational DME sense.

#### Common examples

- support-bearing structures that still answer the same replay or review question,
- bounded continuity objects that remain usable under the same declared lens,
- retained workflow or runtime structures that still support the same local function.

#### Main failure modes

- question drift,
- constraint drift,
- support insufficiency,
- mechanization gap,
- apparent continuity without real reuse path.

#### One-line summary

**M2 is traceable structure that still works for the same bounded use.**

---

### 6.4 M3 — Adaptive memory

#### Definition

Operational memory that remains reapplicable across bounded perturbations or changed local conditions without silently changing the bounded question.

#### Bounded question

**Can this memory survive lawful variation and still answer the same kind of bounded question honestly?**

#### Constraint relation

Operational reapplication + perturbation tolerance + bounded adaptation + explicit scope preservation.

#### Primary role

- robust continuity,
- cross-session continuity,
- replay resilience,
- bounded adaptive reuse.

#### Not enough for

- unbounded generalization,
- scope inflation,
- silent cross-family reuse,
- or “works somewhere” overreads.

#### Common examples

- bounded reuse that remains valid under small local perturbation,
- structures that survive declared compression or moderate boundary shift without losing the original question,
- memory support that remains lawful across narrow context variation.

#### Main failure modes

- overgeneralization,
- adaptation beyond declared scope,
- false continuity,
- changed question disguised as robust memory.

#### One-line summary

**M3 is operational memory that survives bounded variation.**

## 7. Memory staircase

The four classes form a staircase:

- **M0** = retained
- **M1** = retained + traceable
- **M2** = traceable + reusable
- **M3** = reusable + perturbation-resilient

This staircase should remain explicit.

A higher memory class includes stronger claims than the class below it.

The system must not silently round weaker classes upward.

Examples:

- M0 is not automatically M1
- M1 is not automatically M2
- M2 is not automatically M3

---

## 8. Memory functions

Each memory class can be analyzed through three functions.

### 8.1 Persistence function

**What is kept?**

Examples:
- signal,
- structure,
- receipts,
- summaries,
- lineage handles,
- replay support.

### 8.2 Applicability function

**What bounded question can it still answer?**

Examples:
- was it preserved?
- can it be traced?
- can it still be used?
- can it still survive bounded variation?

### 8.3 Scope function

**Under what constraint regime can it still answer that question honestly?**

Examples:
- stable replay only,
- compression-bounded,
- perturbation-bounded,
- boundary-shift-bounded,
- or already ruptured.

This three-function view prevents memory from collapsing into mere storage language.

---

## 9. Memory regimes

Memory classes do not operate in a vacuum.

The same memory object may succeed or fail differently depending on the active constraint regime.

The current taxonomy recognizes five practical regimes.

---

### 9.1 R1 — Stable replay regime

Memory reapplication works under the same bounded question and nearly unchanged declared constraints.

Best fit:

- M1
- M2

Typical use:

- same-lens replay support,
- narrow continuity review,
- local bounded reuse.

---

### 9.2 R2 — Narrow perturbation regime

Memory survives small lawful changes while preserving the same bounded use.

Best fit:

- M2
- M3

Typical use:

- small variation in local conditions,
- bounded continuity under mild perturbation,
- near-family reuse with explicit preservation posture.

---

### 9.3 R3 — Compression regime

Memory remains partly useful after reduction, but support is weakened.

Best fit:

- M1
- M2 with downgrade posture

Typical use:

- compressed support,
- summary reuse,
- retained but weakened continuity.

Typical danger:

- compressed continuity mistaken for full memory preservation.

---

### 9.4 R4 — Boundary-shift regime

Memory is tested under changed interfaces, lenses, or observation boundaries.

Best fit:

- M2 or M3 if the same bounded question survives honestly,
- otherwise downgrade or failure.

Typical use:

- changed observation surface,
- changed replay boundary,
- changed workflow boundary,
- changed query framing.

Typical danger:

- boundary drift mistaken for adaptive memory.

---

### 9.5 R5 — Rupture regime

Memory no longer answers the same bounded question honestly.

Possible outcomes:

- residue,
- degraded support,
- narrower memory class only,
- new memory class required,
- or related identity break.

Typical danger:

- preserving stronger memory language for convenience.

---

## 10. Memory is not replay, usefulness, or compression alone

Several surfaces can look memory-like without qualifying for stronger memory claims.

### 10.1 Replay is not automatically memory

A replay-facing surface may still be:

- declared only,
- displayed only,
- support-trace limited,
- or useful without preserving operational memory.

### 10.2 Usefulness is not automatically memory

A structure may be useful for a new or broader question while failing the original bounded question.

That is not preserved memory.
It is a new use posture and may require reclassification.

### 10.3 Compression is not automatically memory preservation

Compression may preserve:

- a weaker support class,
- a lineage-bearing handle,
- a summary surface,

without preserving the stronger memory claim.

### 10.4 Residue is not memory-bearing

Retained residue can remain valuable as support.
It should not be casually called memory-bearing if it no longer supports bounded reuse honestly.

## 11. Memory-bearing vs support-only posture

A central Leg 3 distinction is:

### Memory-bearing

A structure supports a real bounded memory claim at its current class.

### Support-only

A structure remains useful as support, lineage, audit, or replay substrate, but does not justify the stronger memory claim.

### Residue

A structure remains retained but no longer supports even the stronger support-only posture for the old bounded question without explicit narrowing.

This distinction should remain explicit.

The system must not casually call something memory-bearing when it is only:

- support-bearing,
- lineage-bearing,
- review-only,
- replay-shaped,
- or retained residue.

## 12. Relation to structural identity law

Structural Identity Law answers:

**Is this still the same bounded object under the same question, constraints, support, and mechanized basis?**

This note answers:

**What kind of memory continuity, if any, does the surviving structure still carry?**

The relationship is therefore:

- Structural identity law governs sameness claims.
- Memory taxonomy governs support and reuse class.

A structure may:

- preserve M1 lineage memory while identity is degraded,
- preserve M2 operational memory while identity remains bounded-conserved,
- lose M2 while retaining M0 or M1,
- or preserve only residue after rupture.

Identity audit and memory audit should therefore remain related but distinct.

## 13. Relation to declared vs mechanized and closure gate

A structure must not be called stronger memory than the actual implementation path justifies.

Therefore:

- declared memory is not mechanized memory,
- displayed memory is not mechanized memory,
- prepared memory surface is not fulfilled memory function,
- replay-shaped memory is not automatically operational memory.

Where a reuse claim depends on a real bounded path, the claim must remain below full operational memory if the backend, reachability, rendering honesty, insufficiency posture, or downgrade posture are missing.

Memory language must not outrun mechanization.

## 17. Memory failure modes

Memory most commonly fails through the following modes.

### 17.1 Corruption

Retained structure is damaged and no longer preserves enough of the prior support.

### 17.2 Detachment

Structure is retained but severed from the support path that made it interpretable.

### 17.3 Lineage break

Traceability to origin no longer survives honestly.

### 17.4 Question drift

The structure is now being reused for a different bounded question while preserving the old memory language.

### 17.5 Constraint drift

The structure is being reused outside its declared lens, scope, tier, or regime while preserving the stronger memory claim.

### 17.6 Mechanization gap

A memory claim depends on a real bounded path that does not actually exist.

### 17.7 Overgeneralization

Adaptive-looking reuse silently widens scope and becomes false continuity.

### 17.8 Compression loss

Reduced structure no longer preserves enough distinction for the original memory claim.

## 18. Lawful downgrade postures

When stronger memory cannot be preserved honestly, preferred downgrade postures include:

- `support_only`
- `lineage_only`
- `retained_only`
- `compressed_support`
- `degraded_memory`
- `residue`
- `insufficient_support`
- `constraint_drift`
- `question_drift`
- `new_memory_class_required`
- `review_required`

The governing rule is:

**when a structure can no longer support the stronger memory claim honestly, the system must weaken the claim rather than preserve stronger memory language for convenience.**

## 19. Memory outcomes

A practical memory audit should produce one of the following broad outcomes.

### 19.1 Retentive only

M0 holds; stronger classes do not.

### 19.2 Lineage-conserved

M1 holds; stronger reuse claims remain below threshold.

### 19.3 Operationally conserved

M2 holds for the bounded question under declared constraints.

### 19.4 Adaptively conserved

M3 holds under a bounded variation regime.

### 19.5 Degraded memory

Useful continuity remains, but not enough to preserve the old class without explicit weakening.

### 19.6 Residual only

Retained material remains, but no longer supports the prior memory claim honestly.

### 19.7 Unresolved

Support is too weak to justify either strong continuity or strong failure.

## 21. Decision rule

Preserve a stronger memory claim only when all of the following are true strongly enough for the active bounded question:

enough structure is retained,
the relevant lineage still survives,
the same bounded question remains answerable,
the same declared constraints remain active,
enough support survives for honest reuse,
and any required reuse path is actually mechanized.

If not, the structure may still be useful.
But it must not silently preserve the stronger memory class.

## 22. One-line practical rule

In DME, memory is retained structure that remains honestly reusable for a bounded question under declared constraints; when that reusability weakens, the system must downgrade the memory claim rather than preserve stronger language by convenience.

## 23. One-line summary

The DME memory taxonomy distinguishes retentive, lineage, operational, and adaptive memory so retention, replay, compression, and reuse can be discussed with bounded continuity, explicit downgrade posture, and clear separation from identity claims.
