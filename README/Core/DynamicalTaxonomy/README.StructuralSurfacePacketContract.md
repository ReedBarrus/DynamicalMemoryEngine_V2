- prevent packet prose from silently shifting a structural task into semantic or review work,

## 1. Why this note exists

DME now has enough packetized workflow, helper support, surface-building pressure, and symbolic gravity risk that a recurring failure mode has become explicit:

**a packet can ask for structural exposure in intent while encoding semantic closure in wording, naming, deliverable shape, or acceptance criteria.**

This is dangerous because a structurally incorrect result may still look:

- useful,
- coherent,
- complete,
- inspectable,
- or professionally shaped.

The failure can occur before implementation begins.

Examples include:

- asking for a "viewer" and receiving a semantic dashboard,
- asking for "geometry" and receiving state labels,
- asking for "structure" and receiving counts plus summaries,
- asking for "raw comparison" and receiving review posture,
- or mixing structural, support, semantic, and review objectives inside one packet.

This note exists to force packet discipline at the exact seam where symbolic gravity first enters workflow propagation.

One-line summary:

**If a packet is supposed to produce structural exposure, the packet itself must remain structure-first.**

---

## 2. Core rule

**Any packet whose declared goal is structural exposure must specify the active surface class, active query class, active representation class, and explicit non-goals strongly enough that semantic, support/accounting, and review layers cannot silently substitute for first-order structure.**

Corollary rules:

- a structural packet is not a semantic packet,
- a structural packet is not a support/accounting packet,
- a structural packet is not a review packet,
- a structural packet is not a mixed-surface packet by default,
- and convenience wording must not be allowed to redefine the seam.

If a packet cannot preserve these distinctions, it must be:

- narrowed,
- split,
- renamed,
- reclassed,
- or deferred.

## 4. Definition of a structural-surface packet

In DME, a **structural-surface packet** means:

> a bounded workflow object whose purpose is to produce, repair, expose, export, or validate first-order numeric, geometric, or relation-bearing structure strongly enough that the active structural question can be directly inspected without requiring semantic mediation.

This means a structural-surface packet is always:

- question-relative,
- surface-relative,
- representation-relative,
- naming-sensitive,
- and authority-sensitive.

A packet is **not** a structural-surface packet merely because:

- it mentions "structure",
- it mentions "viewer",
- it references H1/M1 objects,
- it includes a chart,
- or it uses the word "geometric."

One-line distinction:

**A structural-surface packet produces direct exposure, not merely discourse about structure.**

## 5. Structural-surface packet classes

The current workflow should recognize four practical packet classes relevant to exposure-first work.

---

### 5.1 P-S0 — Raw structural exposure packet

#### Purpose

Produce or repair first-order structural exposure.

#### Typical outputs

- arrays
- bins
- coordinates
- transform-space plots
- geometric traces
- field values
- direct exports of declared-lens output

#### Typical questions

- show the raw bins
- expose the transform output directly
- render the relation-bearing field
- export numeric structure for comparison

#### Forbidden substitution

- counts
- semantic labels
- memory/identity posture
- review posture
- support/accounting dominance

#### One-line summary

**P-S0 packets are for direct structural exposure.**

---

### 5.2 P-S1 — Structural support packet

#### Purpose

Expose support/accounting surfaces adjacent to structure without claiming to be raw exposure.

#### Typical outputs

- handles
- counts
- lineage
- retained tier
- provenance traces
- packet/export metadata

#### Typical questions

- how many H1/M1 objects exist?
- what lineage survives?
- what support basis is attached?

#### Important rule

A P-S1 packet must not be misnamed or accepted as P-S0.

#### One-line summary

**P-S1 packets expose support about structure, not structure itself.**

---

### 5.3 P-S2 — Semantic overlay packet

#### Purpose

Attach bounded interpretation downstream of already-exposed structure.

#### Typical outputs

- sparse semantic posture
- descriptor-backed labels
- caution posture
- removable overlay layers

#### Important rule

A P-S2 packet is lawful only if P-S0 exposure is already real or in scope separately.

#### One-line summary

**P-S2 packets interpret exposed structure; they do not replace exposure.**

---

### 5.4 P-S3 — Review / classification packet

#### Purpose

Classify, evaluate, narrow, or route an object or surface.

#### Typical outputs

- review posture
- downgrade posture
- unresolved posture
- packet routing
- readiness blockers

#### Important rule

P-S3 packets must remain separate from P-S0 unless explicitly split into layered surfaces.

#### One-line summary

**P-S3 packets evaluate structure-adjacent objects; they do not expose structure first-order.**

---

## 6. Required packet declarations for P-S0 work

Any packet intended as P-S0 must explicitly declare all of the following.

### 6.1 Active bounded question

The packet must state the structural question explicitly.

Examples:

- What transform-space bins are present under the declared lens?
- What numeric relation changes across baseline, perturbation, and return?
- What field values or traces are directly exposed by this output seam?

Forbidden substitutes:

- show what the signal is doing
- help understand the structure
- visualize the memory behavior
- inspect the pattern

These are too broad and symbolically heavy.

### 6.2 Active surface class

The packet must explicitly declare:

- raw structural exposure surface

and must not leave the surface class implicit.

### 6.3 Active representation class

The packet must explicitly declare what kind of first-order output is required.

Examples:

- numeric arrays
- bins
- coordinates
- field values
- transform-space plot
- geometric trace
- relation-bearing heatmap

If this is not declared, helpers may substitute semantic or support layers.

### 6.4 Active query class

The packet must declare the lowest lawful query class.

Usually this will be closest to:

- Q0 — presence / observation query
- Q1 — structural query
- sometimes Q2 — continuity query

A P-S0 packet should not silently activate:

- Q3 support / lineage
- Q4 memory
- Q5 identity
- Q6 review
- Q7 consultation

unless explicitly split.

### 6.5 Explicit non-goals

The packet must explicitly say what is **not** being asked for.

Examples:

- no semantic labels
- no state summaries
- no support/accounting dominance
- no review posture
- no consultation or readiness
- no object counts as primary output
- no memory or identity claims

### 6.6 Export requirement

The packet must state whether raw output must be directly exportable.

For most P-S0 packets, the answer should be yes.

If raw output cannot be exported, structural exposure attainment should be treated as suspect.

---

## 7. Structural mode declaration

Every structural-surface packet should carry an explicit mode declaration.

### Required form

`ACTIVE_MODE: STRUCTURE`

Optional additional declarations:

- `SURFACE_CLASS: RAW_STRUCTURAL_EXPOSURE`
- `QUERY_CLASS: Q0` or `Q1`
- `REPRESENTATION_CLASS: BINS / ARRAYS / COORDINATES / FIELD / TRACE`
- `SEMANTIC_CONTACT: FORBIDDEN`
- `SUPPORT_LAYER: SEPARATE`
- `REVIEW_LAYER: SEPARATE`

This is not decorative.

It is a workflow anti-drift device.

One-line summary:

**If a packet is structure-first, the packet must say so explicitly.**

---

## 8. Naming rules for structural packets

Names are load-bearing and must be constrained.

### 8.1 Preferred packet titles

Prefer titles such as:

- expose raw transform bins
- render declared-lens output directly
- export numeric relation field
- plot baseline/perturbation/return bin arrays
- surface transform-space distribution

These preserve low symbolic gravity.

### 8.2 High-risk packet titles

Treat the following as drift risks for P-S0 work:

- make a viewer
- build an inspector
- visualize the memory
- show the state
- create a dashboard
- build a pattern explorer
- make an insight panel

These may be lawful in other packet classes, but not as P-S0 defaults.

### 8.3 Naming rule

If the title implies semantic usefulness, insight, or explanation more strongly than direct exposure, the packet should be reviewed and likely renamed before implementation begins.

---

## 9. Deliverable rules for P-S0 packets

A structural-surface packet should require deliverables that prove exposure directly.

### 9.1 Required deliverables

At least one of:

- raw numeric export
- direct bin array
- direct coordinate export
- field-value export
- transform-space plot
- geometric trace surface
- heatmap over declared axes

### 9.2 Optional supporting deliverables

May include:

- axis labels
- declared-lens labels
- minimal numeric descriptors
- separate support/accounting attachment
- separate semantic overlay attachment

### 9.3 Forbidden primary deliverables

These must not be accepted as the primary output of a P-S0 packet:

- object counts
- semantic state panels
- support/accounting summary panels
- review posture
- narrative explanation
- memory/identity labels
- mixed dashboards

### 9.4 Separation rule

If support, semantics, or review are also needed, they must be delivered as separate surfaces or separately declared downstream layers.

## 10. Acceptance criteria for P-S0 packets

A P-S0 packet is not complete merely because something visual was produced.

It is complete only when all of the following are true:

1. raw structure is directly inspectable
2. the active structural question can be answered from the output directly
3. semantics are not required
4. support/accounting is not primary
5. review posture is not primary
6. raw output is exportable where required
7. the surface name and packet title did not silently widen the seam

### 10.1 Minimum acceptance questions

- Can I see the bins / arrays / coordinates / field directly?
- If all labels are removed, does the structural output still answer the question?
- Is the surface still lawful if semantics are absent?
- Did counts or object classes substitute for raw structure?
- Is this surface actually raw exposure, or did it become support/review/semantic output?

If the answer to the first two is no, the packet should fail acceptance.

---

## 11. Structural packet anti-drift checks

Before implementation begins, ask:

1. What exact structure must be directly exposed?
2. What output form proves that exposure?
3. What semantic shortcuts might a helper choose?
4. What support/accounting substitutions might appear?
5. What words in the packet title or prose invite drift?
6. Does the packet accidentally ask for mixed surfaces?
7. Is the acceptance target checking exposure or just usefulness?

After implementation returns, ask:

1. Is raw structure present?
2. Is it primary?
3. Can it be exported?
4. Did semantics arrive before exposure?
5. Did support/accounting dominate?
6. Did the surface answer a different query class than the packet asked?

---

## 12. Failure routes

If a P-S0 packet returns a semantic, support, or mixed surface instead of direct exposure, the return must not be silently accepted.

It should be routed explicitly as one of:

- **revise** — packet stayed near the seam but failed exposure criteria
- **narrow** — packet mixed too many roles and should be reduced
- **split** — exposure and support/semantic/review should become separate packets
- **reclassify** — the result is actually P-S1, P-S2, or P-S3
- **defer** — raw exposure cannot yet be produced lawfully
- **escalate** — representation class or seam role is still unclear

A useful semantic output is not a lawful P-S0 completion.

---

## 13. Helper / agent restrictions

When a helper or agent is working under a structural-surface packet:

It must not:

- substitute labels for raw output
- substitute counts for geometric exposure
- substitute support/accounting for raw structure
- substitute review posture for direct inspection
- widen the query class silently
- choose names that imply stronger semantic closure
- cohabit raw exposure with semantic/review layers by default

It should prefer:

- direct arrays
- direct plots
- direct coordinates
- declared-lens output
- ugly but exposure-first naming
- separated downstream layers where needed

One-line summary:

**A helper succeeds on P-S0 by exposing structure, not by being helpful in a broader sense.**

---

## 14. Mixed packets are high-risk by default

A packet that simultaneously requests:

- raw structure,
- support/accounting,
- semantic overlay,
- and review posture

is high-risk and should usually be split.

Reasons:

- mixed packets amplify symbolic gravity,
- mixed packets invert rendering order,
- mixed packets hide the active query class,
- mixed packets make acceptance ambiguous,
- and mixed packets encourage semantic overwrite.

The default DME posture should be:

**one surface role per packet whenever possible.**
