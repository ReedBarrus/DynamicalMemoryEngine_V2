## 1. Why this note exists

DME now has enough replay, retention, compression, observability, memory, identity, review, and consultation pressure that the project needs one explicit place to answer a recurring question:

**What kind of question is actually being asked here, and what class of support does that question require?**

Without a bounded query taxonomy, several distortion risks become active:

- broad similarity search being mistaken for same-object questioning,
- support questions being mistaken for identity questions,
- review questions being mistaken for consultation questions,
- consultation framing being mistaken for minting readiness,
- query drift hiding inside apparently coherent reuse,
- and runtime surfaces returning something useful while silently answering a different question than the seam claims.

This note exists to prevent those drifts.

---

## 2. Core rule

**In DME, a query is not merely a request for information. A query is a bounded question object that activates a specific seam, requires a specific support class, and constrains what kinds of answer, memory, continuity, and authority claims are lawful at that seam.**

Corollary rules:

- asking is not automatically classed asking,
- retrieval is not automatically same-object questioning,
- usefulness is not automatically question preservation,
- and a broader question must not silently inherit the continuity posture of a narrower one.

If the active query class changes, the system must not preserve the old answer posture automatically.

It must instead narrow, downgrade, split, reclassify, defer, or emit unresolved posture.

## 4. Definition of query

In DME, **query** means:

> a bounded question object that specifies what kind of distinction, support, continuity, or review result is being sought, under what declared constraints, through what seam, and with what lawful class ceiling on the answer.

This means query is always:

- question-relative by definition,
- seam-relative,
- support-relative,
- constraint-relative,
- identity-relevant where same-object claims are involved,
- regime-relative under perturbation, compression, or boundary shift,
- and mechanization-relative where a real bounded path matters.

A query is **not** guaranteed to be lawful merely because:

- it is intelligible,
- it returns something,
- it sounds similar to a prior query,
- it feels adjacent to the same object,
- or it is phrased confidently.

## 5. Query is not answer, claim, or identity

These are related, but not the same.

### Query

Query is a **question activation class**.

It governs:
- what is being asked,
- what seam is being activated,
- what support is required,
- and what kinds of answers are even lawful.

### Answer

An answer is a **returned output class**.

It may or may not satisfy the active query honestly.

### Claim

A claim is an **assertive closure class**.

It states what is being said on the basis of an answered query.

### Identity

Identity is a **continuity class**.

It determines what bounded object remains the same under declared conditions.

Therefore:

- a query may be lawful even when no strong answer is available,
- an answer may be useful while still answering the wrong query,
- a claim may overreach the active query,
- and identity language must never outrun the query class that is actually active.

One-line distinction:

**Query opens the seam. Answer fills it. Claim closes it. Identity constrains when “the same thing” may survive across that seam.**

## 6. The eight query classes

The current bounded taxonomy recognizes eight primary query classes.

---

### 6.1 Q0 — Presence / observation query

#### Definition

A bounded question asking whether something is present, detectable, or observably available at an active seam.

#### Bounded question

**Is anything there, and can it be lawfully observed?**

#### Primary role

- activate exposure and detectability seams,
- support observational rather than interpretive questioning,
- stay below stronger continuity or identity claims by default.

#### Common examples

- is there a detectable anomaly at this boundary?
- is this distinction present in the current frame?
- is the replay surface exposing anything at all?

#### Dominant supports

- observational boundary,
- observability classes O0–O2,
- low or bounded uncertainty posture.

#### Main failure modes

- exposure mistaken for registration,
- observability inflated into support claims,
- decorative visibility mistaken for detection.

#### One-line summary

**Q0 asks whether something is observably present.**

---

### 6.2 Q1 — Structural query

#### Definition

A bounded question asking about configuration, relation, topology, or invariant-bearing structure.

#### Bounded question

**What lawful structure is present here?**

#### Primary role

- activate relation/configuration seams,
- support structural comparison,
- remain below stronger continuity claims unless explicitly upgraded.

#### Common examples

- what relational structure does this transformed object preserve?
- what topology or basin geometry is present?
- what structural invariants remain visible under this lens?

#### Dominant supports

- structural constraints,
- structural coherence,
- structural invariance,
- transform boundary honesty.

#### Main failure modes

- similarity mistaken for structure,
- structural drift hidden by coherent summary,
- transform inflation.

#### One-line summary

**Q1 asks what structure is present and lawful.**

---

### 6.3 Q2 — Continuity query

#### Definition

A bounded question asking whether something remains continuous across time, replay, retention, or transformation.

#### Bounded question

**Does this object remain continuous enough for the same bounded question to stay live?**

#### Primary role

- activate temporal, replay, and retention continuity seams,
- sit upstream of stronger memory or identity claims,
- support downgrade-sensitive continuity review.

#### Common examples

- does this replay surface still support the same replay question?
- does this retained object still preserve continuity under current support?
- does this transformed object remain continuous across the seam?

#### Dominant supports

- temporal coherence,
- persistence constraints,
- retention boundaries,
- bounded uncertainty and drift posture.

#### Main failure modes

- continuity mistaken for identity,
- same-object claims hidden inside continuity language,
- support collapse hidden by local coherence.

#### One-line summary

**Q2 asks whether bounded continuity still survives.**

---

### 6.4 Q3 — Support / lineage query

#### Definition

A bounded question asking what support, provenance, or lineage survives and whether it remains sufficient for later use.

#### Bounded question

**What support survives, and how strongly can it still justify later use?**

#### Primary role

- activate support-trace, lineage, and retention seams,
- keep evidence questions separate from same-object claims,
- support review and replay honesty.

#### Common examples

- what receipts or lineage still back this replay object?
- how much of the original support basis survives compression?
- does this retained surface still preserve support strongly enough?

#### Dominant supports

- persistence constraints,
- retention boundary,
- M0–M1 memory classes,
- support compression posture.

#### Main failure modes

- digest mistaken for receipts,
- lineage mistaken for usability,
- support-only mistaken for memory or identity.

#### One-line summary

**Q3 asks what support and lineage still survive.**

---

### 6.5 Q4 — Memory query

#### Definition

A bounded question asking what class of memory remains lawful for the active object or seam.

#### Bounded question

**What class of memory, if any, does this structure still preserve?**

#### Primary role

- activate memory-class evaluation,
- distinguish retention, lineage, operational, and adaptive memory,
- keep memory claims below identity unless stronger conditions are met.

#### Common examples

- is this object still M2 or only M1?
- did compression preserve operational memory or only support?
- can this retained object still answer the same bounded question?

#### Dominant supports

- memory taxonomy,
- persistence and support sufficiency,
- bounded question preservation,
- mechanized reuse path where required.

#### Main failure modes

- replay-shaped object treated as memory-bearing,
- useful-once object treated as durable memory,
- memory overclaim under drift.

#### One-line summary

**Q4 asks what class of memory remains lawful.**

---

### 6.6 Q5 — Identity query

#### Definition

A bounded question asking whether the same bounded object remains the same at an active identity class.

#### Bounded question

**Is this still the same bounded object at the active identity class?**

#### Primary role

- activate structural identity law directly,
- test same-question, same-constraint, support, and mechanization sufficiency,
- provide the most continuity-sensitive query class currently active in Door One use.

#### Common examples

- is this still the same replay-support object?
- does this compressed object still preserve I2?
- is this packet still the same bounded workflow object?

#### Dominant supports

- Structural Identity Law,
- identity hierarchy,
- B4 identity boundary,
- M1–M3 where applicable,
- coherence, invariance, drift, distortion, and uncertainty thresholds.

#### Main failure modes

- similarity mistaken for identity,
- question drift hidden as same-object continuity,
- class inflation,
- support insufficiency.

#### One-line summary

**Q5 asks whether the same bounded object still remains the same.**

---

### 6.7 Q6 — Review query

#### Definition

A bounded question asking how an object should be classified, downgraded, deferred, accepted, or routed under the review matrix.

#### Bounded question

**What is the lawful evaluation posture for this object at the active seam?**

#### Primary role

- activate the review matrix,
- classify support, memory, identity, mechanism, and regime posture,
- stay below consultation and minting by default.

#### Common examples

- how should this object be classed under the current matrix?
- what downgrade posture is required?
- is this valid, narrowed, degraded, support-only, unresolved, or invalid?

#### Dominant supports

- review matrix,
- relevant taxonomies,
- mechanization posture,
- uncertainty and drift posture.

#### Main failure modes

- review mistaken for consultation,
- evaluation mistaken for decision,
- rhetoric replacing classed audit.

#### One-line summary

**Q6 asks for lawful bounded evaluation posture.**

---

### 6.8 Q7 — Consultation / minting-readiness query

#### Definition

A bounded question asking what risks, readiness conditions, or blockers remain for consultation or minting, without silently promoting the object.

#### Bounded question

**What remains to be interpreted, narrowed, or satisfied before consultation or minting is lawful?**

#### Primary role

- activate consultation or readiness surfaces,
- stay below promotion by default,
- support advisory rather than authoritative posture.

#### Common examples

- what blocks minting for this candidate object?
- what consultation risks remain active?
- what stronger claim is still not justified?

#### Dominant supports

- review matrix,
- mechanization closure posture,
- class inflation checks,
- uncertainty and drift posture,
- identity/memory support stack.

#### Main failure modes

- consultation mistaken for approval,
- readiness mistaken for minting,
- candidate mistaken for canon.


#### One-line summary

**Q7 asks about consultation and readiness without silently promoting.**

## 6.9 Q8 Structural exposure query posture

DME now recognizes a special exposure-first posture inside early query work.

A structural exposure ask should remain inside the lowest lawful query classes and should not silently drift upward into support, memory, review, or consultation just because a more "helpful" answer is available.

### Rule

If the user or packet asks to expose raw structure directly, the system should prefer:

- Q0 — presence / observation query
- Q1 — structural query
- sometimes Q2 — continuity query in narrowly declared form

and should not silently substitute:

- Q3 support / lineage
- Q4 memory
- Q5 identity
- Q6 review
- Q7 consultation

### One-line summary

**A structural exposure ask is a structure-first query, not a request for semantic, support, or review closure.**

## 7. Query staircase

The eight classes form a practical staircase:

- **Q0** = presence / observation query
- **Q1** = structural query
- **Q2** = continuity query
- **Q3** = support / lineage query
- **Q4** = memory query
- **Q5** = identity query
- **Q6** = review query
- **Q7** = consultation / minting-readiness query

This staircase should remain explicit.

The system must not silently round weaker query classes upward.

Examples:

- Q0 is not automatically Q1
- Q2 is not automatically Q5
- Q3 is not automatically Q4
- Q4 is not automatically Q5
- Q6 is not automatically Q7

## 8. Query functions

Each query class can be analyzed through three functions.

### 8.1 Activation function

**What seam is this query activating?**

Examples:
- observational seam,
- structural seam,
- continuity seam,
- retention seam,
- identity seam,
- review seam.

### 8.2 Requirement function

**What support must exist for this query to be answered honestly?**

Examples:
- detectable distinction,
- structural invariants,
- lineage,
- memory class support,
- same-object continuity conditions,
- review matrix coverage.

### 8.3 Ceiling function

**What is the strongest lawful answer class this query can support?**

Examples:
- presence only,
- structure only,
- continuity only,
- support-only,
- memory-class result,
- identity result,
- review posture,
- readiness posture.

This three-function view prevents query from collapsing into generic “ask and answer” language.

## 9. Query regimes

Queries do not operate in a vacuum.

The same query may behave differently under different regimes.

The current taxonomy recognizes five practical regimes.

---

### 9.1 R1 — Stable regime

The bounded seam is steady and support is repeatable.

Best fit:

- Q0–Q6

Typical use:

- same-lens replay review,
- stable structural comparison,
- strong bounded evaluation.

---

### 9.2 R2 — Narrow perturbation regime

The same bounded question remains active under mild variation.

Best fit:

- Q2–Q5

Typical use:

- bounded continuity under perturbation,
- adaptive memory question,
- same-object question under mild change.

---

### 9.3 R3 — Compression regime

Support or representation has been reduced, increasing classification pressure.

Best fit:

- Q3–Q7

Typical danger:

- support queries silently becoming identity queries,
- compressed continuity being overread as same-object continuity.

---

### 9.4 R4 — Boundary-shift regime

The active lens, tier, seam, or interface changes.

Best fit:

- Q1–Q7

Typical danger:

- same-looking question under changed seam conditions,
- structural question quietly becoming review or consultation question.

---

### 9.5 R5 — Rupture regime

The same bounded question is no longer answerable honestly under the old terms.

Best fit:

- any class that now requires narrowing, split, defer, or unresolved posture.

Typical danger:

- preserving the old query language after the seam already changed.

---

## 10. Query is not “whatever information is nearby”

Several bad habits must remain fenced.

### 10.1 Retrieval is not automatically same-question answering

A returned nearby object may be useful while still failing the active query class.

### 10.2 Broader interpretation is not automatically a better answer

A larger semantic answer may violate the original bounded question.

### 10.3 Similarity is not automatically identity questioning

Looking nearby is not the same as asking whether the same object remains the same.

### 10.4 Review is not automatically consultation

A classification result is not the same as advisory readiness interpretation.

## 11. Query-bearing vs answer-bearing posture

A central dynamics distinction is:

### Query-bearing

The system has an explicit bounded question class and knows what support it requires.

### Answer-bearing

The system has a returned object or classification that actually answers that bounded question honestly.

### False-answer posture

The system returns something useful, related, or coherent while failing to answer the active query class honestly.

This distinction should remain explicit.

The system must not casually call something “answered” when it is only:

- nearby,
- suggestive,
- support-related,
- memory-adjacent,
- or review-shaped.

## 24. Query failure modes

Query most commonly fails through the following modes.

### 24.1 Question inflation

A weaker query class is silently treated as a stronger one.

### 24.2 Question drift

The seam begins answering a different bounded question while preserving the old query language.

### 24.3 Support mismatch

The query class requires stronger support than the seam actually provides.

### 24.4 Mechanization mismatch

The query is well-stated, but the real path needed to answer it is missing or partial.

### 24.5 Answer substitution

A nearby, useful, or coherent answer is returned in place of the real answer class.

### 24.6 Query overmixing

One seam or packet is expected to answer multiple materially different query classes without explicit split or route posture.

## 25. Lawful downgrade postures

When a stronger query cannot be answered honestly, preferred postures include:

- `presence_only`
- `structure_only`
- `continuity_only`
- `support_only`
- `memory_unresolved`
- `identity_unresolved`
- `review_only`
- `consultation_not_ready`
- `mechanization_gap`
- `question_drift`
- `split_required`
- `review_required`

The governing rule is:

**when the active seam cannot answer the stronger query class honestly, the system must weaken the query/answer posture rather than preserve stronger question language for convenience.**

## 26. Query outcomes

A practical query audit should produce one of the following broad outcomes.

### 26.1 Lawfully answered
The active query class was explicit and answered within its support limits.

### 26.2 Answered but narrowed
The seam could answer only a weaker query class than originally posed.

### 26.3 Support-only answer
Useful support was returned, but not a full answer to the stronger query class.

### 26.4 Misclassified query
The seam was actually asking a different class of question than it claimed.

### 26.5 Mechanization-limited query
The question is valid, but the real path to answer it is incomplete.

### 26.6 Unresolved query
Support is too weak to answer the active query honestly.

### 26.7 Split required
The original query actually contains more than one real bounded question.

## 28. Decision rule

Preserve a stronger query-answer posture only when all of the following are true strongly enough for the active seam:

the query class is explicit,
the same bounded question remains active,
the same declared constraints remain active,
enough support exists for that query class,
any required memory, identity, or review conditions are actually satisfied,
and any required implementation path is actually mechanized.

If not, the seam may still be useful.

But it must not silently preserve the stronger query class.

## 29. One-line practical rule

In DME, a query is a bounded question object that activates a specific seam and support requirement; when the seam cannot answer that class of question honestly, the system must weaken the query/answer posture rather than preserve stronger question language by convenience.

## 30. One-line summary

The DME query taxonomy distinguishes presence, structural, continuity, support, memory, identity, review, and consultation/readiness queries so bounded question activation can be discussed with explicit support requirements, downgrade posture, and clear linkage to identity, memory, mechanism, and review law.
