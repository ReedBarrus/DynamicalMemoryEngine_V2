## 1. Why this note exists

DME now has enough replay, retention, observability, memory, identity, review, and consultation pressure that the project needs one explicit place to answer a recurring question:

**What kind of assertion is actually being made, and what support ceiling lawfully bounds that assertion?**

Without a bounded claim taxonomy, several distortion risks become active:

- descriptive outputs being mistaken for stronger continuity claims,
- support claims being mistaken for memory claims,
- memory claims being mistaken for identity claims,
- identity-adjacent language being mistaken for minting readiness,
- declared or displayed surfaces being mistaken for mechanized capability claims,
- and query answers being silently over-closed into stronger assertions than the seam actually supports.

This note exists to prevent those drifts.

---

## 2. Core rule

**In DME, a claim is not merely anything that is said. A claim is a bounded assertion object whose lawfulness depends on the query class that generated it, the support that backs it, the seam conditions that constrain it, and the strongest continuity or authority posture the current mechanism honestly justifies.**

Corollary rules:

- saying is not automatically asserting,
- returning is not automatically claiming,
- useful output is not automatically a lawful strong claim,
- and a stronger claim class must not be silently inherited from a weaker query or weaker support basis.

If a stronger assertion cannot be supported, the system must weaken the claim rather than preserve stronger language for convenience.

## 4. Definition of claim

In DME, **claim** means:

> a bounded assertion made about an object, seam, support surface, continuity class, readiness posture, or evaluated result, where the lawfulness of the assertion depends on the active query class, surviving support, declared constraints, and any required mechanized path.

This means claims are always:

- query-relative,
- support-relative,
- constraint-relative,
- class-relative,
- mechanism-relative where a real path matters,
- and regime-relative under perturbation, compression, drift, or uncertainty.

A claim is **not** guaranteed to be lawful merely because:

- an answer was returned,
- the answer is useful,
- the answer is coherent,
- the answer resembles a stronger class result,
- the seam is well-rendered,
- or the statement sounds precise.

## 5. Claim is not query, answer, or canon

These are related, but not the same.

### Query

Query is a **question activation class**.

It governs:
- what is being asked,
- what seam is active,
- what support is required.

### Answer

Answer is a **returned output class**.

It may be partial, narrowed, support-only, or unresolved.

### Claim

Claim is an **assertive closure class**.

It states what the system is willing to say on the basis of the answer.

### Canon

Canon is a **trust-uplift / promotion class**.

It is not implied merely by a lawful claim.

Therefore:

- a query may be valid while no strong claim is possible,
- an answer may exist while only a weak claim is lawful,
- a lawful claim may still remain far below canon,
- and consultation or readiness language must never be mistaken for promotion.

One-line distinction:

**Query opens the seam. Answer returns from the seam. Claim states what the system is willing to assert from that return. Canon is a different later uplift, not a synonym for strong claim.**

## 6. The eight claim classes

The current bounded taxonomy recognizes eight primary claim classes.

---

### 6.1 L0 — Descriptive claim

#### Definition

A bounded statement that describes what is present, detected, or returned without yet asserting stronger continuity, support, or authority.

#### Bounded question source

Typically generated from:
- Q0 presence / observation queries
- Q1 structural queries in weak form

#### Primary role

- state what is there,
- remain below stronger continuity claims,
- prevent observation from inflating into support or identity language.

#### Common examples

- a distinction is present at this boundary,
- this pattern is observably detectable,
- this output exhibits a certain structural feature.

#### Dominant supports

- observability,
- low-level structural description,
- bounded descriptive evidence.

#### Main failure modes

- description mistaken for support,
- visible surface mistaken for stronger claim,
- descriptive overclosure.

#### One-line summary

**L0 describes what is present without stronger continuity closure.**

---

### 6.2 L1 — Support claim

#### Definition

A bounded statement that asserts that some lineage, support trace, provenance, or bounded evidence survives strongly enough for later use at a support-only or lineage-bearing level.

#### Bounded question source

Typically generated from:
- Q3 support / lineage queries

#### Primary role

- assert surviving support,
- distinguish support-bearing from mere display,
- remain below stronger memory or identity claims unless explicitly upgraded.

#### Common examples

- this object preserves lineage to origin,
- sufficient support trace remains for bounded review,
- retained support survives at a support-only class.

#### Dominant supports

- retention boundary,
- persistence constraints,
- M0–M1 memory classes,
- support compression posture.

#### Main failure modes

- digest mistaken for stronger support class,
- lineage mistaken for memory,
- support inflation.

#### One-line summary

**L1 asserts surviving support without automatically asserting memory or identity.**

---

### 6.3 L2 — Memory claim

#### Definition

A bounded statement that asserts that a lawful memory class remains active for the object under the active bounded question.

#### Bounded question source

Typically generated from:
- Q4 memory queries

#### Primary role

- assert memory class explicitly,
- distinguish memory-bearing from support-only,
- remain below identity unless stronger conditions are met.

#### Common examples

- this object remains M1 lineage memory,
- this retained structure still supports M2 operational memory,
- this memory has degraded from M3 to M2.

#### Dominant supports

- memory taxonomy,
- support sufficiency,
- same bounded question preservation,
- mechanized reuse path where required.

#### Main failure modes

- replay-shaped object treated as memory-bearing,
- one-time reuse mistaken for durable memory,
- memory overclaim under drift.

#### One-line summary

**L2 asserts bounded memory class.**

---

### 6.4 L3 — Identity claim

#### Definition

A bounded statement that asserts that the same bounded object remains the same at an active identity class.

#### Bounded question source

Typically generated from:
- Q5 identity queries

#### Primary role

- assert same-object continuity,
- activate the strongest current Door One continuity posture,
- remain highly threshold-sensitive and downgrade-aware.

#### Common examples

- this retained object still preserves I2,
- this packet remains the same bounded workflow object,
- this replay object no longer preserves same-object continuity and must be narrowed.

#### Dominant supports

- Structural Identity Law,
- identity hierarchy,
- same bounded question,
- same declared constraints,
- sufficient support,
- mechanized basis where required.

#### Main failure modes

- similarity mistaken for identity,
- question drift hidden inside descriptive continuity,
- class inflation,
- support insufficiency.

#### One-line summary

**L3 asserts same-object continuity at an active identity class.**

---

### 6.5 L4 — Mechanization claim

#### Definition

A bounded statement that asserts that a capability, seam, or answer path is mechanized, partially mechanized, displayed, or merely declared.

#### Bounded question source

Typically generated from:
- Q6 review queries
- Q7 readiness queries
- or direct bounded capability-status questions

#### Primary role

- assert capability-status posture explicitly,
- enforce anti-overclaim discipline,
- prevent shaped surfaces from being mistaken for real functional attainment.

#### Common examples

- this seam is declared-only,
- this surface is displayed but not mechanized,
- this capability is partially mechanized,
- this bounded path is mechanized at its declared seam.

#### Dominant supports

- Declared vs Mechanized Audit,
- Mechanization Closure Gate,
- reachable backend/render/failure/downgrade conditions.

#### Main failure modes

- displayed = mechanized inflation,
- declared = mechanized inflation,
- surface clarity outrunning mechanism.

#### One-line summary

**L4 asserts capability-status posture.**

---

### 6.6 L5 — Review claim

#### Definition

A bounded evaluative statement asserting how an object should be classified under the review matrix.

#### Bounded question source

Typically generated from:
- Q6 review queries

#### Primary role

- assert evaluation posture,
- classify validity, narrowing, degradation, support-only, unresolved, or invalid status,
- remain below consultation and minting.

#### Common examples

- this object is narrowed but valid,
- this object is support-only,
- this object is unresolved under the active seam,
- this object is misclassified.

#### Dominant supports

- review matrix,
- relevant taxonomy stack,
- uncertainty and drift posture,
- mechanization posture.

#### Main failure modes

- review mistaken for readiness approval,
- rhetoric replacing classification,
- hidden inflation from “sounds good” posture.

#### One-line summary

**L5 asserts bounded review classification.**

---

### 6.7 L6 — Consultation / readiness claim

#### Definition

A bounded advisory statement asserting what readiness, risk, narrowing, or blocking posture remains for consultation or minting-related handling.

#### Bounded question source

Typically generated from:
- Q7 consultation / minting-readiness queries

#### Primary role

- assert advisory readiness posture,
- identify blockers, risks, or remaining work,
- stay below minting or promotion by default.

#### Common examples

- this object is not consultation-ready,
- this object remains readiness-limited by uncertainty and mechanization gaps,
- this object is consultation-eligible under a narrowed scope only.

#### Dominant supports

- review matrix,
- mechanization closure posture,
- uncertainty, drift, and class inflation checks,
- explicit bounded non-promotion posture.

#### Main failure modes

- readiness mistaken for promotion,
- consultation mistaken for approval,
- candidate mistaken for canon.

#### One-line summary

**L6 asserts advisory readiness or blocking posture below promotion.**

---

### 6.8 L7 — Promotion / minting-eligibility claim

#### Definition

A bounded statement asserting only whether the current seam appears to satisfy or fail declared minting-eligibility conditions, without itself performing promotion.

#### Bounded question source

Typically generated from:
- Q7 readiness queries in their strongest lawful form
- explicit bounded minting-readiness review

#### Primary role

- state minting-eligibility posture,
- remain strictly bounded below actual canon activation,
- preserve the difference between “eligible,” “not eligible,” and “promoted.”

#### Common examples

- this object is not minting-eligible under current uncertainty,
- this object appears minting-eligible only under narrowed scope,
- this seam does not yet justify stronger promotion handling.

#### Dominant supports

- review matrix,
- mechanization closure,
- memory/identity support stack,
- uncertainty/drift/distortion checks,
- explicit non-promotion posture.

#### Main failure modes

- eligibility mistaken for promotion,
- advisory posture mistaken for canon,
- same-object confidence inflated into trust uplift.

#### One-line summary

**L7 asserts minting-eligibility posture without performing promotion.**

## 6.9 L8 Structural exposure and claim ceiling

A direct structural exposure surface should usually remain below strong closure classes.

### Rule

When the active seam is raw structural exposure, the lawful claim ceiling should usually remain inside:

- L0 — descriptive claim
- sometimes narrow L1 support reference only when separately declared

and should not silently elevate into:

- L2 memory claim
- L3 identity claim
- L5 review claim
- L6 readiness claim
- L7 minting-eligibility posture

### One-line summary

**Exposure-first surfaces describe what is directly present; they do not silently close upward into stronger continuity or authority claims.**

## 7. Claim staircase

The eight classes form a practical staircase:

- **L0** = descriptive claim
- **L1** = support claim
- **L2** = memory claim
- **L3** = identity claim
- **L4** = mechanization claim
- **L5** = review claim
- **L6** = consultation / readiness claim
- **L7** = promotion / minting-eligibility claim

This staircase should remain explicit.

The system must not silently round weaker claim classes upward.

Examples:

- L0 is not automatically L1
- L1 is not automatically L2
- L2 is not automatically L3
- L5 is not automatically L6
- L6 is not automatically L7
- L7 is not canon

## 8. Claim functions

Each claim class can be analyzed through three functions.

### 8.1 Basis function

**What query class and support surface generated this claim?**

Examples:
- observational answer,
- support trace,
- memory evaluation,
- identity review,
- readiness review.

### 8.2 Strength function

**What is the strongest lawful assertion class actually supported here?**

Examples:
- description only,
- support-only,
- memory-bearing,
- same-object continuity,
- review posture,
- readiness posture.

### 8.3 Ceiling function

**What must not be claimed beyond this class?**

Examples:
- memory from support only,
- identity from memory only,
- promotion from readiness only,
- mechanization from display only.

This three-function view prevents claim from collapsing into generic “output meaning” language.

## 9. Claim regimes

Claims do not operate in a vacuum.

The same seam may support different claim classes under different regimes.

The current taxonomy recognizes five practical regimes.

---

### 9.1 R1 — Stable regime

Support is steady and claim class can be repeated with low ambiguity.

Best fit:

- L0–L5

Typical use:

- stable replay/support evaluation,
- strong review classification,
- bounded memory/identity claims.

---

### 9.2 R2 — Narrow perturbation regime

The same bounded question remains active under mild variation.

Best fit:

- L2–L6

Typical use:

- adaptive memory claims,
- same-object continuity under mild perturbation,
- readiness posture under narrow stress.

---

### 9.3 R3 — Compression regime

Support or representation is reduced, raising claim risk.

Best fit:

- all classes, especially L1–L7

Typical danger:

- compressed support mistaken for memory,
- compressed memory mistaken for identity,
- readiness claims surviving after support degradation.

---

### 9.4 R4 — Boundary-shift regime

Lens, tier, interface, or seam changes.

Best fit:

- L1–L7

Typical danger:

- same-looking claim under changed seam conditions,
- readiness or identity claims preserved after boundary change.

---

### 9.5 R5 — Rupture regime

The old stronger claim class can no longer be preserved honestly.

Best fit:

- downgrade, unresolved, support-only, or non-eligibility postures.

Typical danger:

- preserving strong semantic closure after the seam has already broken.

## 10. Claim is not “whatever the system says”

Several bad habits must remain fenced.

### 10.1 Returned text is not automatically a lawful claim

The system may output something coherent without lawfully supporting the stronger assertion class.

### 10.2 Useful interpretation is not automatically a lawful strong claim

Helpfulness does not by itself justify stronger continuity or readiness language.

### 10.3 Review language is not automatically readiness language

Evaluation does not equal consultation.

### 10.4 Readiness language is not automatically promotion language

Eligibility is not canon.

## 11. Claim-bearing vs support-only posture

A central dynamics distinction is:

### Claim-bearing

The seam supports a real bounded assertion at the declared claim class.

### Support-only

The seam returns useful support, traces, hints, or classification fragments, but does not justify the stronger claim class.

### False closure

The seam preserves stronger semantic closure after support, query, mechanism, or continuity thresholds have already weakened.

This distinction should remain explicit.

The system must not casually treat something as strongly claimed when it is only:

- descriptive,
- support-adjacent,
- memory-adjacent,
- review-shaped,
- readiness-suggestive,
- or surface-coherent.

## 12. Relation to Query Taxonomy

Query Taxonomy defines what kind of bounded question is being asked and what support that question requires.

This note defines what kind of bounded assertion may lawfully be made on the basis of that answered query.

The relationship is therefore:

- query class sets the opening seam,
- claim class sets the closing seam,
- claim inflation often begins as query inflation,
- and claim class must never silently exceed the query class and support basis that generated it.

Useful relation:

- Q0 commonly supports L0
- Q3 commonly supports L1
- Q4 commonly supports L2
- Q5 commonly supports L3
- Q6 commonly supports L4/L5
- Q7 commonly supports L6/L7

One-line pairing:

**Query opens the seam. Claim closes it.**

## 23. Claim failure modes

Claim most commonly fails through the following modes.

### 23.1 Claim inflation

A weaker support basis is used to make a stronger assertion class.

### 23.2 Query-claim mismatch

The assertion class is stronger than the query class lawfully supports.

### 23.3 Mechanization overclaim

The claim presumes more implementation than exists.

### 23.4 Continuity overclaim

Support, memory, or identity claims remain stronger than surviving evidence justifies.

### 23.5 Advisory overreach

Readiness or consultation language is mistaken for approval or promotion.

### 23.6 Semantic closure by convenience

The system says more than the seam can honestly support because a stronger closure feels cleaner.

## 25. Claim outcomes

A practical claim audit should produce one of the following broad outcomes.

### 25.1 Lawful claim
The active claim class is explicit and supported at its bounded seam.

### 25.2 Narrowed claim
A weaker or narrower claim class is the strongest lawful closure available.

### 25.3 Support-only claim
Useful support is available, but stronger memory or identity assertion is not lawful.

### 25.4 Misclassified claim
The assertion class is stronger or different than the seam actually supports.

### 25.5 Mechanization-limited claim
The assertion is plausible in concept, but the real path needed to support it is incomplete.

### 25.6 Unresolved claim
Support is too weak to close the stronger assertion honestly.

### 25.7 Blocked readiness / non-eligibility claim
Readiness or minting language is not currently lawful.

## 27. Decision rule

Preserve a stronger claim only when all of the following are true strongly enough for the active seam:

the claim class is explicit,
the source query class is explicit,
the same bounded question remains active,
enough support exists for that claim class,
any required memory, identity, review, or readiness conditions are actually satisfied,
and any required implementation path is actually mechanized.

If not, the seam may still be useful.

But it must not silently preserve the stronger claim class.

## 28. One-line practical rule

In DME, a claim is a bounded assertion whose lawfulness depends on the query that opened the seam, the support that backs it, and the strongest continuity or mechanism posture the seam honestly justifies; when that support weakens, the system must weaken the claim rather than preserve stronger semantic closure by convenience.

## 29. One-line summary

The DME claim taxonomy distinguishes descriptive, support, memory, identity, mechanization, review, consultation/readiness, and minting-eligibility claims so semantic closure can be discussed with explicit support ceilings, downgrade posture, and clear linkage to query, memory, identity, mechanism, and review law.
