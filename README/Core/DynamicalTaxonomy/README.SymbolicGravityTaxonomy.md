## 1. Why this note exists

DME now has enough replay, retention, interpretation, packet, workflow, and visualization pressure that one recurring pattern has become explicit:

**symbols do not merely describe a seam; they alter the seam’s center of gravity.**

A name, label, summary, or packet phrase can silently change:

- what question the system thinks is being asked,
- what representation it prioritizes,
- what layer becomes cognitively primary,
- what class of answer feels “complete,”
- and what kind of closure the surface begins optimizing for.

This matters because DME depends on a strict order:

1. structure must become directly inspectable,
2. support/accounting may then be attached,
3. semantic overlays may appear later,
4. review and readiness posture must remain later still.

Symbolic pressure tends to invert that order by making meaning arrive before exposure.

This note exists to define that field pressure explicitly.

One-line summary:

**Symbolic gravity is the bounded tendency of names, labels, summaries, and semantic posture to pull a surface toward earlier closure, stronger interpretation, and reduced first-order inspectability.**

## 2. Core rule

**In DME, symbolic material is not neutral. A symbol acts as a bounded control input on representation choice, closure timing, and surface authority.**

Corollary rules:

- naming is not neutral,
- labels are not lightweight,
- summaries are not harmless,
- semantic helpfulness is not free,
- descriptive prose is not merely decorative,
- and stronger symbolic closure must not silently inherit authority from a weaker structural seam.

If symbolic pressure becomes strong enough that the active bounded structural question is no longer directly inspectable, the system must not preserve the old exposure claim.

It must instead:

- remove,
- split,
- narrow,
- downgrade,
- reclassify,
- or isolate the symbolic layer.

## 4. Definition of symbolic gravity

In DME, **symbolic gravity** means:

> the bounded tendency of a name, label, summary, semantic descriptor, packet phrase, or interpretive posture to pull a surface, seam, or workflow object toward stronger closure, stronger abstraction, or stronger authority than the current bounded structural support directly justifies.

This means symbolic gravity is always:

- seam-relative,
- question-relative,
- surface-relative,
- naming-sensitive,
- layer-sensitive,
- support-sensitive,
- and authority-sensitive.

Symbolic gravity is **not** guaranteed merely because:

- symbols are present,
- labels exist,
- prose exists,
- a surface is readable,
- or a packet is well-written.

The issue is not symbol presence by itself.

The issue is **what the symbol begins to pull the seam toward**.

One-line distinction:

**Symbolic gravity is not language existing. It is language changing what the system starts optimizing to expose or conclude.**

---

## 5. Symbolic gravity is not semantics, distortion, or drift

These are related, but not the same.

### Semantics

Semantics concerns:

- what a label or phrase means,
- what bounded interpretation is being made.

### Symbolic gravity

Symbolic gravity concerns:

- what a label or phrase begins to pull the seam toward.

### Distortion

Distortion concerns:

- what invariants or relations were bent or violated.

### Drift

Drift concerns:

- what seam condition changed away from the original justification.

Therefore:

- a symbol may be semantically correct while still exerting bad gravity,
- symbolic gravity may exist before clear distortion is visible,
- symbolic gravity may cause later drift,
- and symbolic gravity may be present even when the resulting output still looks coherent.

One-line distinction:

**Semantics asks what a symbol says. Symbolic gravity asks what the symbol makes the seam become.**

## 6. The six symbolic gravity classes

The current bounded taxonomy recognizes six primary symbolic gravity classes.

---

### 6.1 S0 — Null symbolic gravity

#### Definition

Symbolic material is absent, inert, or strictly local enough that it does not materially alter the seam’s closure posture.

#### Bounded question

**Is symbolic pressure effectively negligible here?**

#### Primary role

- mark clean exposure-first zones,
- preserve first-order structural surfaces,
- define the safest baseline.

#### Common examples

- raw arrays,
- unlabeled transform-space plots,
- coordinates with only axis names,
- declared-lens output with minimal numeric descriptors.

#### Main failure modes

- false confidence that all future labels will remain equally inert.

#### One-line summary

**S0 is effectively no meaningful symbolic pull.**

---

### 6.2 S1 — Local descriptor gravity

#### Definition

A symbol exists and has local descriptive value, but remains tightly anchored to an already-exposed structural distinction.

#### Bounded question

**Does the symbol merely point to something already exposed?**

#### Primary role

- allow minimal labels,
- support axis/index/lens naming,
- preserve inspectability while improving navigation.

#### Common examples

- `bin_index=170`
- `window=42`
- `Fs_target=2400`
- `transform_family=fft`

#### Main failure modes

- local labels gradually accumulating into stronger closure.

#### One-line summary

**S1 is anchored local description without meaningful seam drift.**

---

### 6.3 S2 — Compression gravity

#### Definition

A symbol or summary compresses multiple distinctions into a smaller phrase, increasing the risk that the compressed surface becomes cognitively primary.

#### Bounded question

**Is the symbol beginning to replace multiple distinctions with one smaller closure object?**

#### Primary role

- identify early closure pressure,
- mark where summaries begin to pull harder than the exposed structure.

#### Common examples

- “dominant state”
- “stable region”
- “same pattern”
- compact summary text over a richer field

#### Main failure modes

- hidden loss,
- compression camouflage,
- interpretive substitution.

#### One-line summary

**S2 is symbolic compression beginning to outweigh exposed distinctions.**

---

### 6.4 S3 — Interpretive gravity

#### Definition

A symbol or phrase begins to answer a stronger question than the seam was supposed to answer, pulling the surface toward interpretation, support, or class posture.

#### Bounded question

**Is the symbol moving the seam from exposure into interpretation?**

#### Primary role

- detect when labels become semantic closure,
- mark the boundary where symbolic material stops being merely descriptive.

#### Common examples

- “anomalous”
- “memory-bearing”
- “degraded”
- “returning”
- “stable” on a raw exposure surface

#### Main failure modes

- semantic overwrite,
- query drift,
- authority misalignment,
- exposure inversion.

#### One-line summary

**S3 is symbolic material that begins interpreting the seam rather than just indexing it.**

---

### 6.5 S4 — Authority gravity

#### Definition

A symbol or phrase pulls the seam into a stronger support, memory, identity, review, or readiness posture than the current surface is allowed to carry.

#### Bounded question

**Is the symbol silently importing a stronger class or authority ceiling?**

#### Primary role

- detect support/identity/review inflation,
- prevent raw or support-only surfaces from carrying stronger closure than they justify.

#### Common examples

- “memory state” where only support exists
- “same object” where continuity is not yet established
- “reconstruction” where replay-shaped output exists
- “ready” on a review-incomplete seam

#### Main failure modes

- claim inflation,
- class drift,
- mechanization camouflage,
- candidate-to-canon contamination.

#### One-line summary

**S4 is symbolic pull into stronger claim or authority posture.**

---

### 6.6 S5 — Surface-capture gravity

#### Definition

Symbolic material becomes the true center of gravity of the surface, so that raw structure, support, or lower layers remain only as subordinate evidence for a semantic whole.

#### Bounded question

**Has symbolic closure captured the surface as a whole?**

#### Primary role

- diagnose full symbolic takeover,
- mark when a nominally structural surface is no longer functionally structural.

#### Common examples

- a “viewer” that shows mostly labels, counts, and summaries
- a mixed surface where review and semantics dominate the interpretation before structure is seen
- packet or workflow prose that pre-decides the seam so strongly that implementation merely follows the symbolic closure

#### Main failure modes

- mixed-surface collapse,
- structural occlusion,
- total exposure inversion,
- seam corruption.

#### One-line summary

**S5 is symbolic capture of the surface.**

---

## 7. Symbolic gravity staircase

The six classes form a practical staircase:

- **S0** = null symbolic gravity
- **S1** = local descriptor gravity
- **S2** = compression gravity
- **S3** = interpretive gravity
- **S4** = authority gravity
- **S5** = surface-capture gravity

This staircase should remain explicit.

The system must not silently round lower symbolic gravity classes upward.

Examples:

- S1 is not automatically S2
- S2 is not automatically S3
- S3 is not automatically S4
- S4 is not automatically S5

One-line summary:

**Not all labels are equally dangerous, but every label enters a gravity field.**

## 8. Symbolic gravity functions

Each symbolic gravity class can be analyzed through four functions.

### 8.1 Anchor function

**What is the symbol attached to?**

Examples:

- an axis,
- a bin,
- a coordinate,
- a support object,
- a semantic posture,
- a workflow status.

A symbol with no visible anchor is more dangerous.

### 8.2 Compression function

**How many distinctions is the symbol collapsing?**

Examples:

- one numeric property,
- a cluster of relations,
- an entire state summary,
- a continuity posture,
- a review class.

Greater collapse usually means greater symbolic gravity.

### 8.3 Authority function

**What stronger class or closure posture does the symbol imply?**

Examples:

- descriptive only,
- interpretive,
- support-bearing,
- memory-bearing,
- identity-bearing,
- review-bearing,
- readiness-bearing.

Symbols that imply stronger authority than the seam supports are high-risk.

### 8.4 Propagation function

**How far does the symbol’s pull extend?**

Examples:

- local point label,
- surface-wide summary,
- packet title,
- workflow role term,
- runtime artifact class name,
- system-facing operator name.

One-line summary:

**Anchor, compression, authority, and propagation determine how symbolically heavy a symbol is.**

## 9. Symbolic gravity regimes

Symbolic gravity does not operate in a vacuum.

The same term may be safe in one regime and dangerous in another.

The current taxonomy recognizes five practical regimes.

---

### 9.1 R1 — Raw exposure regime

First-order structural output is the active seam.

Best fit:

- S0
- S1

High-risk:

- S2+
- any authority-bearing symbolic closure

Typical examples:

- bins,
- arrays,
- field plots,
- transform-space exposure.

---

### 9.2 R2 — Support/accounting regime

Handles, counts, lineage, and retention posture are active.

Best fit:

- S1
- S2

High-risk:

- S3+ if support language begins answering memory or identity questions.

Typical examples:

- receipt traces,
- H1/M1 counts,
- retained-tier posture,
- support basis descriptors.

---

### 9.3 R3 — Semantic overlay regime

Late-bound bounded interpretation is active.

Best fit:

- S2
- S3

High-risk:

- S4+ if overlays silently begin functioning as stronger review or authority surfaces.

Typical examples:

- bounded continuity posture,
- semantic triangulator outputs,
- sparse descriptor-backed overlays.

---

### 9.4 R4 — Review/workflow regime

Classification, routing, readiness, or packet movement is active.

Best fit:

- S3
- S4

High-risk:

- S5 if packet prose or review surfaces begin pre-deciding implementation or structural exposure.

Typical examples:

- review summaries,
- packet titles,
- route posture,
- readiness blockers,
- role labels.

---

### 9.5 R5 — Mixed-surface regime

More than one role cohabits one active surface.

Best fit:

- none by default

High-risk:

- S3–S5, especially when symbolic and evaluative layers become cognitively primary.

Typical examples:

- dashboards that combine raw structure, support, semantics, and review,
- “viewer” surfaces that try to expose and interpret at once,
- app shells where semantic helpfulness outruns structural ordering.

One-line summary:

**Mixed surfaces amplify symbolic gravity.**

## 10. Symbolic gravity and naming

Names deserve special treatment because they often act before any rendered label appears.

### 10.1 Naming is a pre-render symbolic force

A name can alter:

- the implied query class,
- the expected answer posture,
- the representation chosen,
- the layer that becomes primary,
- and the closure pressure felt by the implementer.

A surface named “viewer” often begins optimizing for readable interpretation.

A surface named “bin exposure surface” begins optimizing for direct structural exposure.

### 10.2 Naming classes

Suggested shorthand:

- **N0** — inert technical naming
- **N1** — descriptive but local naming
- **N2** — convenience naming
- **N3** — interpretive naming
- **N4** — authority-bearing naming

This shorthand can later be mechanized, but for now the important rule is:

**surface names should be reviewed as symbolic control inputs, not treated as harmless labels.**

### 10.3 Naming rule

If a name implies explanation, readability, insight, memory, or evaluation more strongly than direct exposure, it should be treated as symbolic gravity risk and reviewed before use.

## 11. Symbolic gravity and packets

Symbolic gravity can enter before implementation through packet prose.

### 11.1 Packet phrases are not neutral

Terms like:

- viewer
- inspector
- insight
- state
- memory map
- structure summary
- comparison dashboard

may silently change the active query class or surface role.

### 11.2 Packet gravity rule

A packet that asks for structural exposure should use structural-only naming and structural-only bounded questions.

If the prose already contains review, semantic, or readiness language, the packet may already be drifting.

### 11.3 Packet containment

For structure-first packets, prefer phrases such as:

- expose raw bins
- render transform output directly
- show declared-lens arrays
- plot geometric traces
- export relation-bearing output

Avoid phrases such as:

- help understand the structure
- show what the signal is doing
- summarize the state
- visualize the memory behavior
- inspect the pattern

unless the packet explicitly belongs to a later semantic or review seam.

## 12. Symbolic gravity and workflow mechanization

Symbolic gravity should be considered in:

- packet creation,
- task titles,
- function naming,
- file naming,
- surface naming,
- UI copy,
- tooltip language,
- and README posture.

A seam can drift before code exists if the symbolic field around it is already too heavy.

Therefore:

- symbolic gravity is a workflow concern,
- not only a runtime or UI concern.

---

## 13. Detection questions

When evaluating a suspected symbolic gravity problem, ask:

1. What is the active bounded question?
2. What symbols are present?
3. What is each symbol anchored to?
4. What distinctions is each symbol compressing?
5. What stronger class or authority does each symbol imply?
6. How far does each symbol propagate?
7. Did the symbol arrive before raw exposure?
8. If the symbol is removed, does direct inspectability improve?

If removing a symbol improves exposure, the symbol was exerting meaningful gravity.

---

## 14. Containment posture

When symbolic gravity becomes unsafe, the lawful responses include:

- rename the surface,
- strip non-essential labels,
- split mixed surfaces,
- move semantics downstream,
- restore raw export first,
- narrow the packet question,
- reduce authority-bearing prose,
- and reclassify the surface honestly.

The system should prefer:

- functionally centered but exposure-first naming,
- delayed closure,
- local descriptor anchoring,
- and reversible semantic overlays

over convenience, elegance, or human-friendly summary.
