## 1. Why this note exists

DME now has enough replay, retention, compression, memory, observability, feedback, coherence, invariance, and identity pressure that the project needs one explicit place to answer a recurring question:

**When has a seam changed enough that the old continuity language is no longer safe?**

Without a bounded drift taxonomy, several distortion risks become active:

- slow change being mistaken for preserved sameness,
- narrowed support being mistaken for unchanged support,
- changed question being mistaken for deepened question,
- changed boundary being mistaken for preserved seam,
- class inflation appearing gradually rather than abruptly,
- and continuity language surviving long after the original conditions have shifted.

This note exists to prevent those drifts.

---

## 2. Core rule

**In DME, drift is not mere change. Drift is bounded movement away from the declared question, constraints, support, boundary, class, or continuity conditions that originally justified a claim.**

Corollary rules:

- change is not automatically drift,
- variation is not automatically failure,
- adaptation is not automatically drift,
- and drift is not automatically rupture.

But:

- when the active bounded seam is no longer governed by the same question, constraints, support, or class conditions,
- the system must not silently preserve the stronger continuity claim.

It must instead narrow, downgrade, split, reclassify, defer, or emit unresolved posture.

## 4. Definition of drift

In DME, **drift** means:

> the bounded shift of an object, seam, question, support surface, or classification posture away from the declared conditions that originally justified its continuity claim, strongly enough that the same bounded question may no longer remain answerable under the same terms.

This means drift is always:

- question-relative,
- constraint-relative,
- support-relative,
- boundary-relative,
- class-relative,
- regime-relative,
- and mechanization-relative where a real path matters.

Drift is **not** guaranteed merely because:

- something varies,
- something adapts,
- something updates,
- something is re-rendered,
- something is reused,
- or something persists over time.

---

## 5. Drift is not distortion, uncertainty, or failure

These are related, but not the same.

### Distortion

Distortion is structured deviation from required invariance.

Drift is the gradual or bounded shift in seam conditions that may later produce distortion.

### Uncertainty

Uncertainty is lawful insufficiency in justifying a stronger claim.

Drift may cause uncertainty, but drift can also be clearly detectable.

### Failure

Failure means the stronger claim no longer holds.

Drift is often the precursor that pushes a seam toward failure, but may still remain inside bounded tolerances.

One-line distinction:

**Distortion is what bent. Uncertainty is what cannot yet be justified. Drift is what moved away from the original seam conditions.**

## 6. The five drift classes

The current bounded taxonomy recognizes five primary drift classes.

---

### 6.1 G0 — Bounded variation / no meaningful drift

#### Definition

Change occurs, but the active bounded question, constraints, support, and class remain intact strongly enough that no drift-sensitive downgrade is required.

#### Bounded question

**Has anything moved beyond lawful seam variation?**

#### Primary role

- separate lawful variation from actual drift,
- preserve bounded adaptability without false alarms,
- keep small change from being mistaken for continuity failure.

#### Common examples

- mild replay variation under same lens and support,
- bounded perturbation that preserves the same memory or identity class,
- cosmetic or local update that does not alter active seam conditions.

#### Main failure modes

- false drift detection,
- over-tight seam policing,
- misreading lawful adaptation as degradation.

#### One-line summary

**G0 is lawful variation without meaningful drift.**

---

### 6.2 G1 — Question drift

#### Definition

The active object or seam is now answering a different bounded question while stronger continuity language from the prior question is still being preserved.

#### Bounded question

**Is this still the same question, or has the seam silently shifted what it is answering?**

#### Primary role

- protect question-relative continuity,
- detect when a seam has changed its meaning of “same object,”
- prevent broad resemblance or broader use from masquerading as same-question continuity.

#### Common examples

- replay support question becoming broad interpretation question,
- repair object becoming redesign object,
- same-family comparison becoming broad similarity search,
- narrow continuity question becoming general semantic question.

#### Main failure modes

- semantic inflation,
- hidden scope widening,
- false continuity through reworded purpose.

#### One-line summary

**G1 is the shift from one bounded question into another.**

---

### 6.3 G2 — Constraint / boundary drift

#### Definition

The same surface or object is being interpreted under changed lens, tier, interface, allowed-use, or boundary conditions while preserving the old continuity language.

#### Bounded question

**Are the declared seam conditions still the same, or has the object crossed into a different constraint/boundary posture?**

#### Primary role

- preserve declared seam honesty,
- detect lens shifts, tier shifts, interface shifts, and crossing-condition changes,
- prevent same-surface appearance from masking changed governing rules.

#### Common examples

- changed retained tier treated as same replay claim,
- changed lens treated as same support class,
- changed interface treated as same observation boundary,
- narrowed or widened allowed use without explicit class change.

#### Main failure modes

- same-look different-law illusion,
- hidden tier inflation,
- boundary shift disguised as stable continuity.

#### One-line summary

**G2 is drift in the governing seam conditions.**

---

### 6.4 G3 — Support / memory drift

#### Definition

The object remains related to its prior support path, but the surviving support, lineage, or memory class has weakened or shifted enough that the old continuity claim is no longer stable.

#### Bounded question

**Does enough of the original support still survive, or has the continuity basis itself drifted?**

#### Primary role

- detect support thinning,
- detect lineage weakening,
- detect when memory class has shifted without being renamed,
- preserve support-aware downgrade posture.

#### Common examples

- M2 object drifting toward M1 while still spoken of as operational memory,
- retained summary drifting away from receipt-backed support,
- replay-support object becoming support-only,
- cross-session continuity becoming recollection-dependent rather than evidence-backed.

#### Main failure modes

- memory overclaim,
- support inflation,
- support-only surface treated as memory-bearing,
- continuity language preserved after support collapse begins.

#### One-line summary

**G3 is drift in the surviving continuity basis.**

---

### 6.5 G4 — Identity / class drift

#### Definition

The object continues to appear related to its prior identity or class, but its actual class conditions have shifted enough that the stronger prior identity or classification language is no longer lawful.

#### Bounded question

**Is this still the same class of object, or has it become a different continuity object while retaining the old name?**

#### Primary role

- detect class inflation or misclassification,
- protect the identity hierarchy,
- prevent lower-class continuity from being silently stretched upward,
- surface “new object required” conditions before stronger misuse accumulates.

#### Common examples

- I2 object drifting toward support-only or unresolved posture,
- symbolic-looking continuity being overread from only mechanical support,
- memory-bearing object drifting into narrative convenience object,
- stitched cross-layer object being treated as same whole continuity object.

#### Main failure modes

- class inflation,
- misclassification,
- renamed continuity without renamed object,
- false sameness via gradual reinterpretation.

#### One-line summary

**G4 is drift that changes what kind of object this now is.**

---
## 6.6 G5 Symbolic-gravity drift

DME now recognizes a bounded drift family in which names, labels, packet prose, summaries, or semantic posture shift a seam away from its originally declared structural role before or during implementation.

### Core rule

A seam exhibits symbolic-gravity drift when symbolic material changes the active bounded question, surface class, authority posture, or representation class without explicit reclassification.

### Typical pathways

- structural exposure ask -> semantic overlay result
- raw geometry ask -> support/accounting result
- structural packet title -> mixed-surface implementation
- exposure-first surface -> review-shaped surface
- local descriptor -> authority-bearing label cascade

### Drift mappings

#### G1-S — Symbolically induced question drift

The seam begins answering a stronger or different question because the wording or labels implied a broader closure target.

#### G2-S — Symbolically induced surface/boundary drift

The seam changes from raw structural exposure into semantic, support, review, or mixed-surface posture without explicit reclassing.

#### G3-S — Symbolically induced support drift

Support/accounting material becomes the primary visible form of a structural seam, weakening direct exposure.

#### G4-S — Symbolically induced class drift

Labels or prose pull the seam into stronger memory, identity, or readiness posture than the current support directly justifies.

### One-line summary

**Symbolic-gravity drift is seam drift caused not by new evidence, but by the closure pressure exerted by names, labels, and packet language.**

## 7. Drift staircase

The five classes form a practical staircase:

- **G0** = lawful variation / no meaningful drift
- **G1** = question drift
- **G2** = constraint / boundary drift
- **G3** = support / memory drift
- **G4** = identity / class drift

This staircase should remain explicit.

The system must not silently round drift downward.

Examples:

- G1 is not automatically “same question but richer”
- G2 is not automatically “same seam but updated”
- G3 is not automatically “same memory but weaker”
- G4 is not automatically “same object but evolved”

---

## 8. Drift functions

Each drift class can be analyzed through three functions.

### 8.1 Origin function

**What original seam condition is being drifted from?**

Examples:
- bounded question,
- declared lens,
- retained tier,
- support basis,
- memory class,
- identity class.

### 8.2 Shift function

**What actually changed?**

Examples:
- scope widening,
- constraint change,
- support thinning,
- boundary crossing,
- class reinterpretation.

### 8.3 Consequence function

**What does this shift threaten or break if preserved too long?**

Examples:
- question-preserving identity,
- support sufficiency,
- memory class legitimacy,
- same-object continuity,
- minting readiness.

This three-function view prevents drift from collapsing into vague “change over time” language.

## 9. Drift regimes

Drift classes do not operate in a vacuum.

The same seam may exhibit different drift classes under different regimes.

The current taxonomy recognizes five practical regimes.

---

### 9.1 R1 — Stable regime

Conditions are steady and bounded classification is repeatable.

Best fit:

- G0

Typical use:

- same-lens same-question review,
- stable retention/replay seam,
- steady packet identity.

---

### 9.2 R2 — Narrow perturbation regime

Small lawful variation occurs, and the system must distinguish adaptation from true drift.

Best fit:

- G0
- G1
- G3

Typical use:

- bounded replay change,
- narrow continuity under mild perturbation,
- threshold-sensitive memory or support evaluation.

---

### 9.3 R3 — Compression regime

Reduction pressure increases the risk that preserved seams no longer answer the same question or preserve the same support class.

Best fit:

- G1
- G3
- G4

Typical danger:

- compressed support drifting into support-only or misclassified identity without explicit rename.

---

### 9.4 R4 — Boundary-shift regime

The active lens, tier, interface, or seam changes.

Best fit:

- G2
- G3
- G4

Typical danger:

- changed seam conditions hidden by similar surface appearance.

---

### 9.5 R5 — Rupture regime

The active continuity claim can no longer be preserved honestly.

Best fit:

- G4

Typical danger:

- drift treated as continuity instead of trigger for split, downgrade, or new object.

---

## 10. Drift is not evolution, growth, or refinement alone

Several convenient stories must remain fenced.

### 10.1 Growth is not automatically continuity

A seam can expand and cease to be the same seam.

### 10.2 Refinement is not automatically same-question refinement

A cleaner answer can be answering a different question.

### 10.3 Adaptation is not automatically class preservation

A structure can respond well while changing what class of object it is.

### 10.4 Persistence is not automatically same-object preservation

A thing can remain present while no longer remaining the same continuity object.

---

## 11. Drift-bearing vs support-only posture

A central dynamics distinction is:

### Drift-bearing

The system can explicitly identify what is drifting and preserve lawful downgrade or split posture.

### Support-only

The object remains useful as trace, lineage, memory hint, or related support, but the old stronger continuity language should no longer be preserved.

### Hidden drift

The system continues to use older seam language after the active seam conditions have already shifted.

This distinction should remain explicit.

The system must not casually call something stable when it is only:

- still useful,
- still similar,
- still replay-shaped,
- still name-preserving,
- or still narratively convenient.

- constraint drift is a major precursor to continuity downgrade,

## 27. Drift failure modes

Drift most commonly fails through the following modes.

### 27.1 Silent widening

The seam answers a broader question while preserving the old bounded language.

### 27.2 Silent narrowing

The seam now supports only a weaker use while preserving the old stronger claim.

### 27.3 Seam substitution

A different lens, boundary, or support regime is now active but named as the old one.

### 27.4 Support erosion

Continuity basis weakens gradually without explicit class downgrade.

### 27.5 Class overhang

The object behaves like a weaker or different class while still carrying the old class name.

### 27.6 Drift concealment

Smooth surfaces, coherent summaries, or clean rendering hide that the seam has already moved.

## 29. Drift outcomes

A practical drift audit should produce one of the following broad outcomes.

### 29.1 No meaningful drift
G0 holds.

### 29.2 Question drift active
G1 holds.

### 29.3 Constraint / boundary drift active
G2 holds.

### 29.4 Support / memory drift active
G3 holds.

### 29.5 Identity / class drift active
G4 holds.

### 29.6 Drift resolved by narrowing
The same object remains lawful only under a smaller question or use.

### 29.7 Drift resolved by reclassification
The object is lawful again only once renamed or reclassed.

### 29.8 Drift unresolved
The seam has moved, but the right downgrade/split posture is not yet closed.

## 31. Decision rule

Preserve a stronger continuity claim only when all of the following are true strongly enough for the active bounded question:

the original bounded seam remains explicit,
the same bounded question remains active,
the same declared constraints remain active,
enough support survives for the original seam to remain answerable honestly,
any required memory, boundary, or identity class remains stable enough,
and any required implementation path is actually mechanized.

If not, the seam may still be useful.

But it must not silently preserve the stronger continuity claim.

## 32. One-line practical rule

In DME, drift is the bounded movement away from the original seam conditions that justified a claim; when that movement becomes continuity-relevant, the system must name the drift and weaken the claim rather than preserve stronger sameness language by convenience.

## 33. One-line summary

The DME drift taxonomy distinguishes lawful variation, question drift, constraint/boundary drift, support/memory drift, and identity/class drift so seam change can be discussed with bounded honesty, explicit downgrade posture, and clear linkage to support, continuity, and review claims.
