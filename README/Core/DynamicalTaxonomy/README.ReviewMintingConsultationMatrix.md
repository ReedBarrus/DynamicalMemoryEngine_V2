## 1. Why this note exists

DME now has a full taxonomy stack, but lacks a unified evaluation surface.

Without a review matrix:

- taxonomy remains descriptive instead of operational,
- continuity claims drift during evaluation,
- support is mistaken for memory,
- memory is mistaken for identity,
- identity class inflation occurs,
- and canon decisions become rhetorical instead of structured.

This note exists to convert taxonomy into **decision logic**.

---

## 2. Core rule

**Review, consultation, and minting in DME must operate on classed, bounded, and auditable claims — not on resemblance, usefulness, or narrative coherence alone.**

Corollary rules:

- useful is not automatically valid,
- preserved is not automatically memory,
- memory is not automatically identity,
- lower identity class is not automatically higher identity class,
- and candidate is not automatically mintable.

If a stronger claim cannot be supported, the system must **downgrade rather than inflate**.

---

## 3. Three evaluation layers

The matrix operates across three distinct layers.

### 3.1 Review

- classification and diagnosis
- no promotion intent required
- outputs bounded assessment

### 3.2 Consultation

- advisory interpretation of review results
- explores implications and risks
- remains below promotion

### 3.3 Minting

- decision surface for canon-support
- requires strongest support conditions
- must pass all required thresholds without inflation

---

## 4. Evaluation axes

Each object is evaluated across the following axes.

---

### 4.1 Constraint posture

**Which constraint classes hold?**

- C0 formal
- C1 structural
- C2 boundary
- C3 dynamical
- C4 persistence
- C5 coupling / feedback
- C6 cross-layer consistency

Questions:
- which constraints are satisfied?
- which are degraded or broken?
- is there constraint drift?

---

### 4.2 Boundary posture

**What boundary class is actually active?**

- B0 contact
- B1 observational
- B2 transform
- B3 retention
- B4 identity

Questions:
- what kind of crossing is lawful?
- what is preserved across the boundary?
- is boundary class being overclaimed?

---

### 4.3 Observability / feedback posture

**What is detectable and consequential?**

Observability:
- O0–O3 (contact → registered distinction)

Feedback:
- F0–F3 (contact → state consequence)

Questions:
- what distinctions are actually detectable?
- what interactions are actually consequential?
- is feedback real or decorative?

---

### 4.4 Memory posture

**What class of memory is present?**

- M0 retentive
- M1 lineage
- M2 operational
- M3 adaptive

Questions:
- is this retained only, or reusable?
- does it support the same bounded question?
- is it robust under perturbation?

---

### 4.5 Identity posture

**What identity class is being claimed and what is actually preserved?**

- I0 mathematical
- I1 geometric
- I2 mechanical/topological
- I3 symbolic
- I4 human earthly

Questions:
- what class is claimed?
- what class is actually supported?
- is there class inflation?

---

### 4.6 Support vs identity distinction

**What is actually preserved?**

- support-only
- memory-bearing
- identity-bearing

Questions:
- is this just support?
- does memory exist without identity?
- is identity being overclaimed?

---

### 4.7 Mechanization posture

**Is there a real implementation path?**

Questions:
- is this declared or mechanized?
- is the path reachable?
- is rendering honest?
- is there a mechanization gap?

---

### 4.8 Regime posture

**Under what regime is this being evaluated?**

- R1 stable
- R2 perturbation
- R3 compression
- R4 boundary shift
- R5 rupture

Questions:
- does the claim hold across regimes?
- is it regime-sensitive?

---

## 5. Identity–memory distinction rule

A central evaluation rule:

- support may exist without memory
- memory may exist without identity
- identity requires memory + constraints + boundary + support

Therefore:

**identity claims must be evaluated last, and only after lower classes are verified.**

---

## 6. Class inflation prevention rule

The system must explicitly check:

- M1 → M2 inflation
- M2 → M3 inflation
- I2 → I3 inflation
- I3 → I4 inflation

If detected:

→ downgrade required

---

## 7. Evaluation outcomes

Each evaluation produces one of the following outcomes.

### 7.1 Valid (bounded)

Claim holds at declared class.

### 7.2 Narrowed

Claim holds only under stricter conditions.

### 7.3 Degraded

Partial continuity remains.

### 7.4 Support-only

No valid memory or identity claim.

### 7.5 Misclassified

Wrong class claimed.

### 7.6 Unresolved

Insufficient support.

### 7.7 Invalid

Claim fails under constraints.

---

## 8. Consultation layer

Consultation interprets review results.

It may:

- explore risk of inflation
- identify missing support
- suggest downgrade posture
- propose reframing of bounded question
- assess readiness for minting

It must not:

- silently upgrade class
- assume continuity
- override constraint failure

---

## 9. Minting conditions

Minting requires:

1. correct class identification
2. constraint satisfaction
3. boundary honesty
4. sufficient memory class
5. identity class validity
6. mechanized support (where required)
7. no class inflation
8. no unresolved critical failures

If any fail:

→ minting must not proceed

## 11. Decision rule

No object may be promoted above the strongest class it can support under the current bounded question, constraints, support, and mechanized basis.

## 12. One-line practical rule

Review classifies, consultation interprets, minting decides — and all three must obey constraint, boundary, memory, and identity limits without inflation.

## 13. One-line summary

The DME review, minting, and consultation matrix converts taxonomy into a bounded decision system that evaluates constraint, boundary, observability, memory, and identity classes to prevent inflation and enable lawful canon-support decisions.
