## 1. Why this note exists

DME now has enough execution surfaces, helper models, request objects, replay objects, review objects, and app composition that some capabilities can appear more complete than they actually are.

This is often normal architectural progress:

posture becomes clear,
UI surfaces become legible,
object models become coherent,
and only later does the full backend mechanism arrive.

The risk is that surface clarity can outrun mechanism.

This note exists to prevent that drift.

## 2. Core rule

Declared is not mechanized.
Displayed is not mechanized.
Prepared is not fulfilled.
Candidate is not canon.
Replay-shaped is not reconstructed.

Any capability in DME should be understood through three distinct questions:

Is it declared?
Is it displayed?
Is it mechanized?

Only the third should be treated as actual functional attainment.

Where partial support exists, the capability should remain visibly below full mechanization rather than being rounded upward.

## 3. Audit categories
### 3.1 Declared

The capability has explicit posture, object shape, boundary language, or intended interface, but no real implementation path yet.

### 3.2 Displayed

The capability has visible UI or read-side representation, but may still depend on:

local state,
static shaping,
incomplete backend support,
or manual handoff.

Displayed is not mechanized.

### 3.3 Partially mechanized

The capability has some real implementation substance, but still lacks one or more required closure conditions.

Examples:

backend exists but no real rendering path
rendering exists but backend is not actually called
failure posture is hidden or softened
downgrade posture is not explicit
the surface implies more than the mechanism supports
### 3.4 Mechanized

The capability has a real bounded implementation path and has passed enough of its local closure conditions to count as actual functional attainment at its declared seam.

Mechanized does not imply:

truth
canon
global maturity
permanence
completion of adjacent seams

## 5. Distortion classes to watch for

The following distortions remain active project risks.

### 5.1 Declared posture mistaken for mechanism

A doc/object/surface is described clearly enough that its shape is mistaken for actual implementation.

### 5.2 Displayed surface mistaken for capability

A visible pane, panel, or object gives the impression that the backend function is complete when it is only shaped or partially wired.

### 5.3 Prepared mistaken for fulfilled

Request, replay, review, or candidate objects are treated as if their downstream protocol has actually executed.

### 5.4 Replay-shaped mistaken for reconstructed

A replay-facing object or panel is treated as if it preserved reconstructed identity more strongly than the actual replay/reconstruction seam justifies. This must be judged using deterministic invariance and replay-honesty law, not surface comfort.

### 5.5 Candidate mistaken for canon

Readiness, dossier, or review visibility is treated as promotion.

### 5.6 Composed surface mistaken for backend maturity

A coherent app environment feels more mature than the seams beneath it actually are. This is one of the biggest current product-surface risks.


## 7. One-line review question

Before calling any seam or surface “real,” ask:

Is this capability only declared, only displayed, partially mechanized, or truly mechanized at its bounded seam, and am I overreading structural clarity or surface coherence as if it were stronger backend attainment than the current implementation actually justifies?

## 8. One-line summary

This note classifies DME capabilities by declared, displayed, partially mechanized, or mechanized status so surface clarity, prepared objects, and composed app coherence do not get mistaken for stronger functional attainment than the repo actually supports.
