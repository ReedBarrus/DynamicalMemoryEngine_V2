## 1. Why this note exists

DME now has:

constraint → what is lawful
boundary → where things are delimited
observability/feedback → what is detectable and consequential
memory → what is preserved
coherence → how strongly things hold together
identity → what remains the same

But we are missing one explicit layer:

What specifically must remain unchanged for any of those claims to be valid?

Without invariance:

coherence can be overread as identity
similarity can be mistaken for preservation
compression can silently erase required structure
transformation can drift while appearing stable
identity claims can persist without preserved invariants

This note exists to anchor non-negotiable preservation conditions.

## 2. Core rule

In DME, invariance is the bounded set of properties, relations, structures, or conditions that must remain unchanged for a given class of constraint, coherence, memory, or identity claim to remain valid under a declared question.

Corollary rules:

similarity is not invariance
stability is not invariance
coherence is not invariance
usefulness is not invariance
replay resemblance is not invariance

If required invariants are not preserved:

→ the system must downgrade the claim, not reinterpret the invariants.

## 3. Definition of invariance

In DME, invariance means:

a property, relation, structure, constraint, or support condition that must remain preserved across transformation, perturbation, time, or boundary crossing in order for a bounded question to remain answerable under the same identity class.

Invariance is always:

question-relative
class-relative
constraint-relative
boundary-relative
coherence-relative
regime-relative

Invariance is not guaranteed because:

something looks similar
something remains coherent
something compresses well
something behaves consistently
something is still usable
## 4. Invariance is not identity, coherence, or constraint

These are tightly coupled but distinct:

Constraint → what must be true
Invariance → what must not change
Coherence → how strongly things hold together
Identity → what remains the same

Key relationship:

Constraints define the rules. Invariance defines what must survive those rules. Coherence measures how well they survive. Identity is what remains the same if they do.

Summary

V5 is preserved cross-layer alignment.

## 9. Invariance vs coherence (critical distinction)

This is the key pairing:

Coherence: how well it holds together
Invariance: what must not change

You can have:

high coherence + low invariance → smooth but wrong
low coherence + preserved invariance → noisy but valid

This is HUGE for:

compression
reconstruction
anomaly
identity audit
## 10. Invariance vs identity

Identity requires invariance.

But:

invariance can exist without identity
identity failure = invariance failure at required class

Mapping:

I0 → V0
I1 → V1
I2 → V1–V5
I3 → V4–V5 + symbolic invariants
I4 → full stack + embodied invariants
## 11. Invariance vs compression

Compression is:

selective destruction of invariance

So every compression step must answer:

which invariants are preserved?
which are degraded?
which are destroyed?

Otherwise → identity inflation risk.
