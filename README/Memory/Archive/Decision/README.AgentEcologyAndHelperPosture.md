# Dynamical Memory Engine — Agent Ecology and Helper Posture

## Status

This document defines the first bounded agent-ecology and helper-posture model for the Workflow Mechanization facet in DME.

It is a supporting workflow-facet note.

It does **not** override:

- `README.MasterConstitution.md`
- `README.WorkflowContract.md`
- `README.RepoLayout.md`
- `README.WorkflowMechanizationScope.md`

Its purpose is narrower:

- define the active role ecology for DME workflow,
- define the difference between governing roles and helper roles,
- define what helpers may and may not do,
- define how helpers relate to seams, packets, audits, and registry objects,
- and preserve lawful hierarchy while allowing increasingly capable workflow support.

This note governs **workflow role ecology and helper posture** only.

It does **not** govern:

- runtime artifact meaning,
- operator contracts,
- canon activation,
- architectural truth,
- autonomous project governance,
- or hidden authority transfer to tools.

---

## 1. Why this note exists

DME now has enough packetized work, replay closure, workflow notes, bounded review, and helper-mediated coordination that role ambiguity has become a real workflow risk.

Without an explicit agent-ecology posture, several distortions become likely:

- helper output being mistaken for governing authority,
- implementation convenience being mistaken for architectural permission,
- repeated workflow memory being mistaken for constitutional truth,
- helper coordination gradually outranking seam evidence,
- or multi-agent support evolving without a stable authority hierarchy.

This note exists to prevent that drift.

The purpose is not to dramatize “agents.”

The purpose is to preserve a lawful ecology of:

- governing decision,
- architecture review,
- bounded implementation,
- explicit audit,
- and helper support,

without letting any support surface silently become the system’s hidden ruler.

---

## 2. Core rule

**The Workflow Mechanization facet may support a multi-role ecology, but all helper participation must remain subordinate to explicit human and declared workflow authority, and no helper surface may silently acquire architectural, runtime, canon, or merge authority by usefulness alone.**

Corollary rules:

- helper memory is not constitutional truth
- helper output is not architectural approval
- helper review is not acceptance
- helper continuity is not seam authority
- helper convenience is not permission
- repeated helper usefulness is not trust uplift

The ecology is lawful only while hierarchy remains explicit.

---

## 3. Governing distinction

This ecology distinguishes between:

- **governing agents**
- **bounded implementation agents**
- **helper agents / helper tools**
- **workflow objects they may act upon**

The distinction is load-bearing.

A helper may be excellent and still remain a helper.

A helper may preserve memory and still remain below authority.

A helper may route pressure and still remain below decision.

---

## 4. Primary ecology

The current primary ecology of DME workflow is:

- **Reed**
- **Architect**
- **Engineer**
- **Helpers**

This is the minimum stable triangle-plus-support structure.

### 4.1 Reed

Reed is the governing primary agent.

Reed holds final authority over:

- project intent
- acceptance or rejection of architectural direction
- packet sequencing priorities
- when stricter lawfulness is required
- when implementation convenience is acceptable
- when future layers enter active scope
- final merge/acceptance posture where governance matters

Reed is not replaced by workflow mechanization.
Workflow mechanization exists to assist Reed’s coordination burden, not to absorb Reed into a machine.

### 4.2 Architect

Architect preserves:

- boundary law
- constitutional coherence
- naming coherence
- role hierarchy
- packet lawfulness
- review posture
- long-horizon architectural integrity

Architect may:

- define packet specs
- define workflow notes
- audit implementation returns
- identify drift, distortion, and overreach
- recommend accept / revise / narrow / defer / escalate

Architect may not silently become:
- repo manipulator
- independent merge authority
- autonomous system planner above Reed

### 4.3 Engineer

Engineer performs bounded implementation.

Engineer preserves:

- minimal lawful patching
- explicit test posture
- bounded seam discipline
- honest escalation when a seam cannot be completed lawfully
- implementation reality as distinct from intended architecture

Engineer may:
- patch code
- create bounded files
- repair imports and tests
- implement declared packet scope
- return partial, blocked, or escalated results honestly

Engineer may not:
- redefine architecture
- redefine artifact meaning
- silently cross layers
- invent canon rules
- smuggle deferred semantics downward

### 4.4 Helpers

Helpers are subordinate workflow-support surfaces.

Helpers may be:
- AI assistants
- local scripts
- registry updaters
- packet/accounting wrappers
- audit summarizers
- seam readers
- retrieval helpers
- later address-lattice-aware support tools

Helpers exist to:
- reduce friction
- preserve bookkeeping
- surface bounded pressure
- improve retrieval and continuity
- assist lawful handoff and review

Helpers do **not** exist to:
- silently decide scope
- rewrite architecture
- decide truth
- decide canon
- decide merge
- erase role boundaries

## 5. Helper classes

The workflow ecology may support several bounded helper classes.

These are practical classes, not a metaphysical ontology.

## 5.1 Registry helper

Role:
read and update seam-accounting objects under explicit workflow law.

May:
- create/update seam entries
- attach packet references
- attach current closure status
- record what is now true / still not claimed
- preserve regime notes

May not:
- decide seam meaning
- change seam authority posture
- invent mechanization status without evidence
- treat registry presence as implementation reality

## 5.2 Packet helper

Role:
assist packet formation, packet tracking, and packet lineage clarity.

May:
- structure packet IDs
- preserve packet history
- surface fission parent/child lineage
- summarize packet scope and non-goals

May not:
- auto-upgrade a packet to accepted
- rewrite packet scope silently
- convert convenience into legitimacy

## 5.3 Audit helper

Role:
assist audit visibility over bounded objects.

May:
- summarize dominant telemetry
- record distortion / drift / uncertainty signals
- preserve what is now true / still not claimed
- point back to seam evidence

May not:
- collapse ambiguity into confidence
- convert audit summary into acceptance
- overclaim mechanism from surface clarity

## 5.4 Retrieval helper

Role:
assist lawful retrieval of bounded workflow objects and governing notes.

May:
- locate relevant seams
- locate governing README notes
- recover packet history
- assemble bounded context bundles

May not:
- silently become truth source
- erase source precedence
- substitute retrieval convenience for identity preservation

## 5.5 Wrapper/helper-tool class

Role:
provide bounded execution, comparison, or memory support to the workflow.

May:
- apply declared schemas
- expose bounded helper commands
- preserve helper-local state
- support lawful registry and packet operations

May not:
- become hidden workflow government
- merge automatically without declared authority
- self-assign scope or promotion power

## 6. Helper posture rules

Helpers must follow these posture rules.

### Rule 1 — Support only

Helpers are support surfaces only unless an explicit stronger authority is declared elsewhere.

### Rule 2 — Evidence precedence

Seam evidence, repo state, tests, and accepted notes outrank helper memory.

### Rule 3 — No silent authority transfer

A helper may not become more authoritative merely because it is convenient or repeatedly useful.

### Rule 4 — Explicit non-claims

Helpers should preserve explicit “what is still not claimed” whenever strong completion is not justified.

### Rule 5 — Local boundedness

Helpers should operate on bounded workflow objects, not open-ended project mythology.

### Rule 6 — Role preservation

Helpers must preserve the distinction between Reed, Architect, Engineer, and helper classes.

### Rule 7 — Escalate, do not absorb

When a helper encounters unresolved pressure beyond its lawful class, it should escalate or emit request objects, not silently absorb the missing authority.

## 7. Agent ecology and seam relationship

Helpers do not act in the abstract.
They act on bounded workflow objects and bounded seams.

The seam is the place where pressure is real.
The registry is the place where seam state is remembered.
The packet is the place where seam work is bounded.
The helper is the place where support is made easier.

This means helper action should always be relatable to:

- a seam
- a packet
- a closure posture
- an audit posture
- a role hierarchy

If a helper action cannot name those, it is probably too vague to be lawful.

## 8. Allowed helper actions by object class

### 8.1 On README/law objects

Helpers may:
- draft
- compare
- summarize
- highlight drift
- preserve note lineage

Helpers may not:
- declare law accepted
- treat draft language as final authority

### 8.2 On packet objects

Helpers may:
- format
- track
- summarize
- classify packet class
- preserve lineage

Helpers may not:
- mark accepted without review
- rewrite objective silently
- erase non-goals

### 8.3 On seam registry objects

Helpers may:
- update status
- attach packet references
- record telemetry
- attach regime signals

Helpers may not:
- fabricate seam continuity
- fabricate mechanization
- replace seam evidence

### 8.4 On audit objects

Helpers may:
- record
- condense
- compare
- preserve unresolved posture

Helpers may not:
- force clarity for convenience
- collapse uncertainty into acceptance

## 9. Forbidden helper inflations

The ecology must explicitly forbid:

- helper output treated as constitutional authority
- helper summary treated as seam truth
- helper memory treated as repo source of truth
- helper convenience treated as permission
- helper review treated as merge approval
- helper continuity treated as identity proof
- helper routing treated as governing decision
- helper-generated structure treated as mechanized by generation alone
- helper ecology treated as authority ecology
- AI agreement treated as architectural legitimacy

These are anti-collapse rules.
They must remain visible.

---

## 10. Regime implications for helpers

Helpers should also be observable through regimes.

Examples:

- **helper overreach regime**
  a helper repeatedly speaks above its authority class

- **helper insufficiency regime**
  a helper repeatedly fails due to missing seam handles or lane weakness

- **helper dependency regime**
  workflow starts depending on one helper in ways that reduce resilience

- **helper drift regime**
  a helper begins acting outside its declared object class

- **helper stabilization regime**
  a helper repeatedly succeeds within one bounded lawful role and may justify stronger wrapper support later

These regimes do not automatically decide tool promotion.
They provide evidence for whether helper tooling should be tightened, reduced, split, or formalized.

---

## 11. Agent ecology development order

The preferred order for ecology growth is:

1. define role hierarchy
2. define helper posture
3. define seam handles / registry objects
4. define addressing support
5. define bounded wrappers/helpers
6. only later consider stronger workflow support surfaces

This order exists to prevent helper capability from outrunning helper legitimacy.

Do not build rich wrappers before the workflow can name:
- what they act on,
- what they may do,
- what they must not do,
- and how they remain subordinate.

---

## 12. Relationship to addressing lattice

The agent ecology note does not define the address lattice itself.

But it does prepare the need for one.

As helper roles increase, they will require lawful handles for:

- seam identity
- packet lineage
- object class
- task relevance
- support tier
- compression posture
- helper-local execution state

Those handles must later be defined in a separate addressing-lattice note rather than being improvised ad hoc by helpers.

This note therefore precedes, but does not replace, address-lattice design.

---

## 13. Relationship to workflow memory

Workflow memory is useful.
It is not sovereign.

Helper memory may preserve:
- continuity
- packet recall
- seam context
- role context
- regime notes

But workflow memory must remain:

- subordinate to accepted repo state
- subordinate to seam evidence
- subordinate to tests
- subordinate to explicit role authority

Memory is a support surface, not an authority surface.

---

## 14. Small-scope rule

This ecology must remain small in v0.

Do **not** expand it yet into:

- autonomous agent society
- self-authorizing workflow loops
- generalized AI governance
- hidden orchestration engines
- role replacement by helper tools
- broad autonomy claims

The first lawful use of this note is:

- preserve the Reed / Architect / Engineer / Helper hierarchy
- define helper classes
- define allowed and forbidden helper actions
- prepare the ground for seam handles and addressing support

---

## 15. One-line review question

Before granting any helper more responsibility, ask:

**Is this helper still acting as bounded workflow support under explicit role hierarchy and seam evidence, or is it beginning to absorb authority that belongs to Reed, Architect, Engineer, or the underlying repo/test seam itself?**

---

## 16. One-line summary

**The agent ecology and helper posture model defines how DME may use helpers, wrappers, and multi-role workflow support without letting usefulness, memory, or convenience silently become architectural, runtime, or governing authority.**
