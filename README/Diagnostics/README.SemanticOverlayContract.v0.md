1. Why this note exists

DME now has enough taxonomy, audit, and runtime/report surfaces that a specific risk has become active:

semantic richness can outrun structural legibility.

The recent operator/runtime semantic audit found that core Door One runtime remains mostly structural through ingest, alignment, windowing, transform, compression, anomaly, merge, substrate commit, and replay support preparation. The earliest clear semanticization seam appears later, beginning at TrajectoryInterpretationReport.js, then amplified by AttentionMemoryReport.js, PromotionReadinessReport.js, CanonCandidateDossier.js, CrossRunDynamicsReport.js, and finally co-located inside assembled workbench/HUD/read-side objects.

This confirms that the current problem is not that the core operators require semantics for correctness. The problem is that structural evidence and semantic overlay are being assembled together early enough that later surfaces can blur:

support into memory,
memory into identity,
identity into readiness,
readiness into stronger claim posture,
and interpretation into something that feels more like runtime substance than optional overlay.

This note exists to correct that drift without discarding the real value of semantic interpretation.

One-line summary:

Semantic overlays in Door One must become late-bound, bounded, removable interpretation lenses over structural evidence, not runtime substance and not verdict engines.

2. Core posture

Semantic overlays must remain:

late-bound
question-bound
descriptor-backed
downgrade-aware
claim-ceiling-aware
removable
evidence-addressable
non-authoritative
non-retentive as primary Door One substance

Corollary rules:

semantics are not runtime truth
semantics are not retention substance
semantics are not operator correctness
semantics are not canon
semantics are not promotion
semantics are not a shortcut around structural review

Master test:

If semantics are removed, the structural object must remain intact, inspectable, and sufficient for structural review.

3. What semantic overlay is for

The semantic overlay layer exists to provide bounded interpretive assistance over structurally inspectable evidence.

It is allowed to:

summarize bounded structural readings,
expose likely continuity/support posture under an explicit query class,
expose descriptor dynamics tied to structural signature,
surface uncertainty, drift, downgrade, and non-claim posture,
help humans navigate structural complexity,
and provide downstream review/readiness hints where explicitly allowed.

It is not allowed to:

replace structural evidence,
collapse runtime into verdicts,
persist semantic labels as Door One retention substance,
silently promote support readings into memory claims,
silently promote memory claims into identity claims,
silently promote readiness into approval or promotion,
or imply truth, canon, ontology, or source-equivalence.

One-line summary:

Semantic overlay is annotation, not substrate.

4. Governing rules
Rule 1 — Structural runtime remains primary

Door One runtime operators and structural runtime results remain structurally primary.

Semantic overlays must attach after structurally inspectable output exists.

They may not become a prerequisite for runtime correctness.

This directly follows the current audit finding that the core runtime is already mostly structural and the contamination point is later blending, not operator correctness.

Rule 2 — Semantics are question-bound

No semantic label without an active bounded query class.

Every semantic overlay must declare what kind of question it is answering.

No free-floating semantic summary is lawful.

Rule 3 — Semantics are descriptor-backed

No semantic posture without evidence descriptors tied to structural signature, support traces, thresholds, and relevant boundary/crossing conditions.

A label without inspectable support descriptors is not lawful semantic overlay.

Rule 4 — Semantics are downgrade-aware

Every semantic overlay must remain capable of expressing:

insufficiency,
ambiguity,
uncertainty,
drift,
degradation,
unresolved posture,
or bounded non-closure

where relevant.

Blanket positive labels are not enough.

Rule 5 — Semantics are ceiling-aware

Support readings must not silently become memory readings.

Memory readings must not silently become identity readings.

Identity readings must not silently become readiness or promotional posture.

Every overlay must declare its claim ceiling.

Rule 6 — Semantics are sparse

Prefer:

one primary posture,
a few descriptor dynamics,
a compact caution posture,
and explicit non-claims

over many regimes, banners, verdicts, or rhetorical summaries.

The overlay should not recreate the semantic density that reduced human legibility in the first place.

Rule 7 — Semantics are removable

If the overlay is deleted:

runtime correctness must remain,
structural legibility must remain,
support evidence must remain,
and structural review must still be possible.
Rule 8 — Semantics are evidence-addressable

Every overlay output must point back to explicit evidence references.

No semantic prose should stand alone without evidence refs.

Rule 9 — Semantics are not retention substance

Semantic overlays may be:

generated on demand,
cached optionally,
exported as read-side aids,

but they must not become the primary retained Door One substance.

Retention remains structural-first:
receipts, traces, lineage, thresholds, transforms, support descriptors, structural relations.

The audit already identified overlay persistence drift as a real risk.

Rule 10 — Interpretation reports are bounded semantic lenses, not verdict engines

Interpretation reports should not act like free-floating closure generators.

They should answer bounded questions with bounded semantic posture over inspectable evidence.

That is their role.

5. Overlay placement rule

The semantic overlay seam must be explicitly late-bound.

The runtime semantic audit found that the repo already behaves as if such a seam exists, but currently in distributed form across trajectory, attention, readiness, dossier, workbench, HUD, replay/request/read-side helpers, and adapters.

The lawful target is:

structural_result
semantic_overlay
readiness_overlay
review_overlay

not one mixed runtime/workbench object.

This note does not require the whole final split immediately, but it defines the correct direction.

6. Primary posture model

A single label is not enough.

Suite A reinterpretation showed that top-line labels flatten preservable structure, especially in the pulse-family A7/A8/A9 region, where runtime evidence distinguishes degraded, unresolved discontinuity pressure, and narrowed-conserved irregular retention more sharply than old surface labels did.

Therefore semantic overlay should operate as a semantic triangulator:

A. Primary posture

One restrained top-level reading.

B. Descriptor vector

A small set of structural-signature-derived descriptors that explain why the primary posture was chosen.

C. Caution posture

A compact uncertainty/drift/downgrade marker set that prevents overclosure.

Compressed form:

primary posture + descriptor vector + caution posture

That is the semantic triangulator.

7. Primary posture vocabulary

For v0, keep the top-line posture set sparse.

Recommended continuity-heavy core set
bounded_conserved
narrowed_conserved
degraded
unresolved
broken

These are especially appropriate for continuity/identity-adjacent readings.

Important constraint

Do not force every query class into this exact posture vocabulary.

Q0, Q1, Q3, and Q4 may need smaller query-local posture vocabularies, but they must still obey the same sparse overlay discipline.

This avoids label sprawl while preserving bounded semantics.

9. Descriptor design rules

Descriptors must be:

evidence-derived
structurally traceable
bounded
low-prose
non-authoritative
comparative when useful
sparse

Bad descriptor:

“memory-like intelligence coherence is growing”

Good descriptor:

recurrence_signature_strength: high
support_sparsity: moderate
boundary_stress: elevated
compression_pressure: low
drift_sensitivity: high
v0 descriptor count rule

To preserve usability:

each report should emit at most 3 primary descriptors
and at most 2 secondary descriptors

Anything beyond that belongs in deeper evidence drilldown, not the main overlay surface.

This is a critical anti-sprawl rule.

10. Query-class to overlay mapping

Every overlay must declare query_class.

Use the taxonomies as reasoning grammar, not label dump.

Q0 — Presence / observation

Allowed overlay:
descriptive only

No continuity closure.

Q1 — Structural

Allowed overlay:
structure-focused posture

No automatic memory or identity inflation.

Q2 — Continuity

Allowed overlay:
continuity posture below identity unless explicitly escalated.

This is where the continuity-heavy posture set is most natural.

Q3 — Support / lineage

Allowed overlay:
support-only closure

No automatic memory claim.

Q4 — Memory

Allowed overlay:
bounded memory-class interpretation

No automatic identity claim.

Q5 — Identity

Allowed overlay:
strongest current continuity posture

Must remain downgrade-heavy and threshold-heavy.

Q6 — Review

Allowed overlay:
evaluative posture

Must not silently become readiness or promotion.

Q7 — Consultation / readiness

Allowed overlay:
advisory posture only

Never promotion itself.

11. Claim ceiling mapping

Every overlay must declare claim_ceiling.

For v0, keep this compact.

Recommended v0 ceiling values
descriptive_only
support_only
identity_limited
review_only
readiness_only

These are enough for first implementation.

More detailed ceiling classes can come later if the seam truly needs them.

13. Explicit non-claims footer

Every overlay must end with explicit non-claims.

Recommended footer vocabulary

16. Storage and retention rule

Structural retention remains primary.

Semantic overlays may be:

late-bound,
regenerated,
optionally cached,
optionally exported,

but they must be clearly marked as:

derived
removable
non-authoritative
downstream of structural evidence

They may accompany retained objects.
They must not replace structural evidence.

17. UI rendering rule

The app panel should support at least three modes:

Mode A — Structural only

No semantics shown.

This mode must be useful on its own.

Mode B — Structural + semantic overlay

Overlay visible, but subordinate to structural evidence.

Mode C — Structural + semantic + review/readiness

Only for explicit downstream review use.

Visual rule

Structural evidence remains the visual anchor.

Semantic overlay is annotation, not substrate.

This directly answers the legibility concern raised in the prior thread: if semantics are turned off and the runtime becomes unreadable, the structural viewing surface is still incomplete.

18. Minimal acceptance tests for the overlay seam

A semantic overlay seam is acceptable only when:

structural result stands alone without semantics
overlay can be deleted without breaking runtime correctness
every overlay declares query_class
every overlay declares claim_ceiling
every overlay has evidence refs
descriptor count remains bounded
caution posture appears only where live
explicit non-claims are present
no overlay field can be mistaken for retention substance or runtime truth
review/readiness layers remain downstream of semantic overlay, not blended into base runtime
19. v0 implementation guidance

This note is intentionally more comprehensive than the first implementation wave.

It should not be implemented all at once.

The correct packet posture is:

extract the earliest semantic seam first,
pilot the v0 envelope there,
then expand downstream in bounded steps.

This preserves Codex bandwidth and obeys packet law.

Structural runtime is primary.
Semantic overlay is late-bound, sparse, evidence-backed, downgrade-aware, removable, non-retentive as primary substance, and never authoritative.
