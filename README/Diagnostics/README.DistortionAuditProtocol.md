Distortion Audit Protocol

Door One / RSI-aligned Fidelity and Accountability Framework

1. Purpose

This protocol defines a repeatable method for auditing whether a layer, operator, probe, or summary surface is compressing distinct structural realities into a smaller set of outputs strongly enough that downstream interpretation becomes misleading.

This is a fidelity tool, not a truth tool.

It does not:

establish canon
validate ontology
certify meaning
promote runtime outputs into authority

It does:

identify flattening
preserve ambiguity where needed
locate where distinguishability fails
guide architectural refinement without contaminating upstream layers
2. Core Rule

A layer should be refined only when its current compression begins to produce systematic interpretive error downstream.

Refinement is not justified merely because:

a richer vocabulary is imaginable
the current output feels coarse
a more expressive ontology would be aesthetically satisfying

Refinement is justified when:

distinct structural realities are being collapsed
that collapse materially affects downstream interpretation
the correction belongs at the audited layer or its immediate derived seam
3. Architectural Posture

This protocol operates under the following Door One / RSI-aligned assumptions:

runtime remains below canon
observation is not authority
structure is not meaning
derived read-side interpretation must remain auditable
ambiguity must remain visible where support is weak
upstream observational layers must not be burdened with downstream interpretive work

Accordingly, this protocol should be used primarily to determine:

whether a layer is still preserving the distinctions it is responsible for
whether a downstream seam should be added
whether a summary layer should be softened or expanded
whether a result should remain unresolved rather than overclassified
4. Audit Questions

For any layer under review, ask the following:

4.1 Intended role
What is this layer supposed to preserve?
What is this layer explicitly not supposed to decide?
What authority class does this layer belong to?
4.2 Distinctions under responsibility
What structural differences should this layer be able to keep separate?
Which of those differences matter to downstream interpretation?
4.3 Flattening risk
Which distinct regimes currently collapse into the same label, summary, or digest?
Are multiple lawful readings being merged into one convenient story?
4.4 Downstream consequence
Does the flattening create false sameness?
Does it erase meaningful ambiguity?
Does it encourage over-interpretation downstream?
Does it make different trajectories appear structurally identical?
4.5 Proper remedy
Should the fix occur at this layer?
Should the fix occur in a derived read-side probe?
Should the fix occur only in a summary/digest surface?
Should the output remain unresolved instead?
5. Distortion Classes

Use the following distortion classes during audit.

5.1 Label Flattening

Different structural realities yield the same label.

Example

ingress and rupture both appearing simply as anomaly
5.2 Summary Flattening

Rich local variation is collapsed into one file-level, run-level, or cohort-level story.

Example

several distinct transitions reduced to “return pattern observed”
5.3 Ontology Leakage

A read-side classification begins to sound like intrinsic truth or system nature.

Example

candidate boundary treated as event truth
probe-local exchange behavior treated as actual coupling law
5.4 Basis Drift

A comparison silently changes its lens, measurement frame, or resolution.

Example

changing band definitions between replay phases without declaring it
5.5 Ambiguity Erasure

Weakly supported structure is forced into a confident class instead of staying unresolved.

5.6 Scope Theft

A layer begins solving a problem that belongs to another layer.

Example

ingest performing interpretation
anomaly serving as full transition ontology
digest layer pretending to be a state object
6. Audit Outputs

Each audit should produce the following outputs.

6.1 Preserved distinctions

What the layer still distinguishes well.

6.2 Collapsed distinctions

What the layer is currently flattening.

6.3 Distortion severity

Use one of:

low
moderate
high
critical
6.4 Downstream impact

Which later surfaces are being misled and how.

6.5 Recommended action

Use one of:

keep_as_is
clarify_language_only
add_unresolved_posture
add_read_side_probe
refine_summary_surface
refine_operator_contract
defer_pending_more_evidence
7. Severity Rubric
Low

Compression exists, but downstream interpretation remains mostly honest and bounded.

Moderate

Important distinctions are being merged, but the risk can still be controlled through explicit language or unresolved posture.

High

The layer repeatedly causes false sameness, misleading summaries, or encourages over-interpretation.

Critical

The layer is causing authority confusion, ontology leakage, or major structural misclassification that materially distorts later reasoning.

8. Recommended Audit Order

Apply this protocol in the following order:

8.1 Read-side probes

Safest place to increase fidelity without contaminating runtime authority.

8.2 Basin summaries

Check whether compact summary language is masking distinct continuity patterns.

8.3 Anomaly relationship surfaces

Ensure anomaly remains deviation evidence and does not silently become a total transition explanation.

8.4 Digest / workbench summaries

Check whether ambiguity is being erased for readability or convenience.

8.5 Operator contract review

Only if flattening cannot be resolved at a derived seam or summary surface.

9. Layer-Specific Audit Prompts
9.1 Ingest

Ask:

Is ingest doing anything beyond acquisition, timestamping, and provenance retention?
Has interpretation leaked into observation?

Likely action:

keep ingest narrow and boring
9.2 Transform / compression surfaces

Ask:

Are the current basis choices collapsing distinctions that later layers assume are available?
Are different regimes becoming indistinguishable because resolution or partitioning is too coarse?

Likely action:

adjust basis only when distinguishability failure is demonstrated
9.3 AnomalyOp

Ask:

Are multiple transition types collapsing into the same anomaly pattern?
Is anomaly being treated as explanation rather than deviation evidence?

Likely action:

keep anomaly as evidence
move richer transition classification downstream
9.4 Basin summaries

Ask:

Does one basin summary correspond to multiple continuity realities?
Are split/merge/readiness terms masking branch vs fracture vs re-entry?

Likely action:

refine read-side interpretation before modifying the future basin-classification seam
9.5 Replay / transition probes

Ask:

Is an experiment scaffold being mistaken for primitive ontology?
Are labels such as baseline / perturb / return hiding richer structural regimes?

Likely action:

expand primitive vocabulary
preserve support basis
keep ambiguity visible
9.6 Digest layers

Ask:

Is the summary collapsing multiple lawful readings into one dominant story?
Does readability come at the cost of structural honesty?

Likely action:

add unresolved posture
report multiple local readings where needed
10. Primitive Accountability Rule

When a layer cannot distinguish enough to classify honestly, it must say so.

Preferred bounded outputs include:

unresolved
insufficient_support
not_applicable
ambiguous_transition
multiple_local_readings

Clarity should never be forced for convenience.

11. Lens Accountability Rule

Every audit must record the active lens.

At minimum, capture:

resolution
windowing
decimation
band definitions
segmentation rules
comparison basis
slicing constraints
fixed vs variable lens conditions

A layer cannot be judged fairly if its lens is drifting silently.

12. Support-Basis Rule

Every derived class, transition label, or summary should expose a compact support basis wherever practical.

Examples:

boundary_peak
profile_shift
recurrence_match
anomaly_density_change
basin_split_observed
weak_support_only

This ensures read-side interpretation remains auditable rather than merely persuasive.

13. Decision Rule for Refinement

Refine a layer only when all three conditions are true:

a meaningful distinction is being collapsed
that collapse matters downstream
the correction belongs at that layer rather than somewhere else

If any of the three are false, do not patch the layer yet.

15. Immediate Application Targets

Recommended first applications of this protocol:

continuous master transition probe
anomaly vs transition-class relationship
basin split / continuity summaries
workbench digest wording

This order is preferred because it increases fidelity first at the safest derived seams before touching broader summary or operator surfaces.

16. Practical Interpretation Rule

A layer is not “bad” merely because it compresses.

Compression is lawful when:

it removes redundancy
it preserves the distinctions the layer is responsible for
it does not create misleading downstream sameness

Compression becomes distortion when:

different structural realities become indistinguishable
ambiguity is erased prematurely
downstream layers start compensating with guesswork or ontology leakage
17. One-Line Summary

This protocol checks whether a layer is still preserving the distinctions it is responsible for, and tells us when to refine vocabulary, add a derived probe, preserve ambiguity, or leave the architecture alone.

## First real-source application example (400hz master.wav perturb)

The first tighter-band real-source family provided a concrete example of lawful distortion audit use.

Observed outcome:

- coarse replay and continuous-master detection were excellent
- phase separation and return structure were preserved strongly
- but the structural transition vocabulary still flattened two distinct mechanisms into the same class

Concrete flattening observed:

- prior broadband microphone perturbation:
  multi-band redistribution under room/fan/heater conditions
- tighter-band `400 Hz` sine perturbation:
  single-band concentration with much stronger boundary L1

Both currently appear as `rupture`.

This is a useful example of:

- preserved distinctions at one layer
- collapsed distinctions at a neighboring layer
- and a correction that belongs in a derived read-side probe rather than an operator contract

Recommended action from this example:

- keep current rupture thresholding
- add a read-side concentration descriptor / probe
- avoid prematurely widening runtime vocabulary or rewriting operators

## Structural-lens application examples

The transform and passive multi-scale comparison passes provide a useful distortion-audit pattern.

### Real FFT cleanup

Observed result:

- no meaningful change in the current normalized structural comparison surfaces,
- speed improved materially,
- distortion severity on the comparison surface remained effectively unchanged.

Recommended action:

- treat as backend/runtime optimization candidate,
- do not inflate into a new analytical lens claim.

### Wavelet comparison

Observed result:

- gross replay structure remained partially visible,
- but the narrower-band concentration distinction was weakened or destroyed,
- distortion severity rose at the support-descriptor layer for the `400 Hz` family.

Recommended action:

- reject wavelets as active lens for the current steady-state families,
- preserve FFT/Hann baseline,
- revisit only if future cohorts become transient-dominant.

### Passive multi-scale comparison

Observed result:

- all three declared scales preserved the coarse replay story,
- short scale increased noise / fragmentation pressure,
- long scale introduced temporal smearing near some boundaries,
- medium scale retained the best balance of preserved distinctions.

Recommended action:

- keep medium as active baseline,
- keep short/long as comparison lenses with explicit caveats,
- do not change runtime defaults from this pass.

### General lesson

A structural-lens comparison may be useful even when it does not produce a new active default.

The audit value lies in determining whether a candidate lens:

- preserves distinctions,
- collapses distinctions,
- adds ambiguity,
- or mainly changes operational cost.

That determination should happen before any runtime admission pressure.
