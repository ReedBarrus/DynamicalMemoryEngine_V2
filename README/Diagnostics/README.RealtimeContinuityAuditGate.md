## 1. Why this note exists

Door One can become lawful enough to preserve, replay, reconstruct, and retain structure without yet feeling experientially real as a memory substrate.

That gap appears when the system can:
- detect change,
- preserve support,
- and reconstruct bounded traces,

but the operator still cannot see retained structure actively constraining present interpretation in a way that feels causally clear and useful.

This note exists to define the first bounded proof target beyond static replay/reconstruction honesty:

**a realtime continuity and change surface that shows live signal state against retained structural memory support and makes present constraint visible without invoking agency.**

This remains below:
- agency,
- canon,
- autonomous workflow authority,
- and anthropomorphic language.

---

## 2. Core rule

**The first realtime continuity track is successful only when retained structural support becomes visibly operative in the reading of live or advancing signal state, while remaining explicitly below agency, truth uplift, canon, and anthropomorphic “memory” language.**

Corollary rules:

- continuity visibility is not agency
- retained support is not autonomous judgment
- interpretive usefulness is not authority uplift
- replay/reconstruction honesty must remain intact
- displayed continuity must not outrun mechanized support

## 3. Development rhetoric

This audit track should be described as:

- realtime continuity and change visibility
- continuity-constrained operator model
- retained-support-aware signal reading
- structural memory support becoming present in interpretation
- bounded continuity surface below agency

It should **not** be described as:

- the system remembering in a human sense
- the system thinking
- the system deciding
- the system understanding meaning
- the system becoming agentic
- the system gaining authority

The preferred phrasing is:

**Retained structural support is becoming visibly active in the reading of present signal state.**

## 4. Placement in developmental order

This audit track has two stages.

### Stage A — Post-Leg-3 structural honesty pass

Run the first test pass after:

- Leg 3 identity / memory / compression tracking

Purpose:
- validate that retained objects, replay objects, and reconstruction objects are honest and distinct
- validate that compression and support degradation behave causally
- validate that the system is structurally ready for a realtime surface

This stage is still mostly batch / bounded-window oriented.

### Stage B — Post-realtime-surface experiential pass

Run the second test pass after:

- operator legibility surface gains the time/frequency anchor
- live or advancing window display exists
- retained-support comparison is visible

Purpose:
- validate that continuity, drift, rupture, and recurrence are operator-legible
- validate that retained support visibly constrains present reading
- validate that the surface feels useful without inflating meaning

This stage is the first true experiential proof target for structural-memory usefulness below agency.

## 5. Display add-ons required for the audit gate

The realtime continuity audit should not begin until the following display-support additions exist.

### 5.1 Shared active timeline / advancing window

A single active timeline or advancing window must exist.

It should:
- move source and spectral-state views together,
- expose current position clearly,
- and preserve one bounded time reference across the operator surface.

Multiple competing time controls are not required for the first pass.

### 5.2 Time-frequency structural anchor

The primary structural anchor should include:

- spectrogram-like time-frequency view
- or an equivalent time-evolving spectral-state view

This should remain the main structural reference for continuity/change observation.

### 5.3 Selected-window spectral slice

A secondary view should expose the currently selected window as:

- an EQ-like spectral slice
- or equivalent local spectral-state representation

This is required so the operator can compare:
- evolving time-frequency structure
- with current-window structure

### 5.4 Retained-signature visibility

The operator surface must visibly expose:
- retained signature object
- or retained-support summary sufficient to compare live state against retained state

The retained object must not remain hidden in backend-only accounting.

### 5.5 Replay vs reconstruction distinction

The display must visibly distinguish:
- replay from source/support
- replay from retained basis where applicable
- reconstruction from retained trace

These must not collapse into one generic “replay” view.

This remains required by declared-vs-mechanized and mechanization-closure honesty. 

### 5.6 Current-status badges

The active view must expose bounded status labels such as:

- source_available
- retained_only
- replayable
- reconstructable
- insufficient_support
- partial
- candidate_only

Avoid stronger status language unless separately justified.

### 5.7 Retained-support influence indicator

The surface must visibly indicate when retained structural support is participating in the current read.

This may be implemented as:
- active retained reference marker
- continuity overlay
- support-match highlight
- retained-vs-live alignment strip
- or equivalent bounded indicator

The exact rendering may vary.
The requirement is that retained support must become **presently visible**, not merely stored.

## 6. Audit gate A — Post-Leg-3 structural honesty suite

This is the first required suite.

### A1. Replay vs reconstruction honesty test

#### Purpose
Verify that replay and reconstruction remain distinct objects and distinct actions.

#### Setup
For one bounded segment:
- replay from source/support
- replay from retained basis where applicable
- reconstruct from retained trace

#### Pass condition
The operator can distinguish:
- which path is replay,
- which path is reconstruction,
- what basis each path depends on,
- and what each path is still not claiming.

#### Failure condition
Any path feels interchangeable, or reconstruction is visually/semantically overread as replay.

### A2. Retention fidelity test

#### Purpose
Verify that retained structure preserves support in a bounded and inspectable way.

#### Setup
For one bounded segment:
- derive spectral state
- compress/retain support
- reconstruct from retained support

#### Pass condition
The system makes visible:
- what survived
- what degraded
- and what cannot be claimed from retained support alone

#### Failure condition
Retention behaves like opaque storage or reconstruction feels stronger than support justifies.

### A3. Compression boundary test

#### Purpose
Verify that compression level visibly affects retained support quality and reconstruction behavior.

#### Setup
Run light / medium / heavy compression on the same bounded source region.

#### Pass condition
The operator can see:
- reduced retained detail,
- changed reconstruction fidelity,
- and altered continuity confidence/support behavior.

#### Failure condition
Compression changes are numerically present but not causally legible.

### A4. Basic continuity vs drift test (bounded / non-realtime)

#### Purpose
Verify that the system can distinguish:
- continuity holding
- continuity weakening
- rupture
- and recurrence

before realtime display is introduced.

#### Setup
Use a sequence such as:
- baseline pattern
- slightly altered version
- rupture / unrelated pattern
- return to baseline-like pattern

#### Pass condition
Outputs and operator surface can distinguish:
- continuity
- drift
- rupture
- recurrence

without needing real-time streaming yet.

#### Failure condition
Everything looks equally “similar” or all continuity language becomes vague.

## 7. Audit gate B — Post-realtime-surface experiential suite

This suite should begin only after the display add-ons are present.

### B1. Realtime continuity vs drift test

#### Purpose
Verify that the operator can watch continuity evolve over time against retained support.

#### Setup
Feed a live or advancing signal stream with:
- stable region
- altered region
- rupture region
- return / recurrence region

#### Pass condition
The operator can visually track:
- continuity holding
- drift increasing
- rupture occurring
- recurrence appearing

without reconstructing theory from memory.

#### Failure condition
The display feels dynamic but does not clearly show state relation to retained support.

### B2. Novelty / rupture detection test

#### Purpose
Verify that the system does not hallucinate continuity where real rupture occurs.

#### Setup
Feed:
- stable signal
- sudden unrelated signal family/segment

#### Pass condition
The surface clearly marks discontinuity/rupture and does not pretend retained support still governs strongly.

#### Failure condition
Rupture is visually flattened into continuity, or novelty is under-signaled.

### B3. Cross-window stability test

#### Purpose
Verify that moving across nearby windows does not produce chaotic or misleading identity oscillation.

#### Setup
Advance the active window gradually across a continuous signal region.

#### Pass condition
Identity/continuity indications evolve smoothly where appropriate and break only when support warrants it.

#### Failure condition
The system flickers between incompatible states without a real structural reason.

### B4. Recurrence / reappearance test

#### Purpose
Verify that prior retained support can become visibly relevant again when a structurally similar region returns.

#### Setup
Feed:
- pattern A
- pattern B
- pattern A-like recurrence

#### Pass condition
The surface makes prior retained support visibly relevant again under explicit bounded posture.

#### Failure condition
Recurrence either goes unnoticed or is overstated as full identity without proper qualification.

### B5. Retained-support present-constraint test

#### Purpose
Verify the central proof target:
retained support visibly constrains present reading.

#### Setup
Use a live/advancing stream with previously retained bounded support objects available.

#### Pass condition
The operator can see:
- which retained support object is currently active or relevant,
- how it constrains current interpretation,
- and where that support weakens or breaks.

#### Failure condition
Retained support exists only as stored history, not as present structural influence.

## 8. Minimal output metrics and operator questions

The audit gate does not require a giant metric engine yet.

But it should support enough visibility to answer:

- what retained support object is active?
- what is the current relation between live state and retained support?
- is continuity holding, weakening, broken, or recurring?
- what is source-available vs retained-only?
- what is replayable vs reconstructable?
- what is still not claimed?

If these cannot be answered, the audit gate has not been passed.
