## 1. Why this note exists

Door One can become lawful enough to preserve, replay, reconstruct, and retain structure without yet feeling experientially real as a memory substrate.

That gap appears when the system can:
- detect change,
- preserve support,
- and reconstruct bounded traces,

but the operator still cannot see retained structure actively constraining present interpretation in a way that feels causally clear and useful.

This note exists to define the first bounded proof target beyond static replay/reconstruction honesty:

**a realtime continuity and change surface that shows live signal state against retained structural memory support and makes present constraint visible without invoking agency.**

This remains below:
- agency,
- canon,
- autonomous workflow authority,
- and anthropomorphic language.

---

## 2. Core rule

**The first realtime continuity track is successful only when retained structural support becomes visibly operative in the reading of live or advancing signal state, while remaining explicitly below agency, truth uplift, canon, and anthropomorphic “memory” language.**

Corollary rules:

- continuity visibility is not agency
- retained support is not autonomous judgment
- interpretive usefulness is not authority uplift
- replay/reconstruction honesty must remain intact
- displayed continuity must not outrun mechanized support

---

## 3. Development rhetoric

This audit track should be described as:

- realtime continuity and change visibility
- continuity-constrained operator model
- retained-support-aware signal reading
- structural memory support becoming present in interpretation
- bounded continuity surface below agency

It should **not** be described as:

- the system remembering in a human sense
- the system thinking
- the system deciding
- the system understanding meaning
- the system becoming agentic
- the system gaining authority

The preferred phrasing is:

**Retained structural support is becoming visibly active in the reading of present signal state.**

## 4. Placement in developmental order

This audit track has two stages.

### Stage A — Post-Leg-3 structural honesty pass

Run the first test pass after:

- Leg 3 identity / memory / compression tracking

Purpose:
- validate that retained objects, replay objects, and reconstruction objects are honest and distinct
- validate that compression and support degradation behave causally
- validate that the system is structurally ready for a realtime surface

This stage is still mostly batch / bounded-window oriented.

### Stage B — Post-realtime-surface experiential pass

Run the second test pass after:

- operator legibility surface gains the time/frequency anchor
- live or advancing window display exists
- retained-support comparison is visible

Purpose:
- validate that continuity, drift, rupture, and recurrence are operator-legible
- validate that retained support visibly constrains present reading
- validate that the surface feels useful without inflating meaning

This stage is the first true experiential proof target for structural-memory usefulness below agency.

## 5. Display add-ons required for the audit gate

The realtime continuity audit should not begin until the following display-support additions exist.

### 5.1 Shared active timeline / advancing window

A single active timeline or advancing window must exist.

It should:
- move source and spectral-state views together,
- expose current position clearly,
- and preserve one bounded time reference across the operator surface.

Multiple competing time controls are not required for the first pass.

### 5.2 Time-frequency structural anchor

The primary structural anchor should include:

- spectrogram-like time-frequency view
- or an equivalent time-evolving spectral-state view

This should remain the main structural reference for continuity/change observation.

### 5.3 Selected-window spectral slice

A secondary view should expose the currently selected window as:

- an EQ-like spectral slice
- or equivalent local spectral-state representation

This is required so the operator can compare:
- evolving time-frequency structure
- with current-window structure

### 5.4 Retained-signature visibility

The operator surface must visibly expose:
- retained signature object
- or retained-support summary sufficient to compare live state against retained state

The retained object must not remain hidden in backend-only accounting.

### 5.5 Replay vs reconstruction distinction

The display must visibly distinguish:
- replay from source/support
- replay from retained basis where applicable
- reconstruction from retained trace

These must not collapse into one generic “replay” view.

This remains required by declared-vs-mechanized and mechanization-closure honesty. 

### 5.6 Current-status badges

The active view must expose bounded status labels such as:

- source_available
- retained_only
- replayable
- reconstructable
- insufficient_support
- partial
- candidate_only

Avoid stronger status language unless separately justified.

### 5.7 Retained-support influence indicator

The surface must visibly indicate when retained structural support is participating in the current read.

This may be implemented as:
- active retained reference marker
- continuity overlay
- support-match highlight
- retained-vs-live alignment strip
- or equivalent bounded indicator

The exact rendering may vary.
The requirement is that retained support must become **presently visible**, not merely stored.

## 6. Audit gate A — Post-Leg-3 structural honesty suite

This is the first required suite.

### A1. Replay vs reconstruction honesty test

#### Purpose
Verify that replay and reconstruction remain distinct objects and distinct actions.

#### Setup
For one bounded segment:
- replay from source/support
- replay from retained basis where applicable
- reconstruct from retained trace

#### Pass condition
The operator can distinguish:
- which path is replay,
- which path is reconstruction,
- what basis each path depends on,
- and what each path is still not claiming.

#### Failure condition
Any path feels interchangeable, or reconstruction is visually/semantically overread as replay.

### A2. Retention fidelity test

#### Purpose
Verify that retained structure preserves support in a bounded and inspectable way.

#### Setup
For one bounded segment:
- derive spectral state
- compress/retain support
- reconstruct from retained support

#### Pass condition
The system makes visible:
- what survived
- what degraded
- and what cannot be claimed from retained support alone

#### Failure condition
Retention behaves like opaque storage or reconstruction feels stronger than support justifies.

### A3. Compression boundary test

#### Purpose
Verify that compression level visibly affects retained support quality and reconstruction behavior.

#### Setup
Run light / medium / heavy compression on the same bounded source region.

#### Pass condition
The operator can see:
- reduced retained detail,
- changed reconstruction fidelity,
- and altered continuity confidence/support behavior.

#### Failure condition
Compression changes are numerically present but not causally legible.

### A4. Basic continuity vs drift test (bounded / non-realtime)

#### Purpose
Verify that the system can distinguish:
- continuity holding
- continuity weakening
- rupture
- and recurrence

before realtime display is introduced.

#### Setup
Use a sequence such as:
- baseline pattern
- slightly altered version
- rupture / unrelated pattern
- return to baseline-like pattern

#### Pass condition
Outputs and operator surface can distinguish:
- continuity
- drift
- rupture
- recurrence

without needing real-time streaming yet.

#### Failure condition
Everything looks equally “similar” or all continuity language becomes vague.

---

## 7. Audit gate B — Post-realtime-surface experiential suite

This suite should begin only after the display add-ons are present.

### B1. Realtime continuity vs drift test

#### Purpose
Verify that the operator can watch continuity evolve over time against retained support.

#### Setup
Feed a live or advancing signal stream with:
- stable region
- altered region
- rupture region
- return / recurrence region

#### Pass condition
The operator can visually track:
- continuity holding
- drift increasing
- rupture occurring
- recurrence appearing

without reconstructing theory from memory.

#### Failure condition
The display feels dynamic but does not clearly show state relation to retained support.

### B2. Novelty / rupture detection test

#### Purpose
Verify that the system does not hallucinate continuity where real rupture occurs.

#### Setup
Feed:
- stable signal
- sudden unrelated signal family/segment

#### Pass condition
The surface clearly marks discontinuity/rupture and does not pretend retained support still governs strongly.

#### Failure condition
Rupture is visually flattened into continuity, or novelty is under-signaled.

### B3. Cross-window stability test

#### Purpose
Verify that moving across nearby windows does not produce chaotic or misleading identity oscillation.

#### Setup
Advance the active window gradually across a continuous signal region.

#### Pass condition
Identity/continuity indications evolve smoothly where appropriate and break only when support warrants it.

#### Failure condition
The system flickers between incompatible states without a real structural reason.

### B4. Recurrence / reappearance test

#### Purpose
Verify that prior retained support can become visibly relevant again when a structurally similar region returns.

#### Setup
Feed:
- pattern A
- pattern B
- pattern A-like recurrence

#### Pass condition
The surface makes prior retained support visibly relevant again under explicit bounded posture.

#### Failure condition
Recurrence either goes unnoticed or is overstated as full identity without proper qualification.

### B5. Retained-support present-constraint test

#### Purpose
Verify the central proof target:
retained support visibly constrains present reading.

#### Setup
Use a live/advancing stream with previously retained bounded support objects available.

#### Pass condition
The operator can see:
- which retained support object is currently active or relevant,
- how it constrains current interpretation,
- and where that support weakens or breaks.

#### Failure condition
Retained support exists only as stored history, not as present structural influence.

---

## 8. Minimal output metrics and operator questions

The audit gate does not require a giant metric engine yet.

But it should support enough visibility to answer:

- what retained support object is active?
- what is the current relation between live state and retained support?
- is continuity holding, weakening, broken, or recurring?
- what is source-available vs retained-only?
- what is replayable vs reconstructable?
- what is still not claimed?

If these cannot be answered, the audit gate has not been passed.

## Suggested target stream families for realtime continuity and retention testing

This section defines a bounded set of recommended input-stream families for testing realtime continuity, change detection, recurrence, retention, compression behavior, and support-tier identity posture.

It is a supporting audit-design addendum.

It does **not**:
- define source-family ontology,
- prove cross-family memory equivalence,
- replace source-family-specific ingest semantics,
- or claim that all target streams are equally mature or equally meaningful.

Its purpose is narrower:
- provide a practical testing spectrum,
- help select inputs that stress different aspects of the substrate,
- and support lawful evaluation of continuity, drift, rupture, recurrence, retention, and reconstruction across increasingly difficult stream conditions.

This section is intentionally practical.

### Why this section exists

Realtime continuity testing needs more than one kind of signal.

If all tests use:
- overly clean inputs,
- overly self-shaped inputs,
- or one narrow source family behavior,

then the substrate can appear more coherent or more memory-supportive than it really is.

The goal is therefore to test across a spectrum from:

- highly ordered / recurrence-friendly streams
to
- disruptive / chaotic / weakly repeatable streams

so the system can be evaluated under:

- clean recurrence
- partial recurrence
- drift
- rupture
- noise
- and support degradation

### Core rule

**A target stream is useful for this audit track when it makes one or more of the following structurally visible: continuity, drift, rupture, recurrence, retention loss, reconstruction strength, or support-tier transition.**

The purpose is not to find streams that “look cool.”
The purpose is to find streams that reveal structural behavior honestly.

### Testing-spectrum rule

The target stream set should include a practical spectrum from:

1. highly ordered / highly recurrence-supportive streams
2. moderately variable but still identity-supportive streams
3. mixed and partially chaotic streams
4. highly disruptive or weakly memory-supportive streams

This spectrum helps the operator observe:

- what kinds of streams preserve strong retained support
- what kinds degrade quickly under compression
- what kinds support recurrence-driven constraint tightening
- and where continuity claims should weaken or break

---

## 1. Recommended current target families (audio / synthetic)

### Tier A — Highly ordered / recurrence-supportive

These streams are useful for establishing clean baseline behavior.

#### A1. Pure sine tone
Use:
- one stable frequency
- stable amplitude
- long bounded duration

Useful for:
- continuity holding
- cross-window stability
- clean spectral-state formation
- baseline retained-signature behavior

What it reveals:
- the simplest high-order recurrence-supportive structure
- minimal drift under ideal conditions

#### A2. Multi-tone harmonic stack
Use:
- 2–5 simultaneous stable tones
- fixed amplitude and phase posture where possible

Useful for:
- multi-band identity
- compression under richer but still lawful structure
- replay vs reconstruction distinction across more than one band

What it reveals:
- whether retained support preserves structured relationships, not only single-band persistence

#### A3. Repeating synthetic pulse train
Use:
- repeated impulse / pulse pattern
- stable temporal spacing

Useful for:
- recurrence detection
- rupture visibility when spacing changes
- temporal periodicity support

What it reveals:
- continuity across repeated temporal structure

### Tier B — Moderately variable / identity-supportive

These streams introduce lawful complexity without becoming fully chaotic.

#### B1. Human speech — repeated same phrase
Use:
- the same speaker
- repeating the same short phrase several times
- small natural variation across repetitions

Useful for:
- recurrence with drift
- support-tier testing under realistic human variation
- recurrence-driven constraint tightening

What it reveals:
- whether the substrate can preserve structured recurrence without requiring exact identity

#### B2. Human speech — altered phrase family
Use:
- same phrase
- slight variation
- rupture phrase
- return to original phrase

Useful for:
- continuity vs drift
- rupture
- recurrence / reappearance

What it reveals:
- whether the system can distinguish “same,” “near,” “different,” and “returned”

#### B3. Sustained vocal tone with intentional modulation
Use:
- hummed vowel
- slight pitch shift
- amplitude change
- breath/noise variation

Useful for:
- frequency drift
- amplitude drift
- support degradation
- retained-signature robustness under controlled change

What it reveals:
- whether invariants and local change remain distinguishable

### Tier C — Mixed / partially chaotic

These streams increase disruption while preserving enough structure to remain informative.

#### C1. Speech in mild environmental noise
Use:
- repeated phrase with fan, room noise, or similar background layer

Useful for:
- support survival under contamination
- reconstruction weakness
- continuity confidence degradation

What it reveals:
- when retained support remains useful
- when noise begins to consume identity support

#### C2. Two-speaker alternation
Use:
- alternating speakers
- same phrase or related phrases

Useful for:
- rupture and source-family similarity tension
- identity boundary stress
- support-tier weakening across similar-but-not-same structures

What it reveals:
- whether recurrence is being overclaimed from broad resemblance

#### C3. Synthetic drift stream
Use:
- one stable pattern gradually changing in frequency, amplitude, or pulse timing

Useful for:
- continuity weakening
- distortion threshold observation
- support-tier transition tracking

What it reveals:
- where drift remains lawful continuity versus where it becomes break

### Tier D — Disruptive / weakly memory-supportive

These streams are useful for falsification and anti-overclaim testing.

#### D1. Broadband noise
Use:
- white noise
- pink noise
- band-limited noise variants

Useful for:
- novelty / rupture
- false continuity detection
- weak retained support

What it reveals:
- whether the system hallucinates structure where support is weak

#### D2. Abrupt family switch
Use:
- speech → silence → tone → noise
or
- tone → speech → percussion-like transient sequence

Useful for:
- rupture detection
- seam honesty
- support disjunction

What it reveals:
- whether major structural breaks remain visible

#### D3. Low-recurrence synthetic irregular stream
Use:
- randomized pulses
- variable intervals
- variable amplitude
- limited repeating pattern

Useful for:
- weak recurrence
- compression stress
- “memory not actually supported” detection

What it reveals:
- where the substrate should weaken claims rather than preserve them

## 2. Recommended use by test purpose

### For continuity holding
Prefer:
- pure sine
- harmonic stack
- repeated phrase
- pulse train

### For drift
Prefer:
- synthetic drift stream
- sustained vocal modulation
- phrase with small variation

### For rupture
Prefer:
- abrupt family switch
- unrelated phrase insertion
- noise interruption

### For recurrence
Prefer:
- pulse train
- repeated phrase
- pattern A → pattern B → pattern A return

### For retention/compression stress
Prefer:
- harmonic stack
- repeated phrase
- noisy speech
- synthetic drift stream

### For false continuity / anti-overclaim
Prefer:
- broadband noise
- two-speaker alternation
- abrupt family switch
- low-recurrence irregular synthetic stream

## 3. Suggested support-tier targeting posture

The target stream family should also be used to observe how streams move across retention/support tiers.

### Strong support candidates
Likely examples:
- pure tones
- harmonic stacks
- strongly repeated phrases
- stable pulse trains

Use these for:
- upper support-tier and recurrence-strength testing

### Mid-support candidates
Likely examples:
- repeated speech with variation
- synthetic drift
- vocal tone with modulation
- mild-noise speech

Use these for:
- degraded-but-still-meaningful support posture

### Weak-support candidates
Likely examples:
- heavy noise
- irregular synthetic streams
- abrupt family switches
- speaker alternation with weak shared structure

Use these for:
- downgrade posture
- insufficient_support visibility
- anti-overclaim checks

### Practical rule

A stream is not “bad” just because it yields weak support.

Weak-support streams are required to prove:
- downgrade honesty
- rupture visibility
- and the absence of fake continuity.

## 4. Suggested phased testing order

The preferred first-pass order is:

### Phase 1 — Ordered baselines
- pure sine
- harmonic stack
- pulse train

### Phase 2 — Human recurrence
- repeated same phrase
- phrase variation
- vocal modulation

### Phase 3 — Drift and contamination
- synthetic drift
- noisy speech
- mild interruption patterns

### Phase 4 — Disruption and falsification
- broadband noise
- abrupt family switch
- irregular low-recurrence stream

This order helps the operator learn the substrate from:
- strongest continuity
to
- weakest support

without beginning in chaos.

## 5. Future target families after source-family expansion

Later, once ingest expands beyond current audio emphasis, future target families may include:

- CSV time-series telemetry
- event-stream JSON traces
- sensor logs
- multichannel environmental telemetry
- behavioral interaction traces
- mixed family coherence comparisons

These should not be treated as immediate equivalents to audio testing.

The audio/synthetic set remains the current bounded proving ground.

## 6. Recommended operator review questions for target-stream selection

Before selecting a test stream, ask:

1. What structural behavior is this stream supposed to reveal?
2. Is the stream too clean to falsify anything?
3. Is the stream too chaotic to teach anything yet?
4. Does this stream help test continuity, drift, rupture, recurrence, or support degradation?
5. What support-tier posture is this stream likely to stress?

If the stream does not help answer one of those questions, it is not yet a strong audit target.
